## 1. 这张表是干什么的

这张表不是问题台账，也不是总结报告。

它的作用只有一个：

1. 把每一次真实试用动作，压成可以快速回填、又足够支持后续总结的观察记录。

换句话说：

1. 问题台账回答“出了什么问题”。
2. 总结报告回答“最后决定是什么”。
3. 观察记录表回答“这次真实使用时，到底发生了什么”。

如果没有这层记录，团队很容易出现两种偏差：

1. 只在出问题时才留下痕迹，导致大量“顺利用过”的证据消失。
2. 到窗口结束时，只剩下印象，没有足够扎实的试用证据链。

## 2. 什么时候填这张表

建议每次出现下面任一情况时，都填一条记录：

1. 完成一次完整的 report 浏览与样本判断。
2. 完成一次 feedback 写入或批量写入。
3. 完成一次 queue 认领、取消认领或状态推进。
4. 完成一次周度复盘推进。
5. 完成一次 cache benchmark / 运行证据类阅读。

最小原则：

1. 一次会话至少一条记录。
2. 不要求很长，但必须能支撑后续复盘。

## 3. 一条记录最少要写什么

每条记录至少写下面六项：

1. 我在看哪个 report / trade_date。
2. 我尝试完成什么动作。
3. 结果是否成功。
4. 花了多少额外理解成本。
5. 有没有发现问题。
6. 这条观察更支持“通过”“待观察”还是“需要修复”。

## 4. 推荐字段模板

建议按下面结构记录。

```md
### 观察记录 N

- 日期：2026-03-26
- 观察人：einstein
- report：paper_trading_window_20260316_20260323_live_m2_7_20260323
- trade_date：2026-03-20
- 场景类型：selected / near-miss / blocker / batch label / workflow / cache benchmark
- 目标动作：阅读 selected 样本并写入一条 draft feedback
- 结果：成功 / 部分成功 / 失败
- 是否留下系统内证据：是 / 否
- 关键证据：
  - 例如 feedback 已写入并在 activity 面板回读
  - 例如 workflow item 已从 unassigned 切换为 ready_for_adjudication
- 主要摩擦：
  - 无
  - 或：字段理解成本高 / 筛选路径绕 / 信息需要来回切换
- 问题归类：无 / A 类 / B 类 / C 类
- 是否需要转问题台账：是 / 否
- 一句话结论：
  - 例如：selected 阅读与 feedback 写入链路通过，但 workflow 语义仍需靠文档辅助理解
```

## 5. 最实用的五种观察场景

### 5.1 样本阅读观察

适用于：

1. near-miss
2. selected
3. execution blocker

最该记录：

1. 样本是否容易找到。
2. 关键字段是否容易理解。
3. 能否快速分清研究层和执行层。

一句话模板：

1. `[`样本类型`]` 阅读链路 `[`顺畅 / 一般 / 困难`]`，当前更像 `[`研究问题 / 执行问题 / 流程问题`]`，是否需要额外文档辅助 `[`需要 / 不需要`]`。

### 5.2 feedback 写入观察

适用于：

1. 单条 feedback
2. 批量 feedback

最该记录：

1. 表单是否顺手。
2. 写入后是否立即可回读。
3. 标签和 review_status 是否容易选对。

一句话模板：

1. feedback 写入 `[`成功 / 部分成功 / 失败`]`，回读 `[`即时 / 延迟 / 异常`]`，主要摩擦是 `[`字段理解 / 交互步骤 / 状态反馈`]`。

### 5.3 workflow 观察

适用于：

1. `Assign to me`
2. `Unassign`
3. `draft -> final`
4. `ready_for_adjudication`

最该记录：

1. 条目是否好找。
2. 状态是否好懂。
3. 操作后是否形成闭环。

一句话模板：

1. workflow 队列 `[`可用 / 勉强可用 / 难以使用`]`，当前主要问题是 `[`筛选 / 语义 / 状态可见性`]`。

### 5.4 周度复盘观察

适用于：

1. 批量推进 `draft`
2. 形成 `final`
3. 形成 backlog 映射

最该记录：

1. 周度复盘是否真的能推进状态。
2. 是否能从 report 走到 backlog 结论。
3. 卡点在页面、语义还是流程治理。

一句话模板：

1. 周度复盘闭环 `[`已形成 / 部分形成 / 未形成`]`，当前最大的缺口在 `[`使用习惯 / 流程治理 / 页面能力`]`。

### 5.5 运行证据观察

适用于：

1. cache benchmark
2. report 级运行摘要

最该记录：

1. 用户是否能看懂 benchmark 是否真实执行。
2. 是否容易把 `reuse_confirmed` 误判成永久保证。
3. 页面摘要是否足够支撑结论。

一句话模板：

1. 运行证据阅读 `[`清楚 / 一般 / 模糊`]`，当前页面对 benchmark 成败与复用幅度的表达 `[`足够 / 还需文档辅助 / 不足`]`。

## 6. 最小评分法

如果你不想写太多文字，可以顺手给每条观察打一个最小评分：

| 维度 | 评分规则 |
| --- | --- |
| 可找到性 | 1 = 很难找到，2 = 能找到但绕，3 = 很快找到 |
| 可理解性 | 1 = 需要反复猜，2 = 基本能懂，3 = 一眼能判断 |
| 可操作性 | 1 = 动作失败或容易错，2 = 能完成但不顺，3 = 顺畅完成 |
| 可沉淀性 | 1 = 很难留下结论，2 = 能留下但质量一般，3 = 很容易沉淀成 feedback / queue / 结论 |

最小用法：

1. 每条观察填 4 个分数。
2. 低于 8 分的记录，优先检查是否需要进入问题台账。
3. 连续多条低于 8 分，基本说明这不是偶发问题，而是治理项候选。

## 7. 什么情况必须转入问题台账

出现下面任一情况，不要只留在观察表里，必须同步记入问题台账：

1. 核心动作失败。
2. 数据回读异常。
3. 状态流转与预期不一致。
4. 同类摩擦在不同会话中重复出现。
5. 你无法判断这是自己不会用，还是系统表达不清。

对应文档：

1. [Replay Artifacts 试用问题记录模板](./replay-artifacts-trial-issue-log-template.md)

## 8. 什么情况只记观察，不要立刻开问题

下面这些情况先记观察即可：

1. 第一次使用时不熟悉页面。
2. 需要回看一次术语手册才能确认字段含义。
3. 虽然步骤多一点，但最终顺利完成。
4. 当前只是怀疑存在摩擦，还没有复现第二次。

这样做是为了避免试用窗口一开始就把“学习成本”误报成“产品缺陷”。

## 9. 一个最短的回填示例

```md
### 观察记录 03

- 日期：2026-03-26
- 观察人：einstein
- report：paper_trading_window_20260316_20260323_live_m2_7_20260323
- trade_date：2026-03-23
- 场景类型：blocker + workflow
- 目标动作：确认 300724 的 execution blocker，并完成一次 Assign to me / Unassign
- 结果：成功
- 是否留下系统内证据：是
- 关键证据：
  - `buy_order_blocker = position_blocked_score` 可读
  - queue 条目可找到，Assign to me 和 Unassign 都成功
- 主要摩擦：
  - `workflow_status` 的语义仍需配合术语手册理解
- 问题归类：无
- 是否需要转问题台账：否
- 一句话结论：
  - blocker 阅读与 ownership 操作链路通过，当前更像文档辅助需求，而不是功能缺陷
```

## 10. 试用窗口结束时怎么用这些记录

窗口结束时，不要逐条复述观察记录。

正确用法是：

1. 先统计“顺利用过”的链路有哪些。
2. 再统计“重复出现的摩擦”有哪些。
3. 最后把高频摩擦转成问题台账或治理 backlog。

如果你发现大多数记录都是：

1. 动作成功
2. 证据可回读
3. 只是需要文档辅助理解

那通常说明当前系统已经足够进入常规试用，而不是必须继续扩主线能力。
