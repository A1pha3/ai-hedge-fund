## A/B Walk-Forward 测试流程、设计目的、运行耗时与优化过程记录

**文档日期**：2026 年 3 月 8 日  
**文档用途**：用于团队审阅当前真实 A 股 A/B Walk-Forward 测试的流程设计、目标、耗时原因、优化路线与优化过程记录。  
**适用范围**：本轮 2025-12-01 至 2026-03-04 的真实 A/B 回测任务。

## 1. 执行摘要

当前这轮测试不是普通的单次回测，而是一次 **A/B Walk-Forward 对比测试**。它的核心目标是：

1. 用同一段测试期，比较 baseline 策略与 MVP 策略的效果差异。
2. 判断 MVP 这套更复杂、更昂贵的多阶段流程，是否真的带来更好的风险收益表现。
3. 在尽量避免未来信息泄漏的前提下，得到更可信的真实运行结论。

截至当前时点，已经确认：

1. 本轮只生成了 **1 个 walk-forward window**。
2. **window1 的 baseline 已完成**。
3. 当前正在执行 **window1 的 MVP 阶段**。
4. baseline 阶段最明确、最大、已被 timing 直接证明的瓶颈是：**precise_agent 多分析师 LLM 精确分析**。
5. 当前 MVP 阶段最可疑、最值得优先继续优化的瓶颈是：**进入 LLM 之前的大候选池串行数据抓取与打分阶段**。

基于当前已经掌握的证据，可以形成一个新的阶段性判断：

1. baseline 单窗口约 1 小时左右，虽然不算便宜，但当前仍处于可接受范围。
2. MVP 若继续维持当前形态，预计总耗时可能在十几个小时量级，这一成本不可接受。
3. 因此，后续优化目标不再是继续优先压缩 baseline，而是要把 **MVP 总运行时间压到 3 至 4 小时以内**。

## 3. 测试设计目的是什么

### 3.1 为什么不是只跑 MVP

如果只跑 MVP，只能知道它“跑出来了什么”，却很难知道：

1. 它到底比一个简单策略强在哪里。
2. 它额外消耗的大量模型成本和运行时间是否值得。
3. 如果结果一般，是策略本身不行，还是只是市场环境不适配。

因此，必须保留 baseline 作为对照组。

### 3.2 为什么使用 walk-forward

walk-forward 的设计目的，不是为了把流程做复杂，而是为了减少“看过答案再回头调参数”的风险。

它强调的是：

1. 先用训练区间确定策略配置与运行规则。
2. 再用后续测试区间检验真实表现。
3. 用这种顺序来提高结论的可信度。

### 3.3 这轮测试最终想回答的问题

本轮测试本质上要回答下面这个问题：

> 在真实 A 股测试窗口中，MVP 这套更复杂的筛选、融合、分层分析与多分析师决策流程，是否明显优于 baseline，并且这种提升是否足以覆盖它的时间成本与模型成本？

## 4. 当前运行流程总览

### 4.1 总流程

| 阶段 | 说明 | 输出 |
|------|------|------|
| Baseline | 先跑对照组策略 | baseline 指标与 checkpoint |
| MVP | 再跑多阶段主策略 | MVP 指标与 checkpoint |
| 汇总比较 | 汇总 Sharpe、Sortino、回撤等指标 | Markdown 报告与 JSON 报告 |

### 4.2 Baseline 的业务流程

当前 baseline 的业务流程相对直接：

| 步骤 | 说明 |
|------|------|
| 1 | 获取当日涨幅榜 |
| 2 | 选取涨幅榜前 N 只股票 |
| 3 | 将这 N 只送入多分析师精确分析 |
| 4 | 聚合结果，生成观察名单与计划 |

当前已经完成的 baseline，最新有效配置已经从 **Top 20 缩减到 Top 10**。

### 4.3 MVP 的业务流程

MVP 明显更复杂，当前阶段的主要步骤如下：

| 步骤 | 说明 |
|------|------|
| 1 | 构建 candidate pool |
| 2 | 对候选池进行 Layer B 打分 |
| 3 | 做多策略信号融合 |
| 4 | 对高分池运行 fast agent |
| 5 | 对更小范围高优先级股票运行 precise agent |
| 6 | 进行 Layer C 聚合与交易计划生成 |

这也是为什么 MVP 的前置成本天然比 baseline 更高。

## 5. 为什么初期会消耗这么长时间

### 5.1 结论先行

初期耗时长，并不是因为“程序卡住了”，也不是因为“回测框架本身很慢”，而是因为它在初始阶段连续触发了两类非常昂贵的外部工作：

| 类型 | 描述 | 特征 |
|------|------|------|
| LLM 精确分析 | 多分析师对股票做精细判断 | 单次慢、总量大、受限流影响明显 |
| 多源数据抓取与打分 | 价格、财务、新闻、交易数据的串行加载 | 候选池越大，耗时越长 |

### 5.2 初期为什么看起来“特别慢”

初期慢的原因主要有四个：

1. baseline 要先完整跑完，MVP 才能开始。
2. baseline 一开始仍然包含多分析师精确分析，单日就能吃掉几十分钟。
3. MVP 在进入 LLM 前，还要先完成大候选池的前置数据准备与打分。
4. 当前 LLM 服务存在速率限制与回退链，导致单次请求延迟不稳定。

## 6. 已经被 timing 证明的最大瓶颈

### 6.1 baseline 阶段的定量证据

当前已经拿到两条完整的 baseline 单日 timing 记录，足以证明最大的瓶颈在哪里。

| 日期 | Baseline 配置 | 总耗时 | Post-Market 耗时 | Precise Agent 耗时 | Precise Agent 占总耗时比例 |
|------|---------------|--------|------------------|--------------------|----------------------------|
| 2026-02-26 | Top 20 | 3653.758 秒 | 3650.821 秒 | 3649.642 秒 | 99.89% |
| 2026-02-27 | Top 10 | 3118.656 秒 | 3117.245 秒 | 3115.753 秒 | 99.91% |

### 6.2 这说明了什么

从上表可以直接得出三个结论：

1. baseline 的时间几乎全部消耗在 precise_agent 上。
2. candidate pool、市场状态识别、聚合、买卖计划都不是 baseline 的主瓶颈。
3. 将 baseline 从 Top 20 收缩到 Top 10 后，单日耗时下降了约 14.6%，说明“减少送入精确分析的股票数”是有效的。

### 6.3 基线耗时结构图表

| 耗时构成 | Top 20 | Top 10 | 结论 |
|----------|--------|--------|------|
| 精确分析 | 极高 | 仍极高 | 绝对主瓶颈 |
| 取涨幅榜 | 极低 | 极低 | 可忽略 |
| 市场状态 | 极低 | 极低 | 可忽略 |
| 聚合与计划 | 极低 | 极低 | 可忽略 |

可以把 baseline 理解成一个非常典型的“少量股票、极重分析”的流程。

## 7. 当前 MVP 最可能的主瓶颈

### 7.1 为什么当前还不能像 baseline 一样精确定量

截至当前时点，MVP timing 文件里还只有 prefetch 记录，还没有完整的 MVP 单日 timing。

这意味着：

1. MVP 的第一天还没有完整落盘。
2. 当前仍处于单日重计算过程中。
3. 只能先根据代码路径和实时终端输出来判断当前最大瓶颈。

### 7.2 当前最可疑的瓶颈路径

当前 MVP 最可疑的主瓶颈，是 **候选池进入 Layer B 打分之前后的串行外部数据抓取**。

原因如下：

| 观察 | 含义 |
|------|------|
| 终端持续出现大量 Tushare 日线拉取 | 说明价格数据读取非常频繁 |
| 终端持续出现 AKShare 新闻过滤与新闻去重 | 说明新闻数据处理量大 |
| MVP 的 score_batch 是逐候选股票串行执行 | 候选池越大，总时间越长 |
| 每个候选股票都可能拉价格、财务、新闻、交易数据 | 单股票成本不低 |

### 7.3 MVP 当前耗时结构的推断图表

| 阶段 | 当前判断 | 原因 |
|------|----------|------|
| Candidate Pool | 中等偏重 | 已经做过限流优化，但仍有市场级筛选成本 |
| Layer B 打分 | 很重 | 大量串行数据抓取与指标计算 |
| Fast Agent | 重 | 高分池进入多分析师分析 |
| Precise Agent | 很重 | 高优先级股票进入更贵的精细分析 |
| 聚合与计划 | 较轻 | 纯本地计算为主 |

当前最合理的推断是：

> MVP 是“先被串行数据准备拖慢，再被 LLM 分析继续拖慢”的双重重路径。

## 8. 为什么会在初期集中暴露这些成本

初期之所以最容易让人感觉“几乎没有进展”，主要是因为：

### 8.1 可见成果滞后

在一个完整交易日完成之前：

1. day timing 不会落盘。
2. 新 checkpoint 不一定更新到用户能立刻感知的程度。
3. 大量时间会花在中间步骤，而这些步骤对用户来说不像“完成了一天”那样显眼。

### 8.2 外部依赖的慢不会均匀分布

模型服务、新闻接口、价格接口、财务接口都不是稳定线性的：

| 外部因素 | 表现 |
|----------|------|
| LLM 限流 | 429、回退、重试 |
| 新闻数据质量 | 去重、无关新闻过滤会额外消耗时间 |
| 股票数过多 | 候选池放大后所有前置步骤都会膨胀 |
| A 股市场数据维度多 | 价格、财务、情绪、交易等链路都需要合并 |

因此，初期并不是没有工作产出，而是工作产出大多还在“日内处理中间态”。

## 9. 已实施的优化措施

截至当前，已经实施并上线的优化措施如下。

### 9.1 运行稳定性优化

| 优化项 | 目标 | 当前效果 |
|--------|------|----------|
| Checkpoint / Resume | 避免长任务重启后从头开始 | 已生效 |
| Supervisor 守护 | 自动在重置窗口后恢复运行 | 已生效 |
| Provider 回退链 | Coding Plan → MiniMax → 标准 Zhipu | 已生效 |
| 并发控制 | 限制 analyst 波次并发，降低 429 风险 | 已生效 |

### 9.2 性能优化

| 优化项 | 目的 | 当前状态 |
|--------|------|----------|
| 同日同模型分析结果复用 | 去掉 fast 与 precise 重复分析 | 已生效 |
| LangGraph 编译缓存 | 去掉同进程重复 compile 开销 | 已生效 |
| Candidate Pool 动态限流补差 | 去掉固定 15 秒整批等待 | 已生效 |
| Baseline Top N 从 20 降到 10 | 直接降低 precise_agent 成本 | 已生效 |

### 9.3 这些优化的真实意义

这些优化不是“让程序看起来更聪明”，而是分别在不同层面降低时间消耗：

| 层面 | 优化逻辑 |
|------|----------|
| 框架层 | 少做重复构图、少做重复执行 |
| 数据层 | 少做不必要等待 |
| 策略层 | 减少必须进入昂贵精确分析的股票数量 |
| 运维层 | 降低重启导致的时间损失 |

### 9.4 已完成的优化过程记录

下面这部分用于记录已经实际做过的优化动作，而不是仅记录抽象结论。

| 阶段 | 发现的问题 | 处理方案 | 当前结果 |
|------|------------|----------|----------|
| 第一轮 | fast 与 precise 在同日同模型下存在重复分析 | 引入同日同模型结果复用 | 去掉了一部分重复 LLM 工作 |
| 第二轮 | 重复运行时存在本地工作流重复编译开销 | 对 LangGraph 编译结果做进程内缓存 | 消除了局部重复 compile 成本 |
| 第三轮 | candidate pool 每 50 次 daily 调用固定休眠 15 秒，存在过度等待 | 改为按真实已消耗时间补足限流预算 | 去掉了粗粒度整批等待 |
| 第四轮 | baseline precise_agent 工作量过大 | baseline Top N 从 20 收缩到 10 | baseline 单日耗时下降约 14.6% |
| 第五轮 | 长任务需要人工盯守且重启代价高 | 结合 checkpoint 与 supervisor 运行 | 降低了中断后的时间损失 |

### 9.5 优化过程中遇到的困难与应对方式

这一部分是为了保留真正有复盘价值的上下文。

| 困难 | 表现 | 当时的应对方式 | 经验结论 |
|------|------|----------------|----------|
| 缺少早期定量证据 | MVP 在第一天完成前没有完整 day timing | 先用 baseline timing 与代码路径做阶段判断 | 在没有完整数据前，要区分“已证明”与“高概率推断” |
| LLM 服务限流 | 运行中出现 429、回退、延迟波动 | 使用 provider 回退链并控制 analyst 并发 | 并发不是越高越好，过高会放大不稳定性 |
| 长任务重启成本高 | 优化后如果不重启，新配置无法生效；如果立即重启，又担心损失进度 | 等 checkpoint 落盘后再重启，并使用 supervisor 托管 | 长任务优化必须和恢复机制一起设计 |
| 容易误判瓶颈 | 表面上看像是“程序慢”，实际上是外部数据和 LLM 链路慢 | 从代码执行路径和 timing 拆阶段定位 | 不要把所有慢都归因于框架本身 |
| 优化目标容易分散 | baseline 和 MVP 都慢，但严重程度不同 | 先用已知 timing 证明 baseline 位置，再把主目标收敛到 MVP | 优化要聚焦主矛盾，而不是平均用力 |

### 9.6 当前保留下来的方法论

从这几轮优化里，已经可以沉淀出几条后续应继续遵守的方法：

| 方法 | 含义 |
|------|------|
| 先证据，后判断 | 先读 timing、checkpoint、代码路径，再决定往哪里优化 |
| 先减规模，再提并发 | 先减少总工作量，再考虑并发，不要反过来 |
| 先删重复，再谈提速 | 去重通常比盲目加速更稳、更便宜 |
| 优化必须可恢复 | 长任务中的性能改造，必须考虑重启与续跑机制 |
| 结论要区分层级 | 哪些是已被 timing 证明，哪些只是高概率推断，要明确写出来 |

## 10. 当前阶段性结论

### 10.1 目前已经能确定的结论

1. 本轮测试只包含 1 个 window，不是多窗口长期串行任务。
2. baseline 已完成，当前已进入 MVP。
3. baseline 的最大瓶颈已经被 timing 直接证明为 precise_agent。
4. baseline Top N 收缩确实带来了可观但不是数量级的改善。
5. MVP 当前最值得继续优化的方向，不是回测框架，而是前置候选打分阶段的大量串行外部数据抓取。
6. 从当前业务目标看，baseline 已经不是核心矛盾，MVP 才是下一阶段的主战场。

### 10.2 当前不应误判的几点

| 常见误判 | 实际情况 |
|----------|----------|
| “程序卡死了” | 不是，当前仍在持续推进外部数据与分析流程 |
| “主要是 checkpoint 慢” | 不是，checkpoint 只是结果落盘，不是主要耗时来源 |
| “主要是 compile 慢” | 不是，compile 缓存属于小优化，不是核心瓶颈 |
| “只要再加并发就能解决” | 不一定，过高并发会进一步推高 429 与回退概率 |

## 11. 下一步建议

### 11.1 下一阶段目标

下一阶段的目标需要明确收敛为一句话：

> 在不明显破坏 MVP 策略有效性的前提下，把当前 MVP 的总运行时间压缩到 3 至 4 小时以内。

这意味着后面的优化不再只是“尽量提速”，而是要围绕一个明确时限来做取舍。

### 11.2 路线调整原则

下一阶段建议采用下面这条路线：

| 原则 | 含义 |
|------|------|
| baseline 暂缓 | baseline 目前约 1 小时，先不继续深挖 |
| MVP 优先 | 所有主要优化动作优先服务于 MVP |
| 先砍规模，再砍串行，再砍 LLM | 先减少要处理的股票数，再减少前置串行抓取，再压缩进入 LLM 的样本 |
| 先做高收益改动 | 优先做不需要大改架构、但能明显缩短时长的改动 |

### 11.3 MVP 优化主路线

从当前证据看，下一步建议按下面四条主路线推进。

| 路线 | 目标 | 预期收益 | 优先级 |
|------|------|----------|--------|
| 路线 A：缩小候选池 | 降低后续所有步骤的总工作量 | 非常高 | P1 |
| 路线 B：重做 Layer B 打分链路 | 解决串行数据抓取过重问题 | 非常高 | P1 |
| 路线 C：压缩进入 fast / precise agent 的股票数 | 直接削减 LLM 成本 | 高 | P1 |
| 路线 D：收缩新闻与事件链路覆盖范围 | 降低高成本外部数据依赖 | 中高 | P2 |

### 11.4 建议的实施顺序

#### 第一阶段：先把 MVP 的工作量做小

这一阶段不追求“更聪明”，先追求“更少做事”。

| 动作 | 目的 |
|------|------|
| 收紧 candidate pool 入池条件 | 减少进入 Layer B 的股票数量 |
| 提高前置过滤阈值 | 让明显弱候选更早出局 |
| 降低进入 fast agent 的股票数 | 减少多分析师调用次数 |
| 降低进入 precise agent 的股票数 | 控制最昂贵环节的样本规模 |

这一步的核心思想是：

> 如果后面每只股票都很贵，那就先确保只有更少、更值得的股票能走到后面。

#### 第二阶段：把 Layer B 从重串行改成轻量流程

这一步是 MVP 优化的真正关键。

| 动作 | 目的 |
|------|------|
| 检查价格、财务、新闻、交易数据是否都必须逐只抓取 | 去掉低价值数据依赖 |
| 拆分“硬过滤”与“精细打分” | 先用便宜规则淘汰大部分候选 |
| 评估受控并发抓取 | 避免完全串行导致线性膨胀 |
| 优先缓存同日复用的数据 | 减少同一交易日重复加载 |

如果这一段不动，MVP 很难从十几个小时级别降到 3 至 4 小时。

#### 第三阶段：缩小 LLM 分析面

在候选池和 Layer B 缩小以后，再继续压缩 LLM 进入条件。

| 动作 | 目的 |
|------|------|
| 提高 fast agent 触发门槛 | 避免中等质量标的进入分析 |
| 提高 precise agent 触发门槛 | 只保留最值得深挖的少量股票 |
| 检查 fast 与 precise 的职责边界 | 避免重复判断与过度分析 |

#### 第四阶段：重新测量并决定是否保留复杂链路

如果前面三步之后仍然明显超时，就要开始做价值审查：

| 问题 | 决策方向 |
|------|----------|
| 新闻链路是否真的显著提升结果 | 如果没有，就收缩或降级 |
| 情绪链路是否值得当前成本 | 如果收益有限，就减少覆盖 |
| 某些财务或交易特征是否只是增加耗时 | 如果边际收益低，就删减 |

### 11.5 建议的优先级顺序

| 优先级 | 建议 | 理由 |
|--------|------|------|
| P1 | 缩小 candidate pool 与 Layer B 输入规模 | 这是所有后续耗时的放大器 |
| P2 | 重构 score_batch，降低串行外部抓取成本 | 当前最可能的 MVP 主瓶颈 |
| P3 | 压缩 fast / precise agent 的进入数量 | 直接降低模型调用时间 |
| P4 | 审查新闻、情绪、事件链路的边际收益 | 这类链路通常又慢又贵 |
| P5 | 等第一条完整 MVP timing 落盘后复核路线 | 用新证据校正优化顺序 |

### 11.2 可供下一轮审阅重点讨论的问题

1. MVP 的候选池是否需要更强的前置过滤。
2. score_batch 是否应采用受控并发，而不是完全串行。
3. 新闻与事件情绪链路是否需要缩小覆盖范围。
4. 在当前 provider 速率限制下，是否存在比继续加并发更有效的提速方式。
5. 为了把 MVP 压到 3 至 4 小时以内，哪些特征链路可以接受降级，哪些不能动。

## 12. 结语

当前这轮 A/B 测试之所以耗时长，并不是因为目标不清晰，而恰恰是因为目标很清晰：

它要在真实窗口里比较一套简单 baseline 与一套复杂 MVP 的价值差异。

这使得它天然包含两类昂贵成本：

1. 对照组中的多分析师精确分析成本。
2. 主策略中的候选池前置数据抓取与分层分析成本。

现阶段最重要的不是再猜“哪里慢”，而是继续把 MVP 的 timing 补齐，并在此基础上决定下一刀到底砍向：

1. 候选池规模。  
2. Layer B 打分链路。  
3. 新闻与事件情绪数据范围。  
4. LLM 进入条件。

这份文档的目标，不只是给出某一时刻的结论，而是持续保留优化轨迹，让后续的人工审阅和 AI 接手都能建立在同一组事实、同一条演化路径与同一套运行上下文之上。

## 13. 第六轮优化计划

这一轮开始，优化工作正式从“解释为什么慢”切换到“按目标时限压缩 MVP 时长”。

### 13.1 本轮目标

第六轮优化的核心目标如下：

| 项目 | 目标 |
|------|------|
| 主目标 | 将 MVP 总运行时间继续向 3 至 4 小时目标逼近 |
| 优先对象 | Candidate Pool、Layer B 打分链路、fast/precise agent 进入规模 |
| 原则 | 优先减少总工作量，而不是单纯提高并发 |
| 约束 | 不明显破坏 MVP 的核心有效性，不引入不可控的稳定性风险 |

### 13.2 本轮拟执行顺序

第六轮建议按下面顺序推进，而不是并行乱改。

| 顺序 | 动作 | 预期作用 |
|------|------|----------|
| 1 | 重新审查 candidate pool 入池条件 | 先把进入后续链路的股票数压小 |
| 2 | 拆分 Layer B 为“硬过滤”与“精细打分” | 让大量弱候选在便宜阶段提前出局 |
| 3 | 审查 score_batch 的外部数据依赖 | 去掉低价值但高耗时的数据抓取 |
| 4 | 评估 score_batch 的受控并发与缓存 | 减少串行线性膨胀 |
| 5 | 收紧 fast agent 与 precise agent 触发门槛 | 直接降低 LLM 调用规模 |

### 13.3 本轮计划解决的核心问题

当前 MVP 最大的问题，不是单点过慢，而是整条链路存在“前面放得过宽，后面又很贵”的结构性问题。

因此，第六轮要解决的不是局部加速，而是下面这三个结构性矛盾：

| 矛盾 | 当前表现 | 本轮方向 |
|------|----------|----------|
| 候选池过大 | 后续所有步骤都被放大 | 先收紧入口 |
| Layer B 过重 | 大量候选在昂贵数据链路里串行处理 | 先便宜过滤，再精细打分 |
| LLM 进入过宽 | fast 与 precise 的总成本过高 | 提高进入门槛 |

### 13.4 本轮验证方式

为了避免“改了很多，但不知道哪一步有效”，第六轮每一步都应配套验证。

| 验证项 | 验证方式 |
|--------|----------|
| 候选池是否变小 | 对比优化前后的 candidate 数量 |
| Layer B 是否变轻 | 对比 score_batch 的总耗时与单股票平均耗时 |
| LLM 工作量是否下降 | 对比进入 fast/precise 的股票数与请求次数 |
| 运行是否更稳 | 检查是否引入新的限流、报错、回退异常 |
| 策略是否明显失真 | 观察核心信号数量和结果分布是否异常塌缩 |

### 13.10 第六轮第二批实现方式

第二批改造的关键思想不是“让每一步都更快”，而是“不要让每只股票都走完整条重路径”。

当前实现已经调整为：

| 阶段 | 当前行为 |
|------|----------|
| 轻量阶段 | 所有候选先只计算 trend 与 mean_reversion |
| 临时排序 | 用轻量信号计算 provisional score，对候选重新排序 |
| 重财务阶段 | 只有 provisional score 达标、且排在前列的候选才进入 `fundamental` |
| 重事件阶段 | 只有更小一层候选才进入 `event_sentiment` |

这一步的意义在于：

1. 价格类技术信号仍保留对全量候选的覆盖。
2. 重财务与重事件信号不再对每只股票默认执行。
3. 融合层仍能识别哪些信号被跳过，因为被跳过的重信号 completeness 为 0。

## 14. 第六轮代码级改造清单

这一节用于把第六轮优化路线进一步压缩到代码级入口，避免后续执行时再次从头定位。

### 14.1 当前 MVP 热路径定位

从当前代码实现看，MVP 的主要放大链路如下：

| 层级 | 文件 | 关键函数/位置 | 当前问题 |
|------|------|---------------|----------|
| Layer A | src/screening/candidate_pool.py | build_candidate_pool | 对剩余标的逐只计算 20 日均成交额 |
| Layer B | src/screening/strategy_scorer.py | score_batch | 对每只候选逐只执行 score_candidate |
| Layer B | src/screening/strategy_scorer.py | score_candidate | 每只股票都继续触发价格、财务、新闻、董监高多路数据加载 |
| Layer C 前 | src/execution/daily_pipeline.py | high_pool 过滤阈值 | score_b >= 0.35 后直接进入 fast agent |
| LLM 精筛 | src/execution/daily_pipeline.py | top_20 = ...[:20] | precise agent 默认仍会对 20 只做精细分析 |

当前最关键的结构性事实是：

> Layer A 先把全市场收缩为一个仍然不小的候选池，Layer B 再对这个候选池做逐票多数据源评分，随后又把较大的 high_pool 送入 fast agent，最后再选前 20 进入 precise agent。也就是说，开销是沿着整条链逐层放大的。

### 14.2 第一优先改造点

第六轮第一优先级不应直接改 LLM，而应先改最前面的放大量。

| 优先级 | 改造点 | 文件 | 原因 |
|--------|--------|------|------|
| P1 | 候选池数量控制 | src/screening/candidate_pool.py | 这是后面所有耗时的放大器 |
| P1 | score_batch 瘦身 | src/screening/strategy_scorer.py | 当前最重的串行数据抓取区 |
| P1 | fast/precise 入口收紧 | src/execution/daily_pipeline.py | 直接决定 LLM 工作量 |

### 14.3 候选池的具体改造入口

当前 Layer A 最重的一段是：对通过前几轮筛选后的所有股票，逐只调用 `_get_avg_amount_20d`，而 `_get_avg_amount_20d` 又逐只打 `pro.daily`。

因此，candidate pool 的具体改造建议如下：

| 改造项 | 方向 | 目的 |
|--------|------|------|
| 增加更强的前置粗筛 | 在逐只算 20 日均额前先砍掉一批显然不值得看的股票 | 减少 `_get_avg_amount_20d` 调用次数 |
| 优先使用当日 `daily_basic` 可直接得到的字段做硬过滤 | 把便宜过滤提前 | 降低逐票精算规模 |
| 把“是否进入 Layer B”与“候选池快照”分层 | 不必所有通过 Layer A 的票都进入 Layer B | 防止 Layer A 输出过宽 |

这一段的目标不是“更精确”，而是“更早淘汰明显不值得的标的”。

### 14.4 score_batch 的具体改造入口

当前 `score_batch` 的结构很直接，但也是问题所在：

1. 先构建行业估值中位数。
2. 再对每个 candidate 调用 `score_candidate`。
3. `score_candidate` 再分别计算 trend、mean_reversion、fundamental、event_sentiment。
4. fundamental 与 event_sentiment 又会继续触发外部数据读取。

这意味着每只股票的单位成本并不低，而 `score_batch` 现在完全按候选数线性增长。

建议优先从下面几处切入：

| 改造项 | 文件位置 | 方向 |
|--------|----------|------|
| 把 Layer B 拆成“硬过滤 + 精细打分”两段 | src/screening/strategy_scorer.py | 先用低成本规则剔除大部分候选 |
| 对 event_sentiment 做延后触发 | src/screening/strategy_scorer.py | 不让每只候选都读新闻和董监高数据 |
| 对 fundamental 做更轻量入口判定 | src/screening/strategy_scorer.py | 只有初筛通过后才进入更重的财务分析 |
| 为价格数据和同日数据增加复用 | src/screening/strategy_scorer.py | 减少同一交易日重复数据加载 |

其中，新闻与董监高数据链路应被视为高成本、可延后、可降级的目标，而不是默认对所有候选都执行。

### 14.5 daily_pipeline 的具体改造入口

`daily_pipeline.py` 当前有两个非常明确的数量门槛：

| 位置 | 当前行为 | 风险 |
|------|----------|------|
| `high_pool = [item for item in fused if item.score_b >= 0.35]` | 只要过 0.35 就全部进 fast agent | high_pool 可能过大 |
| `top_20 = sorted(high_pool, ... )[:20]` | precise agent 默认对前 20 做精筛 | 对 3-4 小时目标仍然偏重 |

这说明第六轮不一定需要先改复杂算法，先改门槛也可能立刻产生较大收益。

建议的入口方向如下：

| 改造项 | 目标 |
|--------|------|
| 提高 `high_pool` 进入门槛 | 降低 fast agent 总量 |
| 把 precise 的默认 Top 20 进一步收紧 | 控制最贵分析阶段 |
| 把数量上限做成可配置参数 | 便于快速做 A/B 性能实验 |

### 14.6 第六轮建议先改什么，不先改什么

为了降低回归风险，第六轮建议遵循下面的改造边界：

| 类型 | 建议 |
|------|------|
| 先改 | 阈值、数量上限、触发条件、延后触发逻辑 |
| 先改 | 不改变主接口形状的轻量缓存与过滤 |
| 暂缓 | 一次性重写整套打分框架 |
| 暂缓 | 在缺少直接测试时大幅改动 `score_batch` 返回结构 |
| 暂缓 | 单纯依赖更高并发来掩盖结构性问题 |

### 14.7 当前测试约束

从现有测试覆盖看：

| 区域 | 当前情况 | 含义 |
|------|----------|------|
| candidate_pool | 已有较完整测试 | 可以较安全地增加前置过滤与数量控制 |
| daily_pipeline | 有流程级测试 | 调整 high_pool / precise 规模时相对可控 |
| strategy_scorer | 缺少围绕 `score_batch` 的直接测试 | 第六轮应优先做保接口、减工作量型改造 |

因此，最合理的执行顺序是：

1. 先改 candidate pool 的前置过滤与数量控制。
2. 再改 daily_pipeline 的 fast/precise 数量门槛。
3. 然后再进入 `strategy_scorer.py` 做硬过滤、延后触发和轻量缓存。

### 14.8 第六轮的首批落地任务

如果马上进入代码实施，建议按下面这批任务启动：

| 任务 | 类型 | 目标 |
|------|------|------|
| 任务 A | Candidate Pool 收口 | 明显减少进入 Layer B 的股票数 |
| 任务 B | Fast/Precise 门槛收紧 | 立刻减少 LLM 请求规模 |
