# 2026-03-18 Capital Deployment 与 Concentration 专项

## 1. 任务结论

P0-2 的核心判断已经可以明确：当前系统不是单一问题，而是两个阶段性问题叠加。

1. 在当前默认基线 `baseline` 下，主问题更接近“缺可持续机会”，即 `300724` 被 re-entry guardrail 拦住后，没有第二批足够干净的替代票可以持续接力。
2. 在更早的 live pipeline runs 里，主问题又明显带有“有机会但持仓生命周期与仓配构造错误”的成分，尤其是 `300724` 的重复回补、`603993` 的长时间静态持有，以及长期偏高的现金占比并存。

所以，P0-2 的答案不是二选一，而是：

- 当前固定默认基线偏向“候选不足”
- 历史 live runs 偏向“生命周期与配仓构造错误”

## 3. 机会不足还是配仓错误

### 3.1 当前默认基线更接近机会不足

`baseline` 有三个很强的信号：

1. 平均利用率只有 `14.51%`，期末现金占比在 scoreboard 中达到 `94.19%`。
2. 唯一真正进入 watch / buy / executed 集合的 ticker 只有 `300724`、`601600`、`603993` 三只。
3. `600988`、`300065` 在整组长窗口里都没有进入 watchlist、buy order 或 executed trade，说明它们的问题根本不在仓配尾部，而在更上游的候选生成或语义过滤阶段。

这意味着：

- 当 `300724` 被 re-entry 确认规则拦住时，系统没有自然形成第二批高质量替代票。
- 当前默认基线的低利用率，首先不是因为买单构造把很多已经合格的票压没，而是因为真正能走到漏斗尾部的票本来就少。

### 3.2 历史 live runs 又明显存在配仓与生命周期错误

如果只看 `existing_position_fix` 到 `retrace6` 这些 live runs，问题又不只是机会不足。

关键证据有三条：

1. 这些 runs 的 `watchlist_days` 一直在 `8` 到 `11` 天，说明不是完全没候选。
2. 但 `holding_static_days` 长期在 `8` 到 `13` 天，占非空持仓日的大头，说明组合经常进入“有仓但不动”的静态状态。
3. `300724` 的重复买入极高：`exit_fix` 为 `x4`，`trading_days` 与 `retrace6` 仍有 `x3`，说明生命周期规则没有把它从高 churn 风险里稳定剥离出来。

因此，历史 live runs 的问题更像：

- 上游不是完全没有票
- 但进入组合后，仓位扩展不足、卖出后又容易回补同一弱票、现金长期闲置
- 组合没有把候选转成更均衡的资金展开

## 4. 300724 与 603993 的角色

### 4.1 `300724`：高频边缘机会，同时也是主要 churn / 风险源

从所有 runs 看，`300724` 的角色非常稳定：

1. 它几乎总是最常见的 watchlist ticker。
2. 它在 live runs 里不断重复买入：从 `x1` 到 `x4` 不等。
3. 它的峰值权重稳定落在 `13.49%` 到 `14.76%` 区间，是窗口内最容易形成单票主暴露的对象。

这说明 `300724` 不是纯噪音票，而是：

- 一个真实能反复穿透到漏斗尾部的边缘机会样本
- 但也是当前最容易触发重复回补、集中度和持仓质量问题的风险载体

在 `baseline` 中，它被 re-entry 保护压住后，系统收益变差但 guardrail 更干净；在 `prod_validation` 中，它重新成交后，利用率略升但直接破坏 guardrail。这正说明它是“机会与风险绑在一起”的核心样本。

### 4.2 `603993`：低弹性、可成交、偏静态的资金停泊位

`603993` 的行为模式和 `300724` 明显不同：

1. 它进入 watchlist 的天数始终只有 `2` 天左右，但一旦进来，往往能形成真实成交。
2. 它的峰值权重基本只在 `7%` 到 `7.6%` 左右，远低于 `300724`。
3. 它的持有天数却很长，多个 runs 都达到 `14` 到 `16` 天，是典型的低周转静态仓位。

所以 `603993` 更像：

- 一个成交成功率较高的“稳定填仓票”
- 能提供一定资金部署，但弹性有限
- 更容易把组合带入“仓位不大但长期不动”的低效率状态

换句话说：

- `300724` 是组合里的高波动边缘进攻位
- `603993` 是组合里的低弹性停泊位

当前系统的问题不在于二者都不工作，而在于：只有这两类角色在工作，而且第三类可持续补位角色还没有出现。

## 5. 600988 与 300065 的意义

这两个 ticker 在本轮长窗口统计中都有同一结论：

1. `watchlist_days = 0`
2. `buy_order_days = 0`
3. `executed_buy_days = 0`

因此它们在 P0-2 里的作用不是解释配仓尾部，而是证明当前系统上游候选供给仍然很窄。

也就是说：

- `600988`、`300065` 不是“有机会却在仓位阶段被错杀”的主证据
- 它们更像“压根没进入可部署层”的机制样本

## 6. 最小归因结论

P0-2 可以先收口成三条判断：

1. 当前默认基线的首要问题是候选不足。`300724` 被拦后，系统没有足够多的 clean 替代票接力。
2. 历史 live runs 的首要问题是生命周期和配仓错误。`300724` 的重复回补与 `603993` 的静态持有共同把组合锁在“低利用率 + 低多样性”的状态里。
3. 现在不适合用“放宽全局闸门”解决问题。因为这更可能放出 conflict/suppression 样本，而不是自然补出第三类高质量部署对象。

## 7. 对 P1 的直接指向

这份专项已经把后续优先级压得更清楚：

1. P1-1 re-entry 质量专项应继续盯住 `300724`，因为它是机会质量与 churn 风险的交汇点。
2. P1-2 lifecycle 复盘应优先拆 `300724` 与 `603993` 的持仓演化差异，一个代表重复回补，一个代表静态停泊。
3. 在新增 clean 样本前，不建议把问题描述成“只要再多一点成交就能解决”。当前缺的是第三类可持续部署角色，而不是单纯多做几笔同质交易。
