# Layer B 最小规则变更提案

文档日期：2026 年 3 月 12 日  
适用范围：默认 CLI / backtester pipeline 的 Layer B 评分与融合逻辑  
文档定位：分析与设计提案，不直接修改生产规则

---

## 1. 这份提案要解决什么问题

前面的 20 日窗口扫描已经把问题收敛得比较清楚了：

1. 当前交易覆盖不足的主收缩点在 Layer B fast gate，而不是 Layer C、watchlist 或下单层。
2. 20 个交易日里有 17 天 `layer_b_count = 0`。
3. 整个窗口只有 4 次 Layer B 通过，存在明显的 `0.35 ~ 0.38` 阈值拥挤。
4. 问题不是候选池太小，而是大量边缘样本卡在 `score_b >= 0.38` 之前。

因此，这份提案不讨论 workflow 重构，也不讨论 provider 速度，而只讨论：

**如何用最小规则改动，先缓解当前 Layer B 里已经被证据确认的两个结构性问题。**

---

## 2. 提案边界

### 2.1 本提案明确要做的事

1. 只讨论 Layer B 内部评分与融合语义。
2. 只针对已经被反事实验证过的两个高优先级问题提出候选改动。
3. 只考虑“最小化业务改动”，不做整套新评分框架。
4. 为后续实现和回测提供一个收敛的设计基线。

### 2.2 本提案明确不做的事

1. 不直接下调 `FAST_AGENT_SCORE_THRESHOLD`。
2. 不修改 Layer C 聚合逻辑。
3. 不处理 rolling scheduler、并发调度、provider 路由问题。
4. 不把 `event_sentiment` 缺席误判为纯策略问题，因为当前 A 股回测里它主要是历史新闻可得性问题。
5. 不把窗口级极端反事实结果直接当作上线目标。

---

## 3. 为什么只优先提这两个改动

当前证据已经足够支持两个优先级最高的问题：

1. `profitability` 存在明显 cliff 语义。
2. `mean_reversion` 的中性完整度会在归一化层面稀释正向策略。

这两个问题有一个共同点：

**它们都不是简单调数字就能解决，而是牵涉到当前聚合语义本身。**

也正因为如此，继续做“把 confidence 调小一点”或“把 completeness 改成 0.5”这类直觉调参，业务收益很低，甚至可能让结果更差。

---

## 4. 已确认的证据基础

### 4.1 profitability cliff 已被样本反事实验证

代表性样本：

1. `300065` 在 20260223 至 20260225 连续三天都停在 `0.3735 ~ 0.3739`。
2. `600111` 在 20260204 停在 `0.3454`。

共同特征：

1. `ROE`、`net_margin`、`operating_margin` 三项全部未达当前阈值。
2. `profitability` 直接变成 `direction = -1`、`confidence = 100` 的硬负项。

关键反事实结果：

1. 直接移除这个硬负项后，`300065` 会抬到约 `0.3870 ~ 0.3874`。
2. `600111` 会从 `0.3454` 抬到 `0.3803`。
3. 但如果只是把这个负项的 `confidence` 从 `100` 改到 `40`，分数反而进一步下降。

这说明当前问题不是“confidence 太大”，而是“0 项达标直接强负”的语义过陡。

### 4.2 neutral mean_reversion dilution 已被窗口级验证

代表性样本：

1. `688498` 在 20260205 的 `trend + fundamental` 都为正，但最终停在 `0.3516`。
2. 该样本的 `mean_reversion` 整体是 `direction = 0`，但 `completeness = 1.0`。

关键反事实结果：

1. 单样本移除中性 `mean_reversion` 后，`688498` 从 `0.3516` 升到 `0.4688`。
2. 更保守的窗口级边界测试里，只要整条 `mean_reversion` 满足 `direction = 0 且 completeness > 0` 就排除出归一化，Layer B 通过数会从 `4` 跳到 `69`。
3. 但更温和的“把 completeness 从 `1.0` 改成 `0.5` 或 `0.25`”完全无效。

这说明当前问题不是 completeness 数值大小，而是：

**当前实现里，只要 `completeness > 0`，这条策略就在归一化里被当作 fully active。**

---

## 5. 提案一：先收敛 profitability 的硬负项触发语义

### 5.1 目标

减少“其它因子并不弱，但因为 0 项达标被直接打成强负”这类 cliff 式误伤。

### 5.2 不建议的做法

不建议直接把 `profitability` 的负项 `confidence` 从 `100` 改成更低数值。

原因很简单：

1. 现有 `aggregate_sub_factors()` 不是线性求和。
2. 降低负项 confidence 会同时改变子策略内部的方向一致性语义。
3. 已有样本证明这种改法可能让结果更差。

### 5.3 建议的最小候选改动

当前更合理的最小改动，不是“削弱负项”，而是“缩小硬负项命中面”。

建议优先只讨论下面两个候选方向中的一个：

1. 方案 A：把 `0 项达标 => 强负` 改成 `0 项达标 => 中性`。
2. 方案 B：保留负向，但只在三项都明显低于阈值且满足更严格的极差条件时才给强负。

在“最小改动”原则下，我更倾向把方案 A 作为第一验证优先级，原因是：

1. 语义最简单。
2. 最容易解释和回测。
3. 它只改变最陡的 cliff 分支，不会重写整个 profitability 规则。

### 5.4 推荐的初版实现边界

如果后续进入代码实现，建议初版只做这一件事：

1. 仅修改 `profitability` 中 `0 项达标` 这一支的输出语义。
2. 不改 `2 项及以上达标 => 正向`。
3. 不改其它 fundamental 子因子。
4. 不改 Layer B 总阈值。

这样可以保证：

1. 改动范围小。
2. 能单独观察 `300065`、`600111` 一类样本是否被合理释放。
3. 不会把问题混入其它策略变更里。

### 5.5 风险

这个改动的主要风险是：

1. 可能释放一批“确实盈利质量偏弱”的股票。
2. 其中一部分只是因为缺少硬负项而过线，并不一定真该入池。
3. 如果与 volatility 放松同时上线，可能难以区分归因。

所以这里建议：

1. 单独做反事实。
2. 单独做 20 日窗口回放。
3. 单独看新增过线样本的行业和基本面结构。

---

## 6. 提案二：收紧 neutral mean_reversion 的 active 资格

### 6.1 目标

减少“方向为 0、但完整度大于 0 的中性策略仍占用归一化席位”造成的结构性稀释。

### 6.2 不建议的做法

不建议先做下面这些改法：

1. 把中性 `mean_reversion` 的 completeness 从 `1.0` 改到 `0.5`。
2. 把中性 `mean_reversion` 的 completeness 从 `1.0` 改到 `0.25`。

原因已经被窗口级反事实证明：

1. 当前 `_normalize_for_available_signals()` 只看 `completeness > 0`。
2. 所以 `0.25`、`0.5`、`1.0` 对 active/inactive 判定没有区别。
3. 这种改法几乎不会产生业务效果。

### 6.3 建议的最小候选改动

更合理的最小改动不是调 completeness 数值，而是收紧 active 资格条件。

建议优先验证这个候选语义：

1. 当 `mean_reversion.direction == 0` 时，不让它进入归一化 active 集合。

这是当前最小、最直接、也最符合证据的问题定义。

### 6.4 为什么这里只先动 mean_reversion，不直接改全部中性策略

因为当前强证据只集中在 `mean_reversion`：

1. 20 日窗口级边界测试已经显示它的中性完整度非常普遍。
2. 边缘样本上它的中性稀释效应非常明显。
3. `event_sentiment` 当前还混有历史新闻可得性约束，不适合直接把所有中性策略一刀切出归一化。

所以更稳妥的最小方案是：

1. 先做 `mean_reversion` 专项。
2. 不扩展为“所有 `direction = 0` 的策略都退出归一化”。

### 6.5 推荐的初版实现边界

如果后续进入代码实现，建议初版只做这一件事：

1. 在 Layer B 归一化阶段，对 `mean_reversion` 增加一个更严格的 active 资格判断。
2. 仅当它方向非零时，才参与 active 权重归一化。
3. 不改它自己的子因子计算。
4. 不改其它策略的 active 逻辑。

### 6.6 风险

这个改动的主要风险是：

1. 窗口级边界测试显示影响量级非常大，说明这里高度敏感。
2. 即便只改 `mean_reversion`，也可能显著放大 `trend` 与 `fundamental` 的占比。
3. 如果没有额外防护，Layer B 通过率可能出现跃迁式上升。

因此，虽然这项改动方向明确，但上线前必须加保护：

1. 先做反事实回放。
2. 观察新增过线样本是否过于集中在单一风格。
3. 必要时配合更小的显式门槛或白名单验证，而不是直接全量替换。

---

## 7. 为什么现在不建议直接改 fast gate

从表面看，`0.38` 附近的阈值拥挤很明显，所以最直觉的动作是下调 `FAST_AGENT_SCORE_THRESHOLD`。

但当前不建议先这样做，原因有三点：

1. 现在已经定位出两个更上游的结构问题。
2. 直接降阈值会把“规则误伤”和“真正弱票”一起放进来。
3. 一旦先改阈值，后面再改 profitability 或 mean_reversion，就很难分清每一步到底释放了什么。

因此，更合理的顺序是：

1. 先修正已经明确的语义问题。
2. 再看 `0.38` 是否仍然拥挤。
3. 如果修完以后仍然系统性卡在 `0.35 ~ 0.38`，再讨论 fast gate 是否需要校准。

---

## 8. 推荐的验证顺序

为了让后续实现和回测尽量可解释，建议按下面顺序推进。

### 第一步：单独验证 profitability 语义改动

目标：只回答一个问题。

1. 当 `0 项达标` 不再直接输出强负后，`300065`、`600111` 这类样本会被释放多少。

关注指标：

1. 20 日窗口 Layer B 通过数变化。
2. 新增过线样本的行业分布。
3. 新增样本里 Layer C / watchlist 后续是否还能继续过滤掉明显弱票。

### 第二步：单独验证 neutral mean_reversion active 收紧

目标：只回答一个问题。

1. 当中性 `mean_reversion` 不再占 active 归一化席位时，窗口通过率会增加多少，是否出现过量释放。

关注指标：

1. 20 日窗口新增过线总量。
2. 新增过线样本是否集中在纯 trend + fundamental 双腿结构。
3. 新增样本是否明显偏向高波动趋势股。

### 第三步：再决定是否需要联合版本

只有前两步都单独看过后，才建议评估联合版本：

1. profitability cliff 缓解
2. neutral mean_reversion active 收紧

否则，联合测试虽然快，但解释力会很差。

---

## 9. 推荐的落地节奏

如果后续进入实现阶段，建议按下面节奏推进：

1. 先做分析分支上的反事实实现，不改默认生产规则。
2. 先跑当前已经使用过的 20 日窗口，保证和已有证据可对齐。
3. 再补 1 个额外窗口，避免只对 2026-02 有效。
4. 确认新增通过样本的结构后，再决定是否进入默认 pipeline。

这个节奏的核心价值在于：

1. 先验证语义。
2. 再验证稳定性。
3. 最后才讨论是否变成默认行为。

---

## 10. 当前建议的优先级排序

基于现有证据，我建议的优先级是：

1. 第一优先级：`profitability` 的硬负项触发语义。
2. 第二优先级：`mean_reversion` 的中性 active 资格。
3. 第三优先级：`trend.volatility` 是否对趋势股过于激进。
4. 第四优先级：在完成前三项前，不讨论 fast gate 总体下调。

其中第三优先级虽然也有实证支持，但暂时排在后面，原因是：

1. volatility 更像业务校准问题。
2. profitability 和 neutral mean_reversion 更像实现语义问题。
3. 先修语义，再调参数，顺序更稳。

---

## 11. 一句话总结

如果只用一句话概括这份提案，那就是：

**当前最值得优先验证的，不是降低 Layer B 总阈值，而是先修正两个已经被证据确认的语义问题：profitability 的硬负项 cliff，以及 neutral mean_reversion 的 active 稀释。**
