# 系统整改路线图

> 目标：把“发现的问题”转成“可执行的整改顺序”，避免同时开太多战线。

---

## 总体原则

整改应遵循四条原则：

1. **先 correctness，后 elegance**：先修会导致错误结论和安全事故的问题
2. **先收紧边界，再做优化**：先补安全、校验、错误治理，再谈性能和重构
3. **先建立机制，再清理细节**：先有 CI、类型、日志、测试门禁，再逐步还债
4. **先消除不确定性，再扩功能**：先把依赖、配置、状态边界统一

---

## 第一阶段：立即修复

### 目标

阻断高危漏洞、错误交易信号和明显的不可信输出。

### 必做项

1. **封堵 Web 安全高危项**
   - 禁止生产环境使用 `AUTH_DISABLED`
   - 去掉 JWT 默认密钥 fallback
   - 修复 `storage.py` 路径遍历
   - API key 接口只返回脱敏值

2. **收紧 agent 输出可信度**
   - 禁止在缺失原始数据时输出确定性结论
   - 将“无数据”“数据异常”“推理失败”区分为不同错误类型
   - 为关键 agent 输出绑定证据来源字段

3. **停止全局 monkey patch**
   - 去掉 `akshare_api.py` 对 requests 全局方法的覆盖
   - 改为实例级 session 或局部 transport 策略

4. **建立最小发布门禁**
   - 补 `.github/workflows/`
   - 至少运行：安装、pytest、基础 lint、前端构建

### 阶段完成标准

- 不再存在认证绕过和路径遍历
- 不再向客户端返回真实 API key
- 不再依赖全局 HTTP 行为篡改
- 主分支有最小自动化质量检查

---

## 第二阶段：短期治理

### 目标

提升系统稳定性，让失败变得可观测、可定位、可恢复。

### 必做项

1. **异常治理**
   - 替换核心路径中的 bare `except` / `except Exception`
   - 统一错误类型与日志上下文
   - 区分 provider 错误、数据校验错误、业务错误

2. **缓存与并发治理**
   - 为 `_tushare_df_cache` 增加 TTL 和 size limit
   - 盘点全局共享状态，明确线程安全边界
   - 控制多 ticker / 多 agent 并发抓取策略

3. **前端稳定性治理**
   - 收紧 flow / node / edge 类型
   - 补全关键 effect 依赖
   - 给重型组件增加 memo 化
   - 增加全局 error boundary

4. **配置治理**
   - 统一只保留一种包管理工具
   - 增加环境变量启动校验
   - 明确哪些变量是必需、可选、开发专用

### 阶段完成标准

- 主要失败路径都有可读日志
- 长时间运行时内存增长可控
- 前端复杂 flow 操作不再明显卡顿
- 环境安装与运行过程可重复

---

## 第三阶段：中期演进

### 目标

把系统从“原型可用”推进到“可持续维护”。

### 必做项

1. **后端分层重构**
   - 收敛 route / service / repository 职责
   - 定义统一错误码与 HTTP 映射
   - 为敏感操作加入审计日志

2. **数据契约建设**
   - 给 provider 输出定义统一 schema
   - 增加数据完整度与可信度字段
   - 为 agent 输出增加“证据覆盖率”概念

3. **测试体系补强**
   - 前端补 contexts、hooks、核心交互测试
   - 后端补安全与事务边界测试
   - 给关键 agent/数据链路增加回归测试

4. **基础设施升级**
   - 完善 Docker 构建与 `.dockerignore`
   - 引入 pre-commit、coverage threshold、安全扫描
   - 整理根目录调试脚本到专门目录

### 阶段完成标准

- 关键模块职责清晰
- 数据链路具备显式契约
- 代码变更有回归保护
- 工程流程形成稳定质量门

---

## 不建议的做法

以下做法看似能“快速推进”，实际上会放大问题：

1. 在未修正可信度问题前继续新增更多 agent
2. 在未统一依赖工具前继续叠加构建脚本
3. 在未补错误治理前仅靠 prompt 微调修复幻觉
4. 在未做类型收紧前继续扩展复杂前端节点

---

## 建议 owner 视角的优先级

如果从负责人角度只挑最值钱的 5 件事，应优先：

1. 修 Web 安全高危项
2. 修 LLM 事实可信度
3. 建最小 CI
4. 统一依赖/配置治理
5. 控制缓存、并发与前端核心类型

---

## 最终判断

当前项目已经跨过“功能验证”阶段，正在进入“工程可靠性”阶段。这个阶段最忌讳继续靠局部脚本和临时补丁前进。更有效的策略是：**先建立边界、机制和门禁，再继续扩展功能。**
