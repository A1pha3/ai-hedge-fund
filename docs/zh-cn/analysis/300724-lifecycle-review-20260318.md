# 2026-03-18 300724 Lifecycle 复盘

## 1. 结论摘要

P1-2 可以先收口为一个很具体的判断：`300724` 当前仍是固定基线里的主要亏损源，但它的问题主轴已经不是“仓位失控”，而是“边缘入场质量不足 + 止损后的回补确认仍然偏弱”。

按任务清单要求的四个问题，当前证据对应的答案如下：

1. 买点是否合理：`不够合理`。两段主要开仓都落在 watchlist edge 附近，`entry_score` 大致只在 `0.20 .. 0.225` 或 `0.225` 左右，只有 `20260206` 这一腿曾短暂抬到 `0.2791`，但后续仍很快失效。
2. 退出是否过晚：`不是主要问题`。两段完整持仓都几乎没有形成像样的顺风段，最大浮盈仅约 `0.00% .. 1.51%`，随后都落到 `hard_stop_loss`；当前更像是在切断弱仓，而不是把强仓拿到过晚。
3. 重进是否过快：`是`。无论是 `exit_fix` 的 `1` 天回补，还是 `prod_validation` / `retrace6` 里 `0.20x ~ 0.2250` 的边缘回补，都没有被证明能改善收益质量。
4. 加仓是否受控：`在当前已接受基线里基本受控`。固定基线、`prod_validation`、`trading_days`、`retrace6` 都只保留每腿 `100` 股的单次建仓，没有再出现早前已收口的连续金字塔加仓问题。

因此，P1-2 的定位应是：`300724` 的主要损失先来自信号质量，其次来自弱确认 re-entry；退出只是把已经证伪的弱仓切掉，不是当前第一优先级矛盾。

## 3. 首轮建仓为什么已经说明“买点偏弱”

### 3.1 `20260204 -> 20260205` 的首腿没有任何容错空间

在 `exit_fix`、`cooldown`、`trading_days`、`retrace6` 四组 replay 里，`300724` 的第一腿都发生在 `20260204`，共同特征非常一致：

1. `entry_score` 都只在 `0.2027 .. 0.2228`。
2. 入场即面临次日大跌，从 `147.70` 直接到 `129.40`。
3. 持仓记录中的 `max_unrealized_pnl_pct = 0.0`，意味着这笔仓位在真实持有期里没有形成任何可兑现的顺风段。
4. 结果都是同一笔 `-12.39%` 的硬亏损。

这说明首腿的核心问题不是 exit checker 响应慢，而是买点本身过于贴近边缘阈值，几乎没有给后续风险控制留出缓冲。

### 3.2 `20260206 -> 20260213` 的首腿质量高于 edge 入场，但仍然不够好

在 `baseline` 与 `prod_validation` 里，首腿改成了 `20260206`：

1. 入场价格降到 `132.61`，`entry_score = 0.2791`，比 `20260204` 的边缘开仓明显更高。
2. 持仓期内最高浮盈也只有 `1.22%`，且峰值只出现在 `20260209`。
3. 随后价格连续回落：`134.63 -> 129.98 -> 127.52 -> 127.19 -> 122.70`。
4. 同期 logic score 也从 `0.2059` 继续衰减到 `0.1860`、`0.1856`、`0.1821`。

因此，这条首腿可以说明两件事：

1. 更高一些的入场分数确实把损失从 `-12.39%` 压到了 `-7.47%`。
2. 但它仍不是高质量入场，因为价格和打分都没有形成持续性确认，仓位仅获得约 `1.22%` 的最大顺风后就重新转弱。

## 4. 退出是否过晚

现有证据更支持“退出是在切弱仓”，而不是“把强仓拿太久”：

1. 两条完整止损腿的最大浮盈都很低，分别约为 `0.00%` 与 `1.22%`。
2. `20260206` 这条持仓在 `20260209` 后价格持续走低，同时 logic score 连续跌破 `0.19`。
3. `20260225` / `20260226` 开始的第二腿，在 `trading_days`、`retrace6`、`prod_validation` 里最高浮盈也只有 `0.00% .. 1.51%`，随后迅速转回亏损。

如果一笔持仓从未形成中高质量利润区间，那么当前 `hard_stop_loss` 更像是最后一道止损闸门，而不是主要的净值拖累来源。P1-2 下这意味着：退出可以继续优化，但它不是第一性问题。

## 5. 重进为什么仍然被判定为偏快

### 5.1 过快不只指“时间短”，更指“确认不足”

`300724` 的重进问题不能只看冷却天数，还要看回补时的信号质量是否显著改善。当前并没有看到这种改善：

1. `exit_fix` 在 `20260205` 卖出后，`20260206` 就再次买回，属于最明显的低质量 churn。
2. `cooldown` 把回补延后到 `20260213`，但第二腿 `entry_score` 只有 `0.2005`，到窗口末仍是 `-5.73%` 的弱仓。
3. `trading_days` 与 `retrace6` 把回补推迟到 `20260225`，但 `entry_score` 也只在 `0.2022 .. 0.2047`，最终仍在 `20260304` 第二次 `hard_stop_loss`。
4. `prod_validation` 在 `20260226` 的回补分数虽然抬到 `0.2250`，但也只拿到 `1.51%` 的最大浮盈，期末仍是 `-5.44%` 浮亏，并且结果劣于不回补的 `baseline`。

所以，P1-2 对“重进是否过快”的判断应写成：`当前主要是确认过快，而不是单纯时间过快`。哪怕回补被拖到 13 天或 20 天，只要仍是 edge score 回补，它就没有被证明是高质量行为。

### 5.2 baseline 为什么仍是默认口径

`baseline` 的意义不在于证明 300724 永远不能再买，而在于：

1. 当前窗口内，没有任何一条回补腿被证明优于“不回补”。
2. `20260226 / 300724` 已被 benchmark 固定为 guardrail 样本。
3. 默认阻断弱确认回补，至少能避免已知的额外劣化。

## 6. 加仓是否受控

在 P1-2 当前范围内，这个问题的答案可以明确为 `是`，且无需重开既有 runtime 结论：

1. `baseline`、`prod_validation`、`trading_days`、`retrace6` 的每一条有效腿都只保留 `100` 股单次建仓。
2. 没有再出现早前已收口的“持有期间继续金字塔加仓到失控”的现象。
3. 即便把 300724 压到最小可成交规模，它仍然能独立形成 `-5% ~ -12%` 的亏损腿。

这说明当前剩余问题已经不是“仓位系统没拦住加仓”，而是“即便在受控仓位下，这只票的入场和回补质量仍然不足”。

## 7. 根因归位

P1-2 可以把 300724 的损失来源按优先级归为三层：

1. `第一层：信号质量`。最差的几条腿都起始于 watchlist edge 附近的弱确认开仓，入场后几乎没有顺风区间。
2. `第二层：re-entry 确认不足`。首腿止损后，系统仍会在 `0.20x ~ 0.2250` 的边缘区间尝试回补，而这些回补没有展示正收益质量。
3. `第三层：exit 只是兜底`。`hard_stop_loss` 的确在每条坏腿的末端触发，但它更像在承认前两层已经失败，而不是当前主要可优化点。

按任务清单第二条验收标准，当前最接近事实的描述是：`300724` 的剩余亏损主要来自信号质量，其次来自 re-entry 设计；组合构造与加仓约束在这一阶段已退到次要矛盾。

## 8. 对后续任务的约束

1. P1-3 如果要做 taxonomy，`300724` 应被放进“edge score weak-entry / weak-reentry”而不是“exit too late”类别。
2. 后续局部改进方向应继续围绕“更强回补确认”或“更高最小实际买入分数”，而不是重新放松全局闸门。
3. 在没有新的 clean edge 替代票之前，默认基线继续维持 `baseline` 的阻断策略更稳妥。
