## 4.1 Web 应用概述

### 为什么需要 Web 界面

命令行适合自动化和批量运行，但在以下场景中并不高效：

- 需要频繁切换股票、时间范围和模型。
- 需要对比多个智能体的观点差异。
- 需要把结果交给研究员或非开发角色复核。
- 需要查看历史记录、图表和研究反馈。

Web 界面的核心价值是把“运行系统”变成“操作工作台”。它并不是替代命令行，而是把以下任务做得更直观：

- 参数配置。
- 结果浏览。
- 历史追踪。
- 选股复核。
- 研究反馈回写。

### 前后端分离结构

Web 应用采用前后端分离架构：

```text
浏览器
  |
  | HTTP / JSON
  v
前端应用（Vite + React）
  |
  | 调用 API
  v
后端服务（FastAPI）
  |
  | 调用数据源、LLM、回放文件
  v
业务逻辑与数据层
```

这个结构有三个直接好处：

- 前端迭代快，适合持续调整页面交互。
- 后端职责清晰，便于调试 API 和数据流。
- 同一套 API 可以被 Web、脚本或其他工具复用。

### 两类常见使用场景

| 场景 | 适合页面 | 主要目标 |
| --- | --- | --- |
| 实时或准实时分析 | Stock Analysis | 输入标的并即时得到分析结果 |
| 复盘与研究复核 | Replay Artifacts | 查看历史回放、候选股、blocker 和反馈 |

如果你当前的目标是“如何在页面里做选股复核、near-miss 判读和 blocker 分析”，本章只负责带你找到入口，具体操作请继续阅读：

- [Replay Artifacts 选股复核 5 分钟速查](../manual/replay-artifacts-stock-selection-quickstart.md)
- [Replay Artifacts 选股复核页面使用手册](../manual/replay-artifacts-stock-selection-manual.md)
- [Replay Artifacts 研究反馈标签规范手册](../manual/replay-artifacts-feedback-labeling-handbook.md)

## 4.2 启动 Web 应用

### 启动前的检查清单

先确认以下事项：

- [ ] Python 环境和依赖已安装。
- [ ] 前端依赖已安装。
- [ ] `.env` 中已经配置了可用的模型密钥或相关运行参数。
- [ ] 8000 和 5173 端口未被占用。

可用以下命令快速检查：

```bash
node --version
npm --version
lsof -i :8000
lsof -i :5173
```

### 方式一：使用脚本快速启动

如果你只是想最快进入页面，优先使用项目自带脚本：

```bash
./app/run.sh
```

这个方式适合：

- 初次使用。
- 不关心前后端分别输出什么日志。
- 只是想验证页面是否能正常进入。

### 方式二：手动启动，便于调试

如果你需要定位问题、查看日志或单独重启某一侧，使用两个终端分别启动。

#### 终端 1：启动后端

在仓库根目录执行：

```bash
.venv/bin/python -m uvicorn app.backend.main:app --reload --host 0.0.0.0 --port 8000
```

如果你的环境中可直接使用 Poetry，也可以执行：

```bash
poetry run uvicorn app.backend.main:app --reload --host 0.0.0.0 --port 8000
```

正常启动后通常会看到类似输出：

```text
INFO:     Uvicorn running on http://0.0.0.0:8000
INFO:     Application startup complete.
```

你可以用以下方式验证后端是否可用：

```bash
curl http://localhost:8000/api/health
```

也可以直接打开：

```text
http://localhost:8000/docs
```

#### 终端 2：启动前端

在仓库根目录执行：

```bash
cd app/frontend
npm install
npm run dev
```

正常启动后通常会看到：

```text
VITE v5.x ready in 300 ms
Local: http://localhost:5173/
```

### 为什么开发时需要两个服务

原因并不复杂：

- 前端负责页面渲染、交互和浏览器侧状态。
- 后端负责 API、模型调用、数据获取和回放结果读取。

开发环境里把这两者拆开，定位问题会更快：

- 页面样式或按钮异常，看前端终端。
- 数据返回错误或分析卡住，看后端终端。

### 首次访问

浏览器打开：

```text
http://localhost:5173
```

如果系统启用了登录页，先使用当前环境配置的账户登录；如果本地环境做了默认管理员初始化，请以你当前部署说明或环境变量中的配置为准，不建议把账号口令硬编码进公共文档。

## 4.3 页面结构与导航

### 主界面结构

页面通常可以分成三层：

```text
顶部栏：应用标题、用户信息、设置入口
侧边栏：Stock Analysis、History、Charts、Settings 等功能入口
主内容区：当前页面的参数表单、结果区或明细区
```

### 你最常用的几个入口

| 入口 | 作用 | 典型使用时机 |
| --- | --- | --- |
| Stock Analysis | 直接发起一轮分析 | 想快速看某只股票当前建议 |
| History | 回看过去已运行的结果 | 对比同一标的在不同日期的判断 |
| Replay Artifacts | 做选股复核和反馈回写 | 做研究复盘、看 near-miss 和 blocker |
| Charts | 通过图形方式查看表现 | 需要趋势和可视化辅助 |
| Settings | 管理模型、显示选项和偏好 | 想设置默认行为或页面展示方式 |

### 页面职责边界

这点很重要，避免用错页面：

- Stock Analysis 是“运行分析页”，主要服务于即时查看结果。
- Replay Artifacts 是“研究复核页”，主要服务于回放、选股质量判读和反馈沉淀。

如果你的问题是“今天为什么这只股票没买进去”，或者“这只 near-miss 到底是阈值问题还是执行阻断问题”，应该优先去 Replay Artifacts，而不是只看即时分析页。

## 4.4 核心操作流程

### 流程一：在 Stock Analysis 中发起一次分析

#### 第 1 步：输入股票代码

在股票输入框中填入代码，例如：

```text
AAPL
MSFT
NVDA
AAPL, MSFT, NVDA
```

建议：

- 单次先从 1 只股票开始，确认流程通了再批量。
- 使用标准 ticker，不要输入公司全名。
- 如果页面支持自动补全，优先从补全结果中选取。

#### 第 2 步：选择时间范围

常见取值：

- 7 天：更适合看最近消息和短期变化。
- 30 天：默认推荐，兼顾信息量和速度。
- 90 天及以上：适合看中期叙事和更长窗口。

为什么这一步重要：

- 时间太短，容易被短期噪声主导。
- 时间太长，旧信息可能稀释当前信号。

#### 第 3 步：选择模型

页面通常会提供多个模型选项。选择时建议遵循以下原则：

| 模型倾向 | 优点 | 适合场景 |
| --- | --- | --- |
| 高质量主模型 | 结果通常更稳定 | 重要样本、深度分析 |
| 轻量模型 | 速度快、成本低 | 快速试跑、批量探索 |

只有在环境已经配置对应密钥时，该模型才真正可用。

#### 第 4 步：选择智能体集合

常见做法：

- 初学者直接全选，先理解多智能体输出结构。
- 有明确风格偏好时，只选一组价值型或成长型智能体。
- 做对比实验时，固定股票和日期，只切换智能体集合。

不同智能体的意义不在于“谁对谁错”，而在于让你看到同一标的在不同投资哲学下的解释差异。

#### 第 5 步：点击 Run Analysis

按钮通常会经历以下状态：

| 状态 | 含义 |
| --- | --- |
| Ready | 参数完整，可以发起分析 |
| Running | 后端正在获取数据、调用模型、聚合结果 |
| Completed | 本轮分析已完成，可以查看输出 |

分析期间建议不要频繁刷新页面，否则容易丢失当前浏览状态。

#### 第 6 步：阅读结果

你通常会看到以下信息：

- 总体建议，例如买入、持有、卖出。
- 置信度，表示系统对当前建议的把握程度。
- 一致性，表示不同智能体之间是否高度一致。
- 各智能体观点摘要。
- 可能的风险提示或理由说明。

### 流程二：使用 History 回看历史结果

History 适合做三件事：

- 对比同一股票在不同日期的建议变化。
- 对比不同模型或不同智能体组合的输出差异。
- 找出“当时看起来合理，后来被市场验证错误”的样本。

建议操作顺序：

1. 先按股票代码过滤。
2. 再按日期范围或运行时间排序。
3. 打开两到三个关键样本做横向对比。
4. 记录哪些差异来自模型变化，哪些差异来自市场环境变化。

### 流程三：进入 Replay Artifacts 做研究复核

Replay Artifacts 不是“另一个结果页”，而是“把一次历史回放拆开给你检查”的研究界面。它更适合回答以下问题：

- 这一天有哪些股票进入候选池。
- 哪些股票被选中，哪些成为 near-miss。
- 某只股票未进入买单，究竟是选股质量不足，还是执行层 blocker 触发。
- 研究员应该写什么反馈，后续该进入哪类优化 backlog。

推荐阅读顺序：

1. [Replay Artifacts 选股复核 5 分钟速查](../manual/replay-artifacts-stock-selection-quickstart.md)
2. [Replay Artifacts 选股复核页面使用手册](../manual/replay-artifacts-stock-selection-manual.md)
3. [Replay Artifacts 周度复盘工作流手册](../manual/replay-artifacts-weekly-review-workflow.md)
4. [研究结论到优化 Backlog 的转换手册](../manual/research-conclusion-to-optimization-backlog-handbook.md)

### 流程四：使用 Settings 管理常用配置

Settings 一般用于保存“默认偏好”，而不是处理单次异常样本。

常见设置项包括：

- 默认模型。
- 默认显示密度。
- 自动展开详细结果。
- 历史记录每页数量。
- 常用智能体组合。
- 主题切换。

一个简单但实用的做法是保存两套预设：

- 日常速查：轻量模型 + 常用组合。
- 深度复核：主模型 + 全量组合。

## 4.5 如何解读页面结果

### 三个最容易混淆的指标

| 指标 | 它回答的问题 | 不该误解成什么 |
| --- | --- | --- |
| 总体建议 | 系统当前倾向做什么 | 不等于未来一定涨跌 |
| 置信度 | 系统对这次判断有多确定 | 不等于历史胜率 |
| 一致性 | 智能体之间是否看法接近 | 不等于结论一定更正确 |

### 一个典型的阅读顺序

推荐按下面顺序读，而不是上来只盯一个大结论：

1. 先看总体建议，确定系统方向。
2. 再看置信度，判断系统是否真的“有把握”。
3. 再看一致性，判断内部是否存在明显分歧。
4. 最后进入智能体观点，找出分歧来源。

### 什么时候要谨慎对待结果

以下几类情况都不适合直接把页面输出当成明确行动信号：

- 置信度不高，同时一致性也低。
- 只有少数智能体给出极强判断，其他多数中性。
- 历史结果与当前结果差异极大，但页面没有足够解释。
- 页面显示存在 blocker、阈值压线或执行层约束。

这类样本更适合进入 Replay Artifacts 做复核，而不是立刻转化为交易动作。

### AAPL 示例：如何做一次基础阅读

假设页面显示：

```text
Ticker: AAPL
Recommendation: BUY
Confidence: 0.82
Consensus: 0.75
```

可以这样解读：

- 方向上偏买入。
- 系统对该结论的确定性较高。
- 多数智能体意见大体一致，但不代表没有分歧细节。

下一步应该做的不是立刻停在“BUY”，而是继续看：

- 支持买入的主要理由是什么。
- 反对或保留意见来自哪些智能体。
- 这些反对意见是风险提示，还是根本性反驳。

## 4.6 常见问题与解决

### 问题一：页面打不开

先按以下顺序排查：

1. 前端是否还在运行。
2. 后端是否还在运行。
3. 浏览器打开的是否是正确地址。
4. 本地端口是否被其他进程占用。

常用检查命令：

```bash
lsof -i :5173
lsof -i :8000
curl http://localhost:8000/api/health
```

如果前端正常、后端异常，页面通常会加载出来，但请求数据时失败；如果前端本身没启动，浏览器通常直接打不开页面。

### 问题二：点击分析后长时间无响应

优先看后端终端输出，因为多数卡顿都发生在后端：

- 数据源请求慢。
- LLM 密钥无效。
- LLM 限流或超时。
- 某个分析步骤抛错但前端没有完整展示。

建议先确认健康检查可用：

```bash
curl http://localhost:8000/api/health
```

如果健康检查正常，再查看后端日志中的异常关键字，例如认证失败、超时、限流或数据缺失。

### 问题三：分析结束但结果区域为空

这类问题通常从浏览器开发者工具入手：

1. 打开浏览器开发者工具。
2. 查看 Network 面板中的分析请求。
3. 检查返回状态码和响应体。

一个正常的响应通常至少会包含总体建议、置信度和明细字段，例如：

```json
{
  "ticker": "AAPL",
  "recommendation": "BUY",
  "confidence": 0.72,
  "agents": [
    {
      "name": "Warren Buffett",
      "recommendation": "BUY",
      "confidence": 0.85,
      "reasoning": "..."
    }
  ]
}
```

如果接口返回 200，但页面没渲染，多半是前端状态或字段兼容问题；如果接口本身报错，就先修后端。

### 问题四：端口被占用

查看占用进程：

```bash
lsof -i :8000
lsof -i :5173
```

结束对应进程：

```bash
kill <PID>
```

必要时强制结束：

```bash
kill -9 <PID>
```

### 问题五：Replay Artifacts 页面里看不懂 blocker

这不是页面故障，而是口径问题。要先区分：

- 股票没入选。
- 股票已入选，但未进入买单。
- 股票进入买单，但后续执行受限。

这三类问题分别对应不同层面的结论，不应混写。具体写法请直接参考：

- [Replay Artifacts 研究反馈标签规范手册](../manual/replay-artifacts-feedback-labeling-handbook.md)
- [研究结论到优化 Backlog 的转换手册](../manual/research-conclusion-to-optimization-backlog-handbook.md)

## 4.7 练习题

### 练习 4.1：完成一次基础分析 ⭐

任务：

1. 启动前后端服务。
2. 打开 Web 页面。
3. 输入 `AAPL`。
4. 使用默认时间范围和默认模型发起一次分析。
5. 记录总体建议、置信度和一致性。

完成标准：

- [ ] 能独立把页面跑起来。
- [ ] 能完成一次分析。
- [ ] 能解释三个核心指标分别表示什么。

### 练习 4.2：使用 History 做横向对比 ⭐⭐

任务：

1. 分别对 `AAPL`、`MSFT`、`NVDA` 跑三次分析。
2. 进入 History。
3. 按股票和时间排序查看差异。
4. 记录哪一只股票的结果最稳定，哪一只波动最大。

完成标准：

- [ ] 能找到历史记录。
- [ ] 能筛选并打开历史样本。
- [ ] 能说清楚对比维度。

### 练习 4.3：进入 Replay Artifacts 做一次研究复核 ⭐⭐

任务：

1. 打开 Replay Artifacts 页面。
2. 选择一个已有 replay report。
3. 切换一个 trade date。
4. 找到一只 selected 股票和一只 near-miss 股票。
5. 说明两者差异，并判断 near-miss 更像阈值问题还是质量问题。

完成标准：

- [ ] 能进入正确页面。
- [ ] 能完成样本切换。
- [ ] 能说出至少一个 blocker 或筛选差异。

### 练习 4.4：把研究结论映射成 backlog 动作 ⭐⭐⭐

任务：

1. 从 Replay Artifacts 中找一个你认为“研究结论已经清楚”的样本。
2. 判断它更属于 Layer B、Layer C、Execution、Threshold 还是 Explainability。
3. 写出一条标准化 backlog 动作描述。

建议参考：

- [Replay Artifacts 周度复盘工作流手册](../manual/replay-artifacts-weekly-review-workflow.md)
- [研究结论到优化 Backlog 的转换手册](../manual/research-conclusion-to-optimization-backlog-handbook.md)

完成标准：

- [ ] 能把“页面现象”翻译成“系统动作类型”。
- [ ] 能避免把执行问题误写成选股质量问题。
- [ ] 能输出一条别人可以直接接手的 backlog 项。
