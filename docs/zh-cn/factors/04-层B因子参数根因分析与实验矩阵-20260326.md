## 3. 先说结论

如果只保留最重要的判断，结论如下：

1. 当前系统已经不是工程链路损坏问题，Replay Artifacts 报告可作为可信研究基线。
2. 本窗口最终只有 600988 和 300724 成交，不是因为 Layer C 或执行层先大规模阻断，而是因为 Layer B 本身只放出了极少数名字。
3. Layer B 的主矛盾不是单独一个 `score_b >= 0.38` 阈值太高，而是上游供给裁剪和融合语义共同偏冷。
4. 最关键的两个根因是：
   1. 只靠轻量技术 provisional score 决定谁有资格拿到 fundamental 和 event_sentiment 重评分。
   2. 中性 `mean_reversion` 默认仍参与归一化，持续稀释 trend 与 fundamental 的正向候选。
5. `profitability` 的 hard-cliff 语义是重要放大器，但不是唯一主因。
6. 当前优先级不应是直接下调 fast gate，而应先验证 Layer B 语义层和供给层的最小改动。

## 4. 干净报告到底说明了什么

### 4.1 会话层事实

从 [session_summary.json](../../../data/reports/paper_trading_window_20260323_20260326_live_m2_7_20260326/session_summary.json) 可确认：

1. 窗口覆盖 4 个交易日。
2. 实际发生 2 个执行交易日、2 笔执行订单。
3. 最终持仓只有 `600988 = 100` 与 `300724 = 100`。
4. 本次运行的 selection_artifacts、daily_events、pipeline_timings、session_summary 已相互对齐。

### 4.2 日级漏斗事实

四个交易日的 Layer B 结果分别为：

1. 2026-03-23：`layer_b_count = 1`
2. 2026-03-24：`layer_b_count = 1`
3. 2026-03-25：`layer_b_count = 2`
4. 2026-03-26：`layer_b_count = 2`

换句话说：

1. 200 个候选不是被容量上限截断。
2. 而是根本没有足够多的票能穿过 fast gate。

### 4.3 关键结构判断

本窗口的链路结构是：

1. 200 个候选进入 Layer B。
2. Layer B 先把大多数样本打回到 `below_fast_score_threshold`。
3. Layer C 只是在极少数穿透样本之间进一步做 `watchlist / avoid` 决策。
4. 执行层只对少数已入选样本施加 `position_blocked_score` 与 `position_blocked_single_name` 这类约束。

因此：

1. `200 -> 1` 或 `200 -> 2` 的主因是 Layer B。
2. `2 -> 1` 的主因才是 Layer C 与 final score。

## 5. 两个入选样本说明了什么

### 5.1 300724：稳定边缘幸存者，而非高共识强 alpha

300724 在本窗口里三次进入 selected：

1. 2026-03-23：`score_b = 0.3827`，刚过 fast gate，但被 `position_blocked_score` 挡住。
2. 2026-03-25：`score_b = 0.5305`，本窗口最强一次，最终生成 buy order。
3. 2026-03-26：`score_b = 0.4268`，继续进入 watchlist，但被 `position_blocked_single_name` 挡住。

共同特征：

1. Layer B 因子长期由 `fundamental + trend + 少量 event_sentiment` 驱动。
2. Layer C 始终只是弱正或近中性，并未形成强共识正向支持。
3. 它经常“勉强过线”，但不经常“高置信穿透”。

因此更合理的定性是：

1. 300724 是稳定的阈值边缘样本。
2. 它适合拿来研究 Layer B 与执行层的交互边界。
3. 它不应被误读为高把握度主升浪样本。

### 5.2 600988：一次性穿透更顺畅的样本

600988 的链路更干净：

1. 2026-03-24：成为唯一 watchlist 和唯一 buy order。
2. 2026-03-25：实际成交 100 股。
3. 之后很快掉回 Layer B 阈值下方，不具备持续重复入选特征。

共同特征：

1. Layer B 也主要由 `fundamental + trend + event_sentiment` 驱动。
2. Layer C 的正向支持略强于 300724。
3. 执行层没有在生成 buy order 时额外阻断。

因此 600988 说明：

1. 当前系统并不是完全不能放出买入样本。
2. 它能穿透，但穿透质量更多是“一次性有效”，不是稳定持续释放。

## 6. Layer B 参数链路的真实结构

当前 Layer B 不是单一阈值，而是四段压缩：

1. 技术轻量评分阶段：
   1. 最多只给 160 个候选做技术信号。见 [src/screening/strategy_scorer.py](../../../src/screening/strategy_scorer.py#L48)
2. heavy score 资格阶段：
   1. 只有 provisional score 大于等于 0.05 的候选才会进入基本面评分。见 [src/screening/strategy_scorer.py](../../../src/screening/strategy_scorer.py#L51) 和 [src/screening/strategy_scorer.py](../../../src/screening/strategy_scorer.py#L716)
3. 事件评分阶段：
   1. 只有前 40 个 heavy 候选能拿到 event_sentiment。见 [src/screening/strategy_scorer.py](../../../src/screening/strategy_scorer.py#L50) 和 [src/screening/strategy_scorer.py](../../../src/screening/strategy_scorer.py#L727)
4. fast gate：
   1. 最终还要满足 `score_b >= 0.38` 才能进入 high_pool。见 [src/execution/daily_pipeline.py](../../../src/execution/daily_pipeline.py#L54)

因此 Layer B 的真实问题，不能被简化成“只要调 0.38 就好”。

## 7. 根因拆解

### 7.1 根因一：供给裁剪和最终评分目标不一致

这是当前第一优先级问题。

最终融合权重希望：

1. `trend = 0.30`
2. `mean_reversion = 0.20`
3. `fundamental = 0.30`
4. `event_sentiment = 0.20`

也就是说，Layer B 在设计上把 heavy legs 看得很重。

但上游供给现实是：

1. 40 个候选直接拿不到技术信号。
2. 约一半甚至更多候选拿不到 fundamental。
3. 大多数候选拿不到 event_sentiment。
4. 谁有资格拿 heavy signal，先由轻量技术 provisional score 决定。

这会导致：

1. Layer B 表面上是四策略融合。
2. 实际上很多票只是在做轻量技术筛选后的残缺融合。
3. 技术上不够亮、但基本面本来可能不错的票，会在获得完整评分前就失去机会。

### 7.2 根因二：中性 mean_reversion 默认参与归一化

这是第二优先级问题。

当前语义是：

1. 只要 `completeness > 0`，就被视为 active signal。见 [src/screening/signal_fusion.py](../../../src/screening/signal_fusion.py#L154)
2. `mean_reversion.direction == 0` 默认并不会退出 active normalization。见 [src/screening/signal_fusion.py](../../../src/screening/signal_fusion.py#L149)
3. 只有开启特定 guarded mode 时，才可能被排除。见 [src/screening/signal_fusion.py](../../../src/screening/signal_fusion.py#L108)

这意味着：

1. trend 与 fundamental 明明都为正。
2. mean_reversion 虽然不给正分，但仍然拿走一部分归一化权重。
3. 结果是正向候选系统性被稀释。

从历史验证记录看，完全排除 neutral mean_reversion 会过于激进，但方向本身是正确的，需要 guarded 版本而不是维持现状。

### 7.3 根因三：profitability hard-cliff 语义会放大左尾压制

当前 quality-first guard 与 profitability 语义会把一部分样本直接打到极负：

1. paired quality breakdown 或 hard cliff 命中后，直接 forced avoid。见 [src/screening/signal_fusion.py](../../../src/screening/signal_fusion.py#L76)
2. forced avoid 会把 `score_b` 直接打成 `-1.0`。见 [src/screening/signal_fusion.py](../../../src/screening/signal_fusion.py#L276)

这不是 Layer B 稀缺的唯一原因，但它会显著拉低：

1. 均值
2. 中位数
3. 分布上界附近的可用样本密度

所以它是重要放大器，不是唯一主因。

### 7.4 根因四：决策分类阈值与 fast gate 阈值不一致

当前存在一个小但真实的错位：

1. `score >= 0.35` 就会被分类成 `watch`。见 [src/screening/models.py](../../../src/screening/models.py#L88)
2. 但只有 `score_b >= 0.38` 才能进入 high_pool。见 [src/execution/daily_pipeline.py](../../../src/execution/daily_pipeline.py#L54)

这不是造成 200 变 1 的主因，但会带来：

1. 语义解释噪声
2. watch / high_pool 边界难以直觉理解

## 8. 当前优先级排序

当前建议按如下顺序看问题：

### P1：供给裁剪语义错配

重点变量：

1. `TECHNICAL_SCORE_MAX_CANDIDATES = 160`
2. `HEAVY_SCORE_MIN_PROVISIONAL_SCORE = 0.05`
3. `FUNDAMENTAL_SCORE_MAX_CANDIDATES = 100`
4. `EVENT_SENTIMENT_MAX_CANDIDATES = 40`

### P2：neutral mean_reversion 稀释正向候选

重点变量：

1. `LAYER_B_ANALYSIS_NEUTRAL_MEAN_REVERSION_MODE`
2. `_normalize_for_available_signals()` 的 active 资格语义

### P3：profitability / quality-first hard-cliff 放大左尾

重点变量：

1. `LAYER_B_ANALYSIS_PROFITABILITY_ZERO_PASS_MODE`
2. `_has_quality_first_red_flag()` 的触发覆盖面

### P4：0.35 与 0.38 的语义错位

重点变量：

1. `classify_decision()`
2. `FAST_AGENT_SCORE_THRESHOLD`

## 9. 实验矩阵

本节是当前建议的统一实验矩阵，后续研究优先按此顺序推进。

### 9.1 实验目标

实验目标不是单纯“让 Layer B 数量变多”，而是同时验证四件事：

1. `avg_layer_b_count` 是否从极低水平得到改善。
2. 新增样本是否主要死在 Layer C，还是能真正改善 watchlist / buy_order。
3. 是否出现明显结构性漏放。
4. 运行成本是否上升过快。

### 9.2 验收指标

每个实验建议统一记录以下指标：

1. `avg_layer_b_count`
2. `nonzero_layer_b_days`
3. `avg_watchlist_count`
4. `avg_buy_order_count`
5. `新增样本在 watchlist 被拒绝的原因分布`
6. `avg_total_day_seconds`

### 9.3 参数实验矩阵

| 实验 ID | 改动类别 | 具体改动 | 预期收益 | 主要风险 | 当前建议 |
| --- | --- | --- | --- | --- | --- |
| B1 | profitability 语义 | `zero_pass_mode = inactive` | 缓和 fundamental hard cliff，释放少量边缘样本 | 释放弱盈利票 | 首先验证 |
| B2 | neutral MR 语义 | guarded neutral mean_reversion exclusion | 减少中性均值回归对白票的稀释 | 通过数可能跃迁式上升 | 第二优先级 |
| B3 | event 供给侧 | `EVENT_SENTIMENT_MAX_CANDIDATES: 40 -> 80` | 减少 event 缺席导致的残缺融合 | 成本提升，新闻噪声扩大 | 在 B1/B2 后验证 |
| B4 | technical 供给侧 | `TECHNICAL_SCORE_MAX_CANDIDATES: 160 -> 200` | 让全部候选至少拿到 light signals | 额外算力开销 | 在 B3 前后对比 |
| B5 | heavy 准入 | 放宽 `HEAVY_SCORE_MIN_PROVISIONAL_SCORE: 0.05 -> 0.03` | 让更多非强技术票进入 fundamental | 可能引入大量弱票 | 仅做边界测试 |
| B6 | fundamental 供给侧 | `FUNDAMENTAL_SCORE_MAX_CANDIDATES: 100 -> 140` | 提高 heavy legs 覆盖率 | 成本明显上升 | 仅在 B4 后验证 |
| B7 | fast gate | `FAST_AGENT_SCORE_THRESHOLD: 0.38 -> 0.36` | 直接增加 Layer B 通过数 | 语义问题与阈值问题混淆 | 暂不作为第一步 |
| B8 | 语义一致性 | 对齐 `watch` 分类阈值与 fast gate | 提升解释一致性 | 不一定改善业务结果 | 可并行低风险处理 |

### 9.4 推荐顺序

当前建议按下面顺序推进，而不是同时松多个旋钮：

1. B1：profitability inactive
2. B2：guarded neutral mean_reversion exclusion
3. B3 或 B4：供给侧放宽，二选一先做
4. B5 / B6：只做边界实验
5. B7：最后再考虑直接降 fast gate

1. `baseline_total_layer_b_passes = 1`
2. `variant_total_layer_b_passes = 6`
3. `layer_b_pass_delta = +5`

新增样本包括：

1. `20260323 / 300724`
2. `20260324 / 600988`
3. `20260325 / 300394`
4. `20260326 / 300394`
5. `20260326 / 300724`

这些新增样本的共同标签是：

1. `neutral_mean_reversion_active`
2. `trend_fundamental_dual_leg`
3. 部分样本同时具备 `event_sentiment_missing`

更关键的是，分数提升幅度并不小：

1. `300724` 约 `+0.08`
2. `600988` 约 `+0.088`
3. `300394` 约 `+0.089`

结论：

1. 当前窗口并不是对 mean_reversion 语义完全不敏感。
2. 它只是不对 guarded B2 敏感，但对 full exclude 高度敏感。
3. 这进一步证明 neutral mean_reversion 的确是一个强杠杆位。
4. 同时也证明为什么 full exclude 不能直接默认上线，因为它在这 4 天窗口里已经把 Layer B 通过数从 `1` 直接推到 `6`。

#### combined 边界证据

1. `combined` 与 `neutral_mean_reversion_only` 的结果完全一致，`layer_b_pass_delta = +5`
2. 说明在这个窗口里，profitability inactive 并没有提供额外增量，主导杠杆仍然是 neutral mean_reversion 的 active 资格语义。

因此，当前窗口给出的更细结论是：

1. B1 不是这 4 天的主矛盾。
2. guarded B2 在这 4 天过于保守。
3. full exclude B2 又明显过强。
4. 下一步最值得研究的，不是继续争论要不要直接降 fast gate，而是要不要设计一个介于 guarded 与 full exclude 之间的中间版 MR active 语义。

### 11.3 当前窗口的 MR 中间档位扫描

为了继续执行任务 1，本轮又对当前窗口做了一次完整的 MR 档位扫描：

结果非常干净：

也就是说，当前窗口里：

1. `0.33` 和 `0.32` 两个门槛都没有把任何样本从 baseline 释放出来。
2. `no_hard_cliff` 与否，在这 4 天窗口上也没有表现出差异。
3. 当前仓库里已有的 guarded 语义，在本窗口不存在“温和中间解”。

因此，这轮任务 1 的进一步结论是：

   1. 只在 `trend + fundamental` 更强时给部分排除。
   2. 对 `event_sentiment > 0` 和 `event_sentiment = 0` 分别处理。
   3. 让 neutral mean_reversion 只保留部分权重，而不是二元的“全保留 / 全排除”。

### 11.4 新中间版 MR 语义：partial-weight 第一轮验证

2026-03-27 在当前 workspace 中，已把一个新的“中间版 MR 语义”落成可运行变体：

语义定义不是“直接排除 neutral mean_reversion”，而是：

4. baseline 分数至少达到 `0.33`。
5. 命中后，不把 mean_reversion 完全踢出 active 集合，只把它的 active raw weight 降到原来的一半。

对应当前窗口的验证产物：

这说明新语义确实完成了“中间版”目标：

1. 它不再像旧 guarded 那样几乎没有释放。
2. 也没有像 full exclude 那样直接冲到最激进边界。
3. 但从业务强度上看，它依然偏强，不应直接视为默认候选。

### 11.5 当前窗口里，究竟是哪个因子把大部分股票过滤掉了

2026-03-27 还对当前窗口 `20260323..20260326` 做了基于 baseline 的聚合诊断。

窗口级统计结论如下：

1. 被 Layer B 挡下的样本共 `799` 个，真正过线的只有 `1` 个。
2. 在所有被挡下样本中：
   1. `fundamental.direction <= 0` 的有 `692` 个。
   2. `trend.direction <= 0` 的有 `339` 个。
   3. `event_sentiment` 缺失的有 `642` 个。
   4. `event_sentiment < 0` 的只有 `20` 个。
   5. `neutral mean_reversion` 仍 active 的有 `345` 个。
3. 在真正接近阈值但仍被挡下的样本里，也就是 `score_b >= 0.30` 的 `9` 个 near-threshold 样本中，`9/9` 全部都有 `neutral mean_reversion active`。
4. 如果只看“最大负贡献因子”统计，则出现频次最高的是：
   1. `fundamental = 180`
   2. `trend = 179`
   3. `mean_reversion = 33`
   4. `event_sentiment = 4`

所以这里必须把两个层次分开：

1. 如果问“是谁把大多数股票广义上压在 Layer B 下方”，答案不是 MR，而是 `fundamental` 第一、`trend` 第二。
2. 如果问“是谁把接近阈值的边缘样本继续压在 `0.38` 之下”，答案是 `neutral mean_reversion active`。

更简化地说：

1. 大盘子里，大多数票根本不是被 MR 一脚踢下去的，而是 trend 和 fundamental 自身就不够正。
2. 但在少数已经接近过线的边缘票上，MR active 稀释就是关键压制项。
3. 因此，MR 语义修正的业务价值更像“修边缘误伤”，不是“解决全部 scarcity 的单一总开关”。

如果需要单独理解 `fundamental` 这条策略本体，包括五个子因子、阈值、聚合语义、与 Layer C 的衔接关系，可继续阅读：

1. [docs/zh-cn/factors/07-fundamental-factor-professional-guide.md](./07-fundamental-factor-professional-guide.md)

## 12. 最终结论

当前最重要的判断不是“0.38 要不要降”，而是：

1. Layer B 上游供给过早、过重地依赖轻量技术 provisional score。
2. heavy legs 并没有对全部候选公平开放。
3. 中性 mean_reversion 仍在稀释本就不多的正向候选。
4. profitability hard-cliff 进一步把左尾压扁。

所以：

1. 现在不应该先做粗暴放水。
2. 应该先做语义修正和供给修正。
3. 只有在这些修正仍然不能改善实际 watchlist / buy_order 时，才值得再讨论直接下调 fast gate。
