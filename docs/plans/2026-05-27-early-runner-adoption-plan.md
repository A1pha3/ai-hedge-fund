# Early Runner 接入执行计划

> 面向当前仓库的 BTST 文档流与选股流，先把方案 A 做成稳定、轻量、可复盘的日常系统，再决定是否升级为正式候选增强层。

## 目标

这份计划不追求一句“接进来了”，而是追求三件真正有价值的事：

1. 让 BTST 文档每天都能轻松生成，尽量不靠手工补。
2. 让选股顺序更清楚，把正式主线、交集优先、补充复审三层拆开。
3. 在不扩大执行噪音的前提下，逐步提高命中质量、胜率和盈亏比。

这里要明确一个边界：任何流程优化都只能提高“更容易筛掉差票、优先看到强票”的概率，不能承诺市场结果一定大涨。真正可交付的是更稳定的流程、更少的误判、更好的样本积累和更清晰的执行优先级。

## 现状判断

当前仓库里，方案 A 已经不是空白设想，而是已经具备核心骨架：

- early-runner 主 artifact 已存在：`data/reports/btst_early_runner_v1_latest.json` / `.md`
- early-runner 日表脚本已存在：`scripts/generate_btst_early_runner_daily_tables.py`
- BTST 文档包已支持 early-runner 叠加：`scripts/generate_btst_doc_bundle.py`
- 历史回放验证脚本已存在：`scripts/validate_btst_early_runner_history.py`
- manifest 与 nightly control tower 已能承接 early-runner 状态刷新

真正的短板不在“能不能接”，而在下面四点：

1. 近期板是否稳定跟上当日信号日。
2. 文档包是否能替代手工整理，形成固定读法。
3. 交集票和 only early-runner 票是否被清楚分层，而不是混在一张单里。
4. 是否已经形成一套可量化的升级标准，而不是靠主观感觉推进。

## 方案 A 的最终定位

方案 A 在当前阶段的定位应该固定为：

- 正式 BTST 是执行主线。
- early-runner 是第二观察层。
- 交集票是优先复审层。
- only early-runner 票是补充复审层。
- second-entry / reentry 是机会回补层。

这样定义之后，BTST 文档流和选股流会自然形成四层结构：

| 层级 | 来源 | 是否可直接执行 | 主要作用 |
| --- | --- | --- | --- |
| 正式执行层 | `selected_actions` / `primary_entry` | 是 | 决定主票与主执行顺序 |
| 交集优先层 | 正式 BTST 与 early-runner 同时命中 | 否，先复审再执行 | 提高优先级，优先看共识票 |
| 补充复审层 | only early-runner priority / watchlist | 否 | 提前发现潜在线索，但不抢主决策权 |
| 回补机会层 | `second_entry_reentry` | 否 | 承接二次确认、回踩再起、再入场线索 |

这四层一旦稳定，BTST 文档就不再只是“信息堆叠”，而会变成一套真正可操作的筛选框架。

## 为什么先做方案 A

### 1. 现在最缺的是近期证据，不是更多入口

既然 `generate_btst_doc_bundle.py` 已经能把 early-runner 写进 `BTST-LLM`、`EXEC-CHECKLIST` 和 `EARLY-WARNING` 文档，说明入口问题已经基本解决。现在真正影响结果的是：这些新增内容是否能稳定提升判断，而不是只增加阅读负担。

### 2. 方案 A 是最容易拉升“使用体验”和“判断质量”的阶段

只要把方案 A 跑顺，立刻就会出现三个变化：

- 日常阅读顺序固定，不需要临时决定先看哪份文档。
- 交集票被提前高亮，强共识票更容易优先进入盘中视野。
- only early-runner 票被明确降级到补充复审，噪音更容易被关在主执行层之外。

### 3. 后续所有升级都建立在这一层证据上

中期的“交集优先”是否值得强化，长期的 runtime promotion 是否值得开启，都必须先回答两个问题：

1. 交集票是否真的比普通票更有价值。
2. only early-runner 票是否真的能提前暴露后续强票。

如果这两件事还没被连续样本证明，就没有理由让 early-runner 提前接管正式链路。

## 目标 operating model

方案 A 不该只是“文档里多一节”，而应升级成下面这套日常 operating model。

### 一条最短日常链路

每天只做 5 步：

1. 刷新 early-runner 主 artifact。
2. 生成 early-runner 日表。
3. 刷新 manifest 与 control tower。
4. 生成 BTST 文档包。
5. 记录结果并做收盘复盘。

这条链路的核心目标只有两个：

- 让文档生成变成固定动作，而不是临时拼接。
- 让选股优先级自动进入文档，不再靠人工脑补。

### 一套固定读法

文档包默认按下面顺序阅读：

1. `BTST-LLM-YYYYMMDD.md`
2. `BTST-YYYYMMDD-EXEC-CHECKLIST.md`
3. `BTST-YYYYMMDD-EARLY-WARNING.md`
4. `BTST-YYYYMMDD.md`
5. `YYYYMMDD-两套交易计划通俗说明.md`

如果当天时间紧，只读前 3 份即可：

- `BTST-LLM` 看正式执行层和交集高亮
- `EXEC-CHECKLIST` 看次日开盘顺序
- `EARLY-WARNING` 看 only early-runner 与 second-entry 补充池

### 一组固定判断规则

次日盘前统一按下面规则决策：

| 状态 | 使用方式 | 对选股流的影响 |
| --- | --- | --- |
| `exact` | 正常使用 early-runner | 允许交集票进入优先复审层 |
| `stale_fallback` | 只当参考线索 | 交集票不升级，只保留历史参考属性 |
| `unavailable` | 完全按正式 BTST 走 | 不补任何 early-runner 观察票 |

这张表的意义很大。它能直接防止“旧板高亮被当成当日强信号”的错误升级。

## 选股流怎么优化到更强

真正能影响准确率、胜率和盈亏比的，不是“再多看几只票”，而是把票分对层、排对顺序、过滤掉不该抢优先级的噪音。方案 A 的选股优化要围绕下面四条展开。

### 1. 正式执行层只保留主胜率责任

正式 BTST 继续负责：

- 主票排序
- 执行模式选择
- 盘前主观察顺序
- 收益与回撤的第一责任

early-runner 不能抢这个职责。否则你会把“更早看到”误以为“更值得做”，这是最容易恶化盈亏比的地方。

### 2. 交集票只做优先级提升，不做无条件升级

交集票最合理的定位是：

- 优先看
- 优先复审
- 优先等确认
- 不优先盲打

也就是说，交集票可以提升你的注意力分配效率，但不能直接取代确认流程。这样做的价值是提高命中质量，而不是扩大冲动出手次数。

### 3. only early-runner 票必须关在补充池

only early-runner 票的价值在于“前瞻”，它的风险也恰恰来自“前瞻”。因此必须强制保持三条规则：

- 不自动进入正式执行清单
- 不因为分数高就抢主票位置
- 只有在盘中出现二次确认后，才允许升级到复审重点

这一条不是保守，而是保护盈亏比。很多流程劣化，都是从“补充观察票慢慢漂成主执行票”开始的。

### 4. second-entry / reentry 独立管理

`second_entry_reentry` 不能和普通 watchlist 混用。它本质上是一类不同的机会：

- 它不是最早发现。
- 它不是主票排序。
- 它是错过首波后，等待结构更清楚的再入场机会。

把它单独挂出来，能有效减少你在“早期噪音”和“二次确认机会”之间的混淆。

## 文档流怎么优化到更顺手

### 1. 统一采用“5 份核心文档 + 2 份补充预警文档”

仓库现在已经能默认产出 7 份文件，其中最稳的结构应该定义为：

- 核心文档：`BTST-LLM`、`BTST`、`通俗说明`、`论坛短版`、`EXEC-CHECKLIST`
- 补充文档：`EARLY-WARNING`、`EARLY-WARNING-CARD`

忙碌日只保留 5 份核心文档即可，对应命令是：

```bash
uv run python scripts/generate_btst_doc_bundle.py \
  --signal-date YYYYMMDD \
  --output-dir outputs/YYYYMM/YYYYMMDD_scheme_a \
  --no-refresh-early-runner \
  --core-only
```

这样做的好处是：你不用在“全量产物”和“能否按时生成”之间二选一。

### 2. 文档里的层级名称固定，不再漂

所有 BTST 相关文档统一使用下面这些名称：

- 正式执行层
- 正式观察层
- 交集优先复审
- Only Early Runner
- Second Entry / Reentry

层级名称一旦固定，后续读文档和做复盘就会顺很多，也更适合做长期对比。

### 3. 每份文档都回答一个唯一问题

文档不要互相抢职责：

- `BTST-LLM` 负责完整交易叙述和分层判断
- `EXEC-CHECKLIST` 负责次日执行顺序
- `EARLY-WARNING` 负责补充观察和预警池
- `BTST` 负责规则底稿和正式主线解释
- `通俗说明` 负责快速理解

这样读者不会在 7 份文件里反复找同一件事。

## 阶段划分

### 阶段 1：方案 A 固化

目标是两件事：

1. 文档包连续稳定生成。
2. 交集票与补充票的分层判断稳定进入日常流程。

建议周期：连续 `10` 到 `15` 个交易日。

阶段 1 通过标准：

1. 每天都能产出核心文档，不靠手工补结构。
2. `Early Runner` 章节每天都稳定出现。
3. `exact`、`stale_fallback`、`unavailable` 三种状态都能被清楚区分。
4. 每天都能记录交集票、only early-runner 票和收盘结论。

### 阶段 2：交集优先强化

前提是阶段 1 跑稳之后，再把 early-runner 从“展示层”推进到“优先级增强层”。

本阶段只做三件事：

1. 交集票优先复审。
2. only early-runner 票只留在补充池。
3. 统计交集票是否真的更值得优先看。

建议周期：连续 `10` 到 `20` 个交易日。

阶段 2 通过标准：

1. 交集票的盘中确认率高于普通观察票。
2. 交集票的收盘表现不差于正式主线平均水平。
3. only early-runner 票里能观察到一定比例的后续接力确认。
4. 文档层没有明显增加混乱度和重复劳动。

### 阶段 3：正式 promotion 评估

只有在阶段 1 与阶段 2 合计跑满 `20` 到 `40` 个交易日后，才值得讨论。

重点仍然不是“接不接”，而是“接进去之后会不会伤害正式链路”。

必须重新核验：

- `deployment_mode`
- `ready_for_shadow_rollout`
- `month_oos_pass_count`
- `deduped_closed`
- `abandoned_gap_rate`
- `promotion_blockers`

如果这些 acceptance 指标还没明显改善，就不要推进正式 runtime promotion。

## 每日、每周、每阶段的验证口径

为了让“更顺手”和“更准”这两件事可验证，统一使用下面这组口径。

### 每日看 6 项

1. `early_runner_status`
2. `early_runner_latest_trade_date`
3. 是否出现交集票高亮
4. 是否出现 only early-runner 补充票
5. 核心文档是否全部生成
6. 当天是否需要手工修补

### 每周看 5 项

1. `exact` 占比
2. 交集票出现天数
3. only early-runner 补充票出现天数
4. 手工修补天数
5. 交集票与普通观察票的相对复盘质量

### 每阶段看 5 项

1. 是否稳定减少“文档能跑但不好用”的情况
2. 是否稳定减少“补充票混入主执行层”的情况
3. 是否稳定提高交集票的观察价值
4. 是否稳定控制噪音票数量
5. 是否为后续 promotion 提供足够证据

## 推荐的一周执行顺序

### Day 1

跑通 daily artifact、daily tables、manifest、control tower 和文档包。

### Day 2

确认文档读取顺序、目录规范和记录模板已经固定，不再每天临时调整。

### Day 3 到 Day 5

连续跑日常链路，开始积累交集票、补充票和收盘结论。

### 第 2 周末

做一次方案 A 评审，重点回答下面 4 个问题：

1. 方案 A 是否已经真正替代手工整理。
2. 交集票是否开始表现出更强价值。
3. only early-runner 票是否大多仍是噪音。
4. 是否值得进入中期强化阶段。

## 不要现在做的事

### 1. 不要让 early-runner 抢正式主票决策权

现在它最适合做观察增强，不适合做主选股替代。

### 2. 不要跳过日表，直接手工读大 JSON

短期省一步，长期会让文档流重新退化成人工流程。

### 3. 不要在近期样本不足时谈“正式接管”

没有连续样本，所有升级都会变成主观判断。

## 最终建议

最合理的顺序仍然是：

1. 先把方案 A 固化成稳定、轻量、固定读法的日常系统。
2. 再用 `10` 到 `15` 个交易日样本，判断交集优先是否真的提升判断质量。
3. 最后才看 acceptance 是否允许进入正式 promotion 评估。

对当前仓库来说，最值钱的优化不是“再多造一个入口”，而是把 BTST 文档流和选股流收敛成一套每天都能轻松使用、每天都能复盘归因、并且能持续筛掉低质量机会的工作系统。
