## v2.2.8 (2026-06-16) — AutoDev campaign 18: R41 fundamental ann_date PIT 过滤（lookahead hardening 续集完结）

### Feature Delivery — R41: fundamental 财报 ann_date point-in-time 过滤 ✅
`_should_include_financial_period`（`tushare_financial_metrics_helpers.py`）此前只按 report period
类型（annual/quarterly/ttm）过滤，**从不检查公告日（ann_date）是否 ≤ trade date**。回测/replay
场景下，2 月的模拟交易会读到 4 月才公告的 2023 年报——**系统性前瞻偏差**，虚高所有含 fundamental
agent（Warren Buffett / Michael Burry / Cathie Wood / Ben Graham 等）的回测收益。这是 R37-R40
lookahead hardening 家族的同类残留：已修 macro（R40）、prices（R37 qfq）、calendar（R38），fundamental
路径漏修。Campaign 16 的 Bug Hunt 发现并记为 deferred 大切片（beta veto 风险：触及 backtest lookahead 核心）。

**修复**（红绿 TDD，8 个 fixture 测试）：
- `_should_include_financial_period` 新增 `ann_date_str` + `as_of_date` 可选参数。当二者均存在、格式合法、
  且 `ann_date > as_of` 时返回 `False`（排除前瞻行）。
- `build_financial_metrics_from_frames` 的主循环传入 `ann_date=str(row.get("ann_date"))` 与 `as_of=end_date`
  （trade date），让 PIT 过滤在回测/replay 自动生效。
- **live 模式不变**：`as_of=None` 或缺省/格式异常时回退到历史行为，避免过度过滤误丢合法数据
  （对齐 C2-BH2 鲁棒性契约——一行坏数据不能毒化整批）。
- trade date 与 ann_date 均做 dashed→compact 归一化（兼容 agent 端 `YYYY-MM-DD` 与 backtest 端 `YYYYMMDD`）。

验证：`pytest tests/tools/test_tushare_financial_metrics_helpers.py -q` → 17 passed（8 新 R41 + 9 既有）；
`pytest tests/tools/ -q` → 35 passed。测试覆盖 live-mode-unchanged / PIT-exclude / PIT-include-on-or-before /
dashed-normalize（双向）/ malformed-ann_date-fallback / malformed-as_of-fallback / 集成回测路径。

domain overlay：finance-quant `risk_floor=4`，fixture 证据 + disclosure 满足 beta veto 前置条件。
服务于产品目标"更高确信"——历史回测可信度。

## v2.2.7 (2026-06-16) — AutoDev campaign 17: checkpoint crash-safety + TTM 数据完整性

### Bug Hunt — C2-BH1: checkpoint 非原子写 → crash recovery wedge ✅
`write_checkpoint`（`engine_checkpoint_helpers.py:56`）此前用 `path.write_text` 直接写最终路径
（truncates-then-streams），crash 中途（SIGKILL/OOM/disk-full）留下 truncated/空 checkpoint 文件。
恢复链随后 wedge：`reset_output_artifacts_for_fresh_run`（campaign 15 加固的 reset guard）看到
`checkpoint_path.exists()=True`（truncated 文件存在）跳过 cleanup，而 `read_checkpoint` 的
`json.loads` 在 truncated 文件上抛 `JSONDecodeError` → 整个 resume crash，session 卡死，唯一出路
是手动 `rm` checkpoint 丢失全部进度。这是 campaign 15 保护的不变量上游写入层不 crash-safe。

修复（红绿 TDD，已验证复现）：
- `write_checkpoint` 改原子写：`NamedTemporaryFile` + `os.fsync` + `os.replace`（POSIX 原子，同 mount）。
  复用仓库既有模式（`watchlist.py`/`recommendation_tracker.py`）。os.replace 失败时清理 tmp 残留。
- `read_checkpoint` 对 corrupt/truncated 文件容错：quarantine 为 `.corrupt` + 返回 `{}`，不再抛
  `JSONDecodeError` wedge session。
- `_load_checkpoint`（engine.py:327）适配：空 payload（corrupt 被清理）时走 fresh run（返回
  `None, None`），不再 `KeyError` on `payload["portfolio_snapshot"]`。

复现测试：truncated/empty/corrupt/missing checkpoint 全部 graceful degrade；原子写不留 tmp 残留。

### Bug Hunt — C2-BH2: malformed end_date 毒化整个 ticker 财务数据（静默 `[]` 丢失）✅
`_build_prior_period_keys`（`tushare_financial_metrics_helpers.py:735`）此前 `int(end_date_str[:4])`
无条件执行——Tushare `fina_indicator` 返回 null/missing/短 end_date 的行（新股/部分年度重述）时
`int(""[:4])` 抛 `ValueError`，冒泡到 `get_ashare_financial_metrics_with_tushare` 的外层 except
被静默吞掉，返回 `[]`——**整个 ticker 的全部财务数据**因一行坏数据丢失，只有一条通用日志。

修复：`_build_prior_period_keys` 对 malformed end_date（len<8 或前 4 位非数字）返回 sentinel key
（下游 `.get()` 自然 miss），单坏行不再毒化整批。9 个 robustness 测试（valid/empty/short/non-digit/None）。

### Refactor Batch

## v2.2.6 (2026-06-16) — AutoDev campaign 16: NaN rank corruption 链路修复 + persist fail-loud

### Bug Hunt — BH-012: NaN/Inf score_b bubble 到推荐榜首（money-path ranking corruption）✅
`composite_score.py:225` 此前用 `float(rec.get("score_b", 0.0) or 0.0)` 读 score_b，但
`float(nan or 0.0)` 仍是 NaN（Python 中 NaN 为 truthy），且 `max(-1.0, min(1.0, nan))` 在
CPython 返回 **1.0**——所以一个 corrupt/NaN 的 score_b 会静默拿到最高 composite 分并
**bubble 到 front-door 推荐列表顶部**，挤掉合法候选。这是 `--top-picks` 排序链路上的真实
correctness bug（money-path）。

修复链路（红绿 TDD，已验证复现）：
- `composite_score.py:225` 改用 `coerce_score_b`（来自 `src.utils.numeric`，拒绝 NaN/Inf）。
- `signal_momentum.py:181/192` 同根因 drain——历史 score_b 含 NaN 会污染 `_simple_slope` →
  `momentum_bonus` → 回流 composite 的 `mom` 项，即使 base_score 已 coerce 仍会让 composite=1.0。
  改用 `coerce_score_b` 堵住 momentum_bonus NaN 链路，使 BH-012 修复完整。
- 2 个复现测试：`test_nan_score_b_does_not_bubble_to_top` + `test_inf_score_b_does_not_corrupt_ranking`。

### Bug Hunt — BETA-007-drain: hydrate_position_prices 绕过停牌 guard ✅
`hydrate_position_prices`（`engine_market_data.py:246`）此前重取价格时**无** BETA-007 的
`volume <= 0` 停牌 guard——当持仓 ticker 停牌（被 `load_current_prices` 因 volume=0 排除），
hydrate 会用停牌日 carry-forward close mark-to-market，**完全绕过** BETA-007 保护，导致
NAV/drawdown/Sharpe 在持仓停牌日失真。同一模块、同一数据源、guard 不一致 = same-class bug。

修复：hydrate 加同款停牌 guard，停牌持仓回退 cost_basis。复现测试
`test_suspended_held_position_falls_back_to_cost_basis`（停牌持仓不再以停牌价 25.0 mark，
回退 cost_basis 10.0）。提取 `_is_suspended_row` helper 消除 load/hydrate 重复（DRY）。

### Product Quality — persist 层 NaN/Inf fail-loud（BH-012 根因堵截）✅
`_save_json_report`（`main.py:322`）此前用 `json.dump(..., default=str)` 无 `allow_nan=False`——
corrupt NaN 会被写为字面 `NaN` token（非标准 JSON），Python `json.loads` 读回时接受，
使 NaN 静默穿越 round-trip 重新污染排序（BH-012 的持久化根因）。

新增 `_sanitize_nonfinite` 纯函数（递归把 NaN/Inf→null）+ `allow_nan=False` 硬守卫。磁盘报告
永远是合法 strict JSON；下游 reader 通过 `coerce_score_b` 把 null→0.0。12 个测试（纯函数 +
端到端 round-trip）。这是 R40 lookahead hardening 之后的产品信任增量——corrupt 数据写入即
被 sanitize，不再静默污染 front-door 排序。

### Refactor Batch
- DRY: 提取 `_is_suspended_row` module-level helper，`load_current_prices` +
  `hydrate_position_prices` 共用停牌 guard（消除漂移风险）。
- 大文件重构（stock_detail 253行、main.py 2995行）在 finance-quant domain 风险过高，defer。
- 其余 15+ `float(... or 0.0)` display/sort 站点为 display-only，留 backlog。

### Backlog refill — R41 (deferred)
Bug Hunt 发现 fundamental 财报数据无 `ann_date`（公告日）point-in-time 过滤——
`_should_include_financial_period` 只按 period 类型过滤，回测会用模拟交易日后才公告的财报
（如 2 月回测读到 4 月才公告的年报），系统性虚高 fundamental agents 回测收益。这是 R40
lookahead hardening 的同类残留。属大切片（触及 backtest lookahead 核心，需 Tushare fixture +
beta veto 评审），记为 R41 backlog，不在本 campaign 擅自实施。

## v2.2.1 (2026-06-16) — AutoDev campaign 10: ALPHA-006 + R40 + macro DRY

### ALPHA-006 (P2) ✅ — walk-forward drawdown_p10 小样本插值修正
`src/backtesting/early_runner_walk_forward.py` 的 `_distribution_p10` 此前用离散索引
`max(0, floor((N-1)*0.10))`，对 N<11 的样本索引恒为 0，p10 静默返回**最小值**（最差回撤）
而非真实 p10，系统性高估小窗口 walk-forward 报告的尾部风险。改为 `numpy.percentile` 线性插值，
与其它分析层一致，小样本也有可解释的尾部估计。该字段为 informational（不在 `_ranking_key`

## v2.2.3 (2026-06-16) — AutoDev campaign 13: GAMMA-007 embargo-bound prefetch

### GAMMA-007 (P2) ✅ — walk-forward prefetch embargo guard (latent look-ahead hardened)
`MarketDataLoader.prefetch_data` 此前无条件拉取到 `end_date`（1年窗口），依赖每个
walk-forward 调用方自行保证 `end_date` 是测试窗口的最后交易日而非整个回测末日，否则
会把窗口后的价格灌入特征缓存（latent point-in-time look-ahead）。

`min(end_date, data_through)`，对 per-ticker 价格/指标/新闻 **和** benchmark 序列
全部生效，让 no-look-ahead 契约自执行而非依赖调用方约定。该参数已贯穿
`BacktestEngine.__init__`，walk-forward 调用方可传 `data_through=test_end` 启用。
`data_through=None`（默认）保持原单窗口行为。

审计确认当前调用方（`run_backtest_param_grid._build_engine` 每窗口 `end_date=test_end`）
已天然夹住迭代，故 bug 为 latent——本次为防御性硬化，把潜在风险变为不可能。

## v2.2.5 (2026-06-16) — AutoDev campaign 15: paper_trading 崩溃恢复不变量加锁

### paper_trading 崩溃恢复不变量回归测试 ✅
深入审计 `reset_output_artifacts_for_fresh_run`（option 3 域研究延续）。结论：崩溃恢复设计
**正确且完整**——当 checkpoint 存在时（前次运行中断），cleanup 早返回不动任何产物，
backtest 引擎从 checkpoint resume 继续写入既有产物集；只有无 checkpoint 的全新运行才清理
stale 产物。write 路径全部用 overwrite 语义（`.write_text`），天然幂等。

此前该恢复不变量（`if checkpoint_path.exists(): return` 早返回）**无显式测试覆盖**——它是
paper_trading 崩溃恢复的命门，一旦未来重构静默回归将破坏 resume 安全。新增回归测试
`test_reset_output_artifacts_skips_cleanup_when_checkpoint_exists` 锁定：checkpoint 存在时
partial 产物（daily_events / timing_log / selection_artifacts / session_summary / trade_brief）
全部不被清理。服务于"更高确信"——崩溃恢复可被客观验证而非依赖 code review。
