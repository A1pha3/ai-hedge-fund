# 20. R20.31 产品调研 — 全功能完成后下一步规划 (2026-06-11)

> **背景**: R20.31 三轮迭代修复 10 bug (2 高危 NameError), 完成 P2-2/P2-6/P2-9 前端组件。全部 P0/P1/P2 功能 100% 完成。
>
> **调研方法**: 基于现有功能矩阵 + 业界 2025-2026 趋势 + 产品核心目标 ("找到未来 30 天最有投资价值的 A 股") 进行差距分析。

## 一、现有系统优势 (保持不变)

1. **20-agent persona 架构** — 业界唯一的多人格协作量化系统
2. **完整 CLI 决策链** — `--auto` → `--conviction-ranking` → `--why-not` → `--daily-brief`
3. **四策略融合 + 自适应权重** — 趋势/均值回归/基本面/事件情绪 + 市场状态动态调权
4. **因子 IC 分析** — 客观量化因子预测力
5. **连续推荐标记** — 跨日推荐稳定性追踪
6. **信号衰减检测** — 过期推荐自动预警

## 二、核心差距分析

以 "找到未来 30 天最有投资价值股票" 为标准, 对照业界 (聚宽/米筐/同花顺/Numerai):

| # | 差距 | 严重度 | 说明 |
|---|------|--------|------|
| G1 | **无历史推荐胜率闭环** | 高 | 系统每天推荐, 但缺少自动化的 "推荐→T+N 实际收益→反馈→调权" 闭环。`lookback_audit.py` 存在但需手动触发 |
| G2 | **无行业轮动策略** | 中 | 有行业集中度检查但无主动行业轮动策略。A 股行业轮动是最强 alpha 来源之一 |
| G3 | **无组合构建器** | 中 | 有条件单建议和再平衡, 但缺少从 Top N 推荐→组合权重→风险预算的端到端组合构建 |
| G4 | **无回测验证闭环** | 中 | 回测引擎存在但与日常选股流程脱节。用户无法快速验证 "如果按系统推荐操作, 过去 3 个月收益如何" |
| G5 | **推送通知缺失** | 低 | `--push` 已实现但需要手动配置 webhook。缺微信/Telegram 集成 |

## 三、Phase 5 需求提案 (按优先级排序)

### P3-1: 推荐闭环验证 (RECOMMEND→VERIFY→ADJUST) — P0

**问题**: 系统每天输出推荐, 但用户无法快速回答: "按系统推荐操作, 过去 N 天实际赚了多少?"

**方案**:
- `--verify-recommendations [--lookback=30]` — 自动回测过去 30 天的每日推荐, 计算实际 T+1/T+3/T+5 收益
- 输出: 推荐总胜率 / 平均收益 / 最大亏损 / 与基准 (沪深 300) 对比
- 自动识别: 哪些策略/因子贡献正收益, 哪些持续亏损
- 数据源: 复用 `recommendation_tracker.py` + `lookback_audit.py`

**用户价值**: 用户可以量化信任系统推荐的真实收益, 而非仅看 score_b 数字

**实现量**: 中 (2-3 天, 后端为主)

### P3-2: 策略动态权重校准 (Factor Weight Auto-tuning) — P1

**问题**: 当前策略权重由市场状态 (trend/range/mixed/crisis) 调整, 但权重参数是静态的。市场风格切换时可能滞后。

**方案**:
- `--auto-calibrate-weights` — 基于过去 N 天因子 IC, 自动调整策略权重
- IC 高的策略权重↑, IC 低/负的策略权重↓
- 输出: 调整前 vs 调整后权重对比 + IC 依据
- 回测验证: 展示校准后 vs 默认权重的历史表现差异

**用户价值**: 系统自适应市场风格, 不需要用户手动调权重

**实现量**: 中 (2 天, 纯后端计算)

### P3-3: 行业轮动信号 (Sector Rotation Signal) — P1

**问题**: 有行业集中度检查但无主动行业轮动推荐。

**方案**:
- 在 `--auto` 输出中增加 "本周强势行业 Top 5" 板块
- 计算方法: 行业动量 (5 日/20 日) + 北向资金行业分布 + 资金净流入排名
- 推荐 Top 行业中评分最高的标的 (行业+个股双重筛选)
- `--sector-rotation` 独立 CLI 命令

**用户价值**: 行业轮动是 A 股最强 alpha 之一, 系统主动推荐行业方向

**实现量**: 小 (1-2 天, 大部分数据源已有)

### P3-4: 推荐组合构建器 (Portfolio Builder) — P2

**问题**: 从 Top N 推荐→实际持仓缺少组合层面优化。

**方案**:
- `--build-portfolio [--top-n=10] [--risk-budget=0.05]` — 从推荐列表自动构建最优组合
- 约束: 行业分散 (≤20%/行业), 个股权重上限 (≤15%), 总风险预算
- 输出: 各标的权重 + 组合预期 Sharpe + 与等权组合对比

**用户价值**: 从 "推荐 N 只票" 升级到 "推荐一个组合"

**实现量**: 大 (3-5 天, 需优化求解器)

## 四、不建议做的功能 (避免臃肿)

| 功能 | 原因 |
|------|------|
| 深度学习因子挖掘 | 算力成本高, 4 策略 IC 分析已覆盖主流需求 |
| 实时行情推送 | CLI/Web 定时运行已够用, 实时需要 WebSocket 基础设施 |
| 社区/策略分享 | 运营成本高, 当前定位个人量化工具 |
| 多账户管理 | 个人量化场景不需要 |
| 自定义回测脚本编辑器 | 安全风险 + Jupyter 更适合 |

## 五、重复功能审查

| 功能 A | 功能 B | 关系 | 建议 |
|--------|--------|------|------|
| `--watchlist` (P0-5) | `--auto` + 自选池 | watchlist 是 auto 的子集 | **保持**: watchlist 提供独立关注列表管理 |
| `--explain` (P0-2) | `--why-not` (P0-8) | explain = 正向解释, why-not = 反向解释 | **保持**: 互补关系, 不重复 |
| `--confidence-calibration` (P0-9) | `--conviction-ranking` (P0-11) | calibration 提供校准数据, ranking 综合排名 | **保持**: 上下游关系 |
| `--winrate-dashboard` (P2-4) | `lookback_audit` | winrate 是 lookback 的可视化摘要 | **保持**: 不同粒度 |
| `--compare` (P1-8) | `--stock-detail` (P2-6) | compare = 多标的雷达图, detail = 单标的深度 | **保持**: 互补 |

**结论**: 现有功能无冗余, 各有独立用途。
