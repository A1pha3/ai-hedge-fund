# BTST 5D/+15% Round1 Unclassified Split Design

- **Date:** 2026-05-22
- **Topic:** BTST 5D/+15% round1 next-step design after empty shortlist
- **Recommended direction:** Build an `unclassified` split analysis layer first, then use its output to decide whether trend/breakout refinement deserves a narrower validation cycle

## 1. Problem statement

The freshly generated round1 artifact pack answered an important question: the current round1 surface does **not** contain a promotable factor candidate yet.

The key evidence is:

1. `row_count = 856`
2. `alpha_beta_gamma_shortlist = []`
3. `trend_continuation` is the best prototype, but only reaches `hit_rate_15pct = 0.2587`
4. `trend_family` and `breakout_family` are the best factor families, but both stay near `hit_rate_15pct = 0.2574`
5. `unclassified` still accounts for `454 / 856` rows

That last point changes the next question. The main uncertainty is no longer "which current family ranks first?" The stronger question is:

> How much of round1's alpha failure comes from genuinely weak trend/breakout quality, and how much comes from a very large `unclassified` bucket that dilutes the research surface?

If we do not answer that first, the next trend/breakout refinement cycle risks optimizing on noise instead of signal.

## 2. Goal and non-goals

### Goal

Design one narrow analysis cycle that:

1. explains why rows land in `event_prototype = unclassified`
2. separates missing-feature noise from near-threshold structure
3. identifies the most recoverable `unclassified` cohorts for the 5D/+15% objective
4. produces a clear recommendation about whether the next implementation cycle should focus on:
   - structural classification recovery
   - trend/breakout refinement
   - or both in a governed sequence

### Non-goals

- Do not promote any factor into `docs/prompt/find_actor/`.
- Do not update `ai-hedge-fund-btst`.
- Do not reopen broad factor-family search.
- Do not relax execution/beta gates just to raise hit counts.
- Do not rewrite the round1 contract itself.

## 3. Approaches considered

### Approach A - refine trend/breakout thresholds immediately

Use the current round1 leaderboard as-is and immediately design a more granular trend/breakout validation cycle.

**Pros**

- fastest path toward a more targeted factor test
- stays close to the current top-ranked prototypes

**Cons**

- ignores that more than half the rows are `unclassified`
- risks treating noise dilution as genuine trend/breakout weakness
- may produce a second round of alpha failure without explaining the surface

### Approach B - split the `unclassified` bucket first (**recommended**)

Design a dedicated analysis layer that decomposes `unclassified` rows by factor sparsity, candidate source, blocker shape, decision state, and near-threshold structure before any new refinement cycle.

**Pros**

- addresses the largest uncertainty in the current surface
- most directly explains why round1 alpha failed
- can reveal whether the top trend/breakout families are being suppressed by noise or are simply not strong enough
- creates a cleaner handoff for the next validation cycle

**Cons**

- adds one extra analytical step before the next factor-validation implementation
- may conclude that the current families are still weak even after de-noising

### Approach C - reopen broad round2 search with more prototypes and interactions

Expand the prototype/family/interaction search space immediately.

**Pros**

- maximizes exploration
- could discover a different alpha pocket

**Cons**

- violates the round1 principle of not widening scope before understanding current failure
- likely increases noise further
- creates search creep before current evidence is interpretable

## 4. Recommended design

The next design should be an **unclassified split board** that sits beside the round1 leaderboard, not instead of it.

It should answer four questions:

1. which `unclassified` rows are missing factor evidence entirely
2. which `unclassified` rows are near-threshold and may be structurally recoverable
3. which candidate sources / decisions / blocker families dominate the `unclassified` mass
4. whether removing or reclassifying the most recoverable `unclassified` cohorts would plausibly improve the round1 alpha surface enough to justify a narrower refinement cycle

The output of this cycle should not be "new factor accepted." It should be a governed routing decision for the next implementation step.

## 5. Design boundaries

This design stays narrow in five ways:

1. it only investigates rows already generated by the current round1 surface
2. it does not expand prototype, family, or interaction scope
3. it treats beta as already acceptable and avoids execution-gate loosening
4. it uses `unclassified` analysis to improve interpretation, not to manufacture alpha
5. it ends in a recommendation artifact, not a promotion artifact

## 6. Proposed component design

### 6.1 Raw-row sidecar or rebuild path

The split analysis needs access to row-level round1 data, not just the aggregate leaderboard artifact.

Two acceptable implementation shapes:

1. **preferred:** extend the round1 script to optionally emit a detailed row sidecar JSON
2. **fallback:** let the split script rebuild round1 rows directly from `selection_snapshot.json` files using existing round1 helpers

The recommended default is the fallback rebuild path first, because it avoids changing the already-verified round1 aggregate contract unless row-level reuse becomes expensive.

### 6.2 Unclassified cohort classifier

Create a focused classifier for rows where `event_prototype == "unclassified"` that groups them into interpretable buckets such as:

1. `missing_all_core_features`
2. `missing_breakout_inputs_only`
3. `missing_trend_inputs_only`
4. `near_trend_threshold`
5. `near_breakout_threshold`
6. `blocked_before_structure_matures`
7. `watchlist_only_low_signal`

The exact bucket names should be deterministic and derived from the same round1 inputs, not from freeform prose.

### 6.3 Split leaderboard artifact

Generate one artifact that summarizes each unclassified bucket with:

1. row count
2. closed-cycle count
3. 15% hit rate
4. mean 2-5d max return
5. candidate-source composition
6. decision composition
7. blocker / sparsity pattern summary
8. recoverability verdict

This artifact should make it obvious which buckets are merely dead noise and which deserve recovery work.

### 6.4 Recovery recommendation board

Generate a second artifact that collapses the split leaderboard into a short next-action board:

1. `ignore_noise`
2. `recover_threshold_near_miss`
3. `inspect_candidate_source_contract`
4. `revisit_blocker_family`
5. `advance_trend_breakout_refinement`

The board should recommend **one** primary next cycle, not a vague list of everything worth doing.

## 7. Alpha / Beta / Gamma responsibilities

### Alpha

Alpha owns:

- defining recoverability criteria for near-threshold rows
- determining whether de-noised cohorts are materially closer to the 55% / 15% objective
- deciding whether trend/breakout refinement has earned another round

### Beta

Beta owns:

- wiring the split-analysis artifact pipeline
- preserving deterministic bucket assignment
- keeping the analysis reproducible and aligned with existing round1 helpers

### Gamma

Gamma owns:

- judging whether the split board justifies another validation cycle
- preventing the process from turning into uncontrolled search expansion
- keeping promotion posture at hold until a later validation cycle actually clears alpha

## 8. Data flow

The next cycle should flow in this order:

1. read the current round1 aggregate artifact
2. rebuild or load row-level round1 data
3. filter `event_prototype == "unclassified"`
4. assign each row to one deterministic split bucket
5. compute per-bucket objective support and recoverability summaries
6. synthesize one recovery recommendation board

## 9. Error handling and fail-closed rules

The split cycle must fail closed when:

1. round1 source artifacts cannot be resolved
2. row rebuild and aggregate artifact disagree materially on counts
3. bucket assignment logic sees an unsupported state
4. required factor inputs are malformed
5. recoverability metrics cannot be computed consistently

No bucket should silently default to "recoverable" when evidence is missing.

## 10. Validation design

Validation for this design should prove:

1. the split script reproduces the expected `unclassified` population from the round1 surface
2. each bucket assignment is deterministic and testable
3. missing-feature rows are separated from near-threshold rows
4. the recommendation board can distinguish:
   - noise worth ignoring
   - rows worth structural recovery
   - situations where trend/breakout refinement should advance

## 11. Promotion rules

This cycle still does **not** promote factors.

Promotion remains blocked until a later cycle demonstrates:

1. alpha clears the 55% / 15% objective
2. beta stays tradeable
3. gamma still has multi-window support

## 12. Expected outcome

If this design works, the next implementation cycle will stop guessing about why round1 failed. We will know whether the dominant problem is:

1. `unclassified` noise overwhelming the surface
2. insufficient trend/breakout quality even after de-noising
3. a specific recoverable bucket that deserves focused factor work

That is the cleanest route to the next truly evidence-backed BTST 5D/+15% improvement.
