1. 在“买入后 5 个交易日内，55% 以上概率涨幅超过 15%”这个高门槛目标下，应该先用什么**方法**去挖掘因子，而不是先盲目堆因子。

3. 什么样的方法更可能挖出真正能进入 BTST 运行时体系的强因子，什么样的方法只会制造过拟合幻觉。

## 1. 先说结论

我们推荐的首选方法，不是“全市场先扫一遍所有因子，再挑相关性最高的”，而是：

**以横截面事件驱动为主线，用可执行买点定义标签，用多窗口稳健性筛选控制过拟合，再在事件子宇宙里挖掘因子与因子交互。**

原因很简单：

1. “5 日内涨超 15%”是一个**稀缺且强非线性**的目标，直接在全市场日频样本上扫相关性，极容易被噪声、极端样本和市场阶段漂移误导。
2. BTST 本质上是“今天收盘前识别明天到未来几天仍有冲击力的结构”，所以研究单位不应是“所有股票-所有日期”，而应是“**具备短线事件原型的股票-日期切片**”。
3. 这个目标必须从一开始就把**可买到、能成交、不过度滑点、不过度拥挤**纳入研究框架，否则会挖出一批纸面上优越、落地后却不可交易的候选。

所以，这条主线可以直接压成后文会展开的 3 步：

1. **先锁可执行标签**：统一买点和收益观察口径。
2. **再锁事件原型宇宙**：把样本限制在“可能成为 BTST 赢家”的结构区域。
3. **最后做因子与治理**：在事件宇宙中完成因子生成、交互挖掘、多窗口筛选和 rollout 治理。

## 2. 三种可选挖掘路线，以及为什么推荐第一种

### 路线 A：横截面事件驱动 + 多窗口稳健性筛选（推荐）

做法：

1. 先定义短线事件原型，例如突破启动、趋势延续、缩量蓄势后放量、催化共振等。
2. 只在这些事件切片里研究因子与未来 5 日 15% 达标概率的关系。
3. 用滚动窗口、purged walk-forward、稳定性约束、去冗余约束筛选真正可复用的因子。

优点：

1. 最符合 BTST 的交易语义。
2. 样本噪声更低，更容易找到“结构性高赔率区”。
3. 更容易把结果直接转成 runtime 的 score、gate、relief 和 blocker。

缺点：

1. 前期要先花力气定义事件原型和标签口径。
2. 如果事件原型定义太窄，会丢掉一部分潜在赢家。

### 路线 B：先定义 5 日 15% 标签，再做全市场反向搜因子

做法：

1. 对全市场每日样本直接打上“未来 5 日是否涨超 15%”标签。
2. 再从大量技术、行为、成交、题材、市场状态特征中筛选预测因子。

优点：

1. 启动快，看起来最全面。
2. 容易做大规模特征搜索。

缺点：

1. 稀有事件占比太低，噪声极大。
2. 很容易挖出“事后解释因子”，不是可执行交易因子。
3. 结果通常缺乏 BTST 结构语义，难落入 runtime。

### 路线 C：双轨并行，事件模板挖掘 + 标签反推因子

做法：

1. 一条线在事件宇宙中挖结构因子。
2. 一条线在全样本中做标签反推和 feature discovery。
3. 只保留两条线都支持、且通过可交易性检验的因子。

优点：

1. 容易发现事件模板未覆盖的新特征。
2. 对探索型研究有帮助。

缺点：

1. 成本最高。
2. 很容易把研究节奏拖散。
3. 如果当前目标是先找到“稳定的方法”，这条线过重。

### 最终建议

当前阶段的结论可以直接写成：**路线 A 做主线，路线 C 只保留为补充探索，路线 B 不作为当前主研究入口。**

## 3. 最终方法设计

### 3.1 研究目标不是“找相关性”，而是找“可执行 uplift”

这次的方法设计，主目标不是提升一般收益率均值，而是寻找满足下列条件的因子或因子组合：

1. 以**可执行买点**进入后，未来 5 个交易日内，出现 `>= 15%` 最大涨幅的概率显著提升。
2. 这种提升在多个市场窗口里稳定存在，不依赖少数极端行情。
3. 这种提升能够在成交、滑点、拥挤、风险约束下仍保持可交易。
4. 因子最终能沉淀成 runtime 可消费的 score、gate、threshold、rank cap、relief 或 blocker。

因此，这套方法实际要优化的不是一个单指标，而是下面这个组合目标：

`方法得分 = 目标达标 uplift × 稳健性 × 可交易性 × 去冗余程度 × rollout 可治理性`

---

### 3.2 先统一标签定义，不允许收益口径漂移

这是 alpha 和 beta 的第一共同任务。

#### 主标签

主标签统一定义为：

1. 信号产生于 `T` 日收盘前。
2. 买入参考价不是 `T` 日收盘价，而是 **`T+1` 可执行买点**。
3. 推荐优先使用 `T+1` 开盘可成交参考价，或更保守的 `T+1` 开盘后短时 VWAP 代理价。
4. 观察区间是 `T+1` 到 `T+5`。
5. 若这 5 个交易日内任一天的盘中最高价相对买入参考价涨幅 `>= 15%`，则记为主标签命中。

业务层可统一把这类目标简称为：

`future_5d_high_ge_15pct`

进入首轮实际研究时，再拆成后面的 raw / executable 双账本版本。

#### 次标签

为了避免只看“是否命中”而忽略交易质量，还应同时定义：

1. `future_5d_close_ge_8pct`
2. `future_3d_high_ge_10pct`
3. `future_5d_max_drawdown_le_x`
4. `t1_open_gap_within_execution_band`
5. `entry_day_liquidity_pass`

目的：

1. 主标签回答“能不能抓到高赔率目标样本”。
2. 次标签回答“这块肉是不是大概率真能吃到，而不是纸面高点”。

#### 首轮统一买点代理规则

首轮研究必须把买点代理规则固定下来，不能每轮因子扫描都换口径。

推荐的首轮默认顺序是：

1. **主研究口径**：`T+1` 开盘可成交参考价。
2. **稳健性复核口径**：`T+1` 开盘后短时 VWAP 代理价。
3. **禁止口径**：直接用 `T` 日收盘价代替真实买点。

这样设计的原因：

1. 主研究口径要足够贴近真实执行。
2. 稳健性复核口径要能防止“只在开盘瞬间成立”的伪优势。
3. 如果一开始就混用多个买点代理，后面无法判断 uplift 到底来自因子，还是来自买点口径漂移。

#### 首轮标签采用“双账本”

首轮不建议只保留一个标签版本，而建议保留两套并行账本：

1. **价格路径账本**：只问价格路径上有没有在 `T+1 ~ T+5` 触达 `15%`。
2. **可执行账本**：在价格路径账本基础上，再扣除明显不可复制成功的样本。

对应地，建议同时保留：

1. `future_5d_high_ge_15pct_raw`
2. `future_5d_high_ge_15pct_executable`

为什么必须双账本：

1. raw 标签回答“这个结构有没有理论爆发力”。
2. executable 标签回答“这个爆发力是不是交易上有机会复制”。
3. 如果一个因子只在 raw 标签上好看、在 executable 标签上明显掉队，它更像结构提示因子，不该直接进入主交易层。

#### 首轮排除规则：什么样本不计入 executable 成功

beta 首轮必须先定义清楚哪些样本虽然价格触达过目标，但默认不算 executable 成功。

至少包括下面 5 类：

1. `T+1` 明显无法成交或成交空间极差的样本。
2. 高开过度、瞬时跳空过大，导致研究买点与真实成交严重脱钩的样本。
3. `T+1` 或持有窗口内缺少可靠成交与价格质量的样本。
4. 流动性明显不足、稍加容量就会显著扭曲结果的样本。
5. 因停牌、异常状态或关键行情缺失而无法形成可比观察窗的样本。

这里最重要的纪律不是“排除得越多越好”，而是：

**排除规则必须在首轮开始前固定，不能在看到结果后反向修改。**

#### 次标签的冻结纪律

首轮可以同时看多个次标签，但一旦进入某一轮候选筛选，就必须冻结：

1. 哪些次标签参与解释；
2. 每个次标签的阈值如何定义；
3. 哪些次标签只是观察项，哪些会参与候选升降级。

推荐的首轮分工是：

1. `future_5d_close_ge_*` 负责看“不是只有高点漂亮”。
2. `future_3d_high_ge_*` 负责看“爆发是否太晚才发生”。
3. `future_5d_max_drawdown_le_*` 负责看“拿到目标前要不要先吃过大回撤”。
4. `t1_open_gap_within_execution_band` 与 `entry_day_liquidity_pass` 负责看“这是不是交易上就不该碰的样本”。

如果某次实验在结果出来后才临时改次标签阈值，这轮统计应直接降级处理。

#### 围绕 `5 日 / 55% / 15%` 主目标的标签与样本分层

如果只把样本简单分成“命中 / 未命中”两类，方法很容易挖出两种伪优因子：

1. **高胜率但低赔率**：很容易抓到小肉，却很难真正打到 `5 日 15%`。
2. **高赔率但低可复制性**：少数样本炸得很猛，但大多数时候买不到、拿不住、复现不了。

因此，围绕短线**高胜率 + 高盈亏比**主目标，建议不要只保留一个二元标签，而是把研究样本至少拆成下面 4 层：

| 层级 | 样本定义 | 研究意义 | 是否作为主挖掘目标 |
| --- | --- | --- | --- |
| **A 层：核心达标样本** | executable 账本下 `5` 日内触达 `15%`，且次标签不过分恶化 | 最接近“高胜率高盈亏比”主目标的样本 | **是** |
| **B 层：结构高赔率但执行偏弱样本** | raw 触达 `15%`，但 executable 明显掉队，或回撤/成交质量较差 | 说明结构可能有爆发力，但交易复制性不足 | 否，更多用于 beta / gamma 诊断 |
| **C 层：高胜率低赔率样本** | `3` 日/`5` 日表现不差，但始终难以触达 `15%` | 说明它更像稳态赚钱结构，而不是高赔率结构 | 否，主要用于防止把“安全但不够强”误当主线 |
| **D 层：噪声 / 不可复制样本** | impossible-fill、异常跳空、关键行情缺失或明显失真样本 | 主要用于排除，不应用来支撑因子优越性 | 否 |

这 4 层里，真正值得 alpha 优先挖的，不是“让总体上涨概率更大”的因子，而是**能显著提高 A 层占比，同时不主要把样本推向 B / C / D 层**的因子。

换句话说，真正好的短线强因子，至少应同时满足：

1. **更容易把样本推向 A 层，而不是只推向 C 层**  
   - 否则它只是“更容易小幅上涨”，不是“更容易达到高赔率目标”。
2. **不会主要通过 B 层抬高统计结果**  
   - 否则它只是“纸面爆发力强”，不是“可复制强因子”。
3. **不会让 D 层噪声样本变多**  
   - 否则 uplift 只是在脏样本里变亮。

在这 4 层之外，还建议对 A / B / C 三层继续做一层**路径分层**：

1. **快达标样本**：`T+1 ~ T+3` 就接近或触达目标。  
2. **慢达标样本**：主要靠 `T+4 ~ T+5` 才勉强触达。  
3. **尖峰样本**：盘中高点触达，但收盘承接和持有质量较差。  
4. **平滑样本**：涨幅路径更干净、回撤负担更轻。

这样做的原因，是为了避免把下面两类样本混成一类：

1. **真强势样本**：启动后很快给出空间，且承接不差。  
2. **勉强达标样本**：直到窗口后段才靠一下尖峰摸到 `15%`。

对“高胜率高盈亏比”的方法设计来说，这两者都叫“命中”，但质量差别非常大。  
如果不做路径分层，alpha 很容易把“晚到尖峰”误看成“高质量高赔率样本”。

因此，建议每次评估因子时，至少同时回答 3 个问题：

1. **它提升的是 A 层，还是主要提升 C 层？**  
2. **它贡献的是 executable A 层，还是主要停留在 raw 的 B 层？**  
3. **它改善的是快达标 / 平滑样本，还是只改善慢达标 / 尖峰样本？**

如果一个因子：

- 主要提升 **A 层 executable 快达标样本** → 更值得进入 P0 / 候选层；  
- 主要提升 **C 层样本** → 更像稳态辅助因子，不该误升主线；  
- 主要提升 **B 层样本** → 更像结构提示或 frontier 线索，应先退回 beta / gamma；  
- 主要抬高 **D 层附近噪声** → 直接降级或排除。

一句话说，**围绕短线高胜率高盈亏比因子挖掘时，最好的标签方法不是只问“有没有在 5 日内涨超 15%”，而是继续追问：它命中了哪一层、以什么路径命中、是不是 executable、是不是高质量高赔率样本**。

#### 基于 A / B / C / D 分层样本，因子评估表应固定展示哪些核心指标

如果已经把样本拆成 A / B / C / D 四层，再继续只看一个总 uplift，信息仍然不够。  
真正有用的评估表，必须能回答：**这个因子到底在提升哪一层，伤害哪一层，以及它改善的是高质量命中，还是低质量假象。**

建议每个候选因子都固定展示下面 10 项：

| 指标 | 为什么必须看 | 更理想的方向 |
| --- | --- | --- |
| **A 层占比变化** | 直接看核心 executable 高赔率样本有没有变多 | 上升 |
| **B 层占比变化** | 防止把纸面高赔率错当主线 | 不显著上升，最好下降 |
| **C 层占比变化** | 防止把高胜率低赔率结构误升主线 | 不应成为主要改善来源 |
| **D 层占比变化** | 防止噪声 / 不可复制样本扩张 | 下降 |
| **A/B 比** | 看可复制强样本相对纸面爆发样本是否更优 | 上升 |
| **A/C 比** | 看是否更偏高质量高赔率样本，而不是只偏小幅稳态样本 | 上升 |
| **快达标占比** | 看命中是否更偏早段爆发，而不是拖到窗口后段 | 上升 |
| **平滑样本占比** | 看涨幅路径是否更干净、回撤负担是否更轻 | 上升 |
| **raw / executable 差值** | 看结构优势有没有被执行口径吃掉 | 收敛或缩小 |
| **coverage** | 看改善是不是只来自过窄样本簇 | 保持在可解释范围内，不可过稀 |

如果希望更贴近“高胜率高盈亏比”双目标，建议把这 10 项再压成 4 个总面板：

1. **主目标面板**  
   - A 层占比变化  
   - A/B 比  
   - A/C 比
2. **路径质量面板**  
   - 快达标占比  
   - 平滑样本占比
3. **执行可信度面板**  
   - raw / executable 差值  
   - B 层占比变化  
   - D 层占比变化
4. **统计稳健性面板**  
   - coverage  
   - 跨窗同向性  
   - uplift 方差

为了避免“指标很多，但不会读”，建议固定按下面顺序解释：

1. **先看 A 层有没有真的变多**  
   - 如果 A 层不增，后面很多改善都不算主线改善。
2. **再看改善是不是主要落在 B / C / D 层**  
   - 如果主要是 B 层，说明更像纸面爆发；  
   - 如果主要是 C 层，说明更像稳态小肉；  
   - 如果 D 层也抬高，说明更像噪声扩张。
3. **再看命中路径质量是不是更好**  
   - 快达标 / 平滑样本是否同步提升。
4. **最后才看覆盖度和稳健性**  
   - 防止把很窄样本里的漂亮结果误判为通用主线。

也可以直接记住下面这条最短解读法：

- **A 升，B/C/D 不乱升，快达标和平滑占比也升** → 更像主线强因子。  
- **A 不升，只是 C 升** → 更像高胜率低赔率辅助因子。  
- **A 不升，只是 B 升** → 更像纸面高赔率提示因子。  
- **D 也升** → 更像脏样本驱动，直接警惕。

一句话说，**A / B / C / D 分层后的因子评估表，核心不是展示更多数字，而是强制回答：它到底在增加高质量 executable 高赔率样本，还是只在 B / C / D 层制造看起来不错的假改善**。

#### 一页版因子评估卡模板

如果不把上面的指标压成固定版式，团队很容易每轮都换说法，最后看似讨论了很多，实际上无法横向比较。  
因此建议每个候选因子都固定出一张“一页版评估卡”，并且按下面顺序填写：

| 模块 | 固定内容 | 目的 |
| --- | --- | --- |
| **因子身份** | 因子名、所属原型、所属家族、当前轮次、对照基线 | 先把比较对象锁死，避免换口径 |
| **主目标结果** | A 层占比变化、A/B 比、A/C 比 | 先回答有没有真正靠近 5 日 / 55% / 15% 主目标 |
| **路径质量** | 快达标占比、平滑样本占比、慢达标 / 尖峰样本变化 | 判断命中质量，而不是只看命中有无 |
| **执行可信度** | raw / executable 差值、B 层变化、D 层变化 | 防止纸面改进掩盖执行落差 |
| **统计稳健性** | coverage、跨窗同向性、uplift 方差 | 防止单窗好运或过窄样本假亮点 |
| **语义建议** | 更像准入（admission）/ 过滤门（gate）/ 排序分（score）/ 放行修正（relief）/ 观察层 | 提前约束运行时映射，不让后面乱放位置 |
| **一句话结论** | 升级、观察、降级、排除 | 让每轮输出能被快速消费 |

推荐直接使用下面这个最简模板：

```text
因子名：
事件原型 / 因子家族：
本轮基线：

一、主目标结果
- A 层占比变化：
- A/B 比变化：
- A/C 比变化：

二、路径质量
- 快达标占比变化：
- 平滑样本占比变化：
- 慢达标 / 尖峰样本是否扩张：

三、执行可信度
- raw / executable 差值变化：
- B 层变化：
- D 层变化：

四、统计稳健性
- coverage：
- 跨窗同向性：
- uplift 方差：

五、运行时语义建议
- 更适合：准入（admission）/ 过滤门（gate）/ 排序分（score）/ 放行修正（relief）/ 观察层
- 不适合放到：

六、一句话决策
- 升级 / 观察 / 降级 / 排除
- 原因：
```

如果想把这张卡真正用起来，建议固定 3 条纪律：

1. **每张卡都必须先写主目标结果，再写故事**  
   - 先看 A 层和 A/B、A/C，再看解释，不允许先写叙事后补数字。
2. **每张卡都必须同时写“更适合放哪”和“明确不适合放哪”**  
   - 这样可以减少运行时语义漂移。
3. **每张卡都必须给出升级 / 观察 / 降级 / 排除四选一，不接受模糊表态**  
   - 否则卡片会变成信息堆积，而不是决策材料。

进一步说，一页版评估卡最重要的价值，不是好看，而是把 alpha、beta、gamma 的关注点压到同一张纸上：

- **alpha** 负责解释主目标结果和统计稳健性；  
- **beta** 负责解释执行可信度和 raw / executable 裂口；  
- **gamma** 负责解释运行时语义建议和升级 / 观察 / 降级决定。

一句话说，**固定评估卡的意义，是把“因子研究结论”从口头判断，压成可横向比较、可追溯、可直接进入 rollout 讨论的标准输出件**。

#### 一页版评估卡写完后，如何固定判成“升级 / 观察 / 降级 / 排除”

如果评估卡只负责展示信息，不负责固定判决，团队很容易出现同一张卡在不同会议里被解读成不同结论。  
因此建议在方法上直接规定：**评估卡写完后，必须先按固定判定顺序做四选一，再进入后续讨论。**

先强调一个边界：

- 这里的**升级**，是指“相对当前研究层级的提升”，例如从观察层升到候选层，或从候选层升到交互候选层；  
- 它**不等于直接进入运行时发布**，后面仍然要经过 `G0-G5` rollout gate。

建议按下面顺序判：

1. **先问：它是不是已经具备升级资格？**  
2. **如果不是，再问：它是不是值得继续观察？**  
3. **如果连观察都不值，再问：它应该降级还是直接排除？**

可以固定成下面这张四选一判定表：

| 决策 | 默认适用情形 | 最少应满足什么 | 最不该出现什么 |
| --- | --- | --- | --- |
| **升级** | 已经能作为更强候选进入下一层验证 | A 层改善明确；A/B、A/C 不恶化；B/D 不扩张；raw / executable 差值不放大；coverage 与跨窗同向性可接受；运行时语义大体清楚 | 主要靠 B 层、C 层或单窗样本撑结果 |
| **观察** | 结构有信号，但证据还不够稳定或不够可执行 | 至少有一条清楚主线值得保留；没有明显 D 层污染；失败原因能写成 1 句；下一轮知道该补哪一层 | 说不清到底在观察什么，只是“感觉还有点意思” |
| **降级** | 还有局部信息价值，但已不适合作为当前主候选 | A 层改善不足或不稳定；改善主要来自 C 层或局部 raw；仍可作为辅助提示、边界线索或负向约束候选 | 继续按主线强因子口径讨论它 |
| **排除** | 已经不值得继续占用主研究资源 | D 层扩张明显，或主要优势来自不可复制样本、单窗偶然、口径漂移、语义混乱 | 试图靠补故事、补交互、补阈值继续拖延退出 |

为了减少主观发挥，建议再加 4 条最短路由规则：

1. **A 层真升，B/D 不乱升，执行裂口没扩大**  
   - 默认优先判 **升级**。
2. **A 层还不够稳，但主问题层很清楚，且没有脏扩张**  
   - 默认判 **观察**。
3. **A 层没起来，主要只是 C 层或局部 raw 变好**  
   - 默认判 **降级**，把它退回辅助层或结构提示层。
4. **D 层扩张、raw / executable 裂口加大、只能靠单窗讲故事**  
   - 默认判 **排除**。

如果想把判定再压得更刚性，可以直接按下面顺序读卡：

1. **A 层有没有实质改善**  
   - 没有，就基本不进入升级讨论。
2. **B / C / D 哪一层在主导改善**  
   - B 主导，更像纸面高赔率；  
   - C 主导，更像高胜率低赔率；  
   - D 扩张，直接危险。
3. **raw / executable 差值有没有扩大**  
   - 扩大，则优先怀疑执行或口径问题，而不是给更高层级。
4. **coverage 和跨窗同向性是否足够支撑“继续占资源”**  
   - 不足时，观察或降级通常优先于升级。
5. **运行时语义是否已经基本明确**  
   - 连准入（admission）/ 过滤门（gate）/ 排序分（score）/ 放行修正（relief）都放不清时，不应升到更高层。

也可以记住下面这版最短结论法：

- **强 A、低噪声、低裂口、语义清楚** → 升级。  
- **有主线、但还不稳** → 观察。  
- **有信息、但不是主线强因子** → 降级。  
- **脏、漂、不可复制、不可解释** → 排除。

为了避免和后面的 gamma gate 冲突，还要固定一条纪律：

- **评估卡的四选一，是研究层的资源配置决定；G0-G5 是发布治理层的决定。前者不能替代后者，后者也不要跳过前者。**

一句话说，**评估卡的真正闭环，不是“把指标写全”，而是把指标强制压成统一判决；只有先固定升级 / 观察 / 降级 / 排除，后面的候选层级和 rollout gate 才不会漂。**

#### 什么情况下，才允许从单因子推进到“交互候选层”

很多研究一到中段就容易犯一个错：  
**单因子还没真正站稳，就急着把两个普通因子拼在一起，希望组合后看起来更强。**

这类做法最危险的地方不在于“复杂”，而在于它会把本来还能诊断的问题，迅速变成很难追责的黑盒改进。  
因此建议方法上固定一条硬纪律：

- **交互候选层不是用来救一阶主线的，而是用来放大已经站稳的主线。**

只有当下面 5 条同时满足时，才允许把某个因子从单因子讨论推进到交互候选层：

1. **已经有一个站稳的 P0 主因子**  
   - 这个 P0 在 executable 账本下，A 层改善已经相对明确；  
   - 如果 P0 还在摇摆，就先不要做交互晋升。
2. **拟加入的第二因子职责清楚**  
   - 它是确认型（P1）、约束型（P2），还是明确边界下的放行修正（relief）候选，要能说清；  
   - 不能一会儿像确认，一会儿像主线，一会儿又像放行条件。
3. **组合后的改善，主要落在 A 层 executable，而不是 B / C / D 层假改善**  
   -    如果交互只是让 B 层更活跃，或让 raw 看起来更好，不应进入交互候选层。
4. **组合没有明显扩大 raw / executable 裂口**  
   - 如果一组合就变得更难成交、更依赖极端样本，这通常不是“更强”，而是“更虚”。
5. **组合后的解释仍然不变形**  
   - 组合应该是“让既有主线更清楚”，而不是把原来简单的交易语义扭成只有回测能解释的复杂门。

可以直接压成下面这张判定表：

| 判断项 | 允许进交互候选层时应满足什么 | 不满足时该怎么做 |
| --- | --- | --- |
| **P0 是否已站稳** | 单因子主线已明确，不再轮流漂移 | 继续做单因子，不做交互晋升 |
| **第二因子职责是否清楚** | 能明确归到 P1 / P2 / 放行修正（relief）候选之一 | 先定语义，不急着组合 |
| **改善是否落在 A 层 executable** | 组合后 A 层质量继续改善，B / D 不扩张 | 退回观察或降级 |
| **执行裂口是否受控** | raw / executable 差值没有明显放大 | 先退回 beta 修执行口径 |
| **解释是否仍然可讲清** | 组合后的交易语义仍能用 1 到 2 句说清 | 不升交互候选层 |

为了减少“组合后数字更好，就忍不住想升”的冲动，建议再固定下面 4 条禁令：

1. **禁止在 P0 还没选稳时，先做交互晋升**  
   - P0 一直换人，说明一阶主线未收敛，交互只会让问题更模糊。
2. **禁止用两个都不强的因子，靠组合硬凑主线**  
   - “1 个不够强 + 1 个也不够强 = 主线” 通常只是过拟合警报。
3. **禁止把放松 P2 约束当成交互改进**  
   - 如果所谓交互改善只是因为约束放宽了，不算真正交互价值。
4. **禁止同时并行验证多组二阶交互**  
   - 一次只允许围绕一个已站稳 P0，验证少量最有交易语义的组合。

也可以记住下面这条最短路由法：

- **先有站稳的 P0，再问谁能确认它、约束它、或在明确边界下补救它。**  
- **如果组合是在放大主线，才进交互候选层；如果组合是在掩盖主线，直接退回单因子层。**

为了和前面的四选一决策保持一致，建议把“交互候选层”理解成：

- **它不是一个更好看的结论，而是一个更受控的下一步动作。**

也就是说，某个因子被放进交互候选层，不代表它已经更强；只代表：

1. 单因子主线已经足够清楚；  
2. 这个组合值得被单独验证；  
3. 下一轮可以明确知道该验证哪一组组合，而不是重新大面积扩搜。

一句话说，**允许进入交互候选层的前提，不是“组合后看起来更猛”，而是“单因子主线已站稳、第二因子职责清楚、A 层 executable 改善真实、解释没有变形”；交互应该放大主线，而不是遮盖主线。**

#### 进入交互候选层后，如何做“受控组合验证”

把因子送进交互候选层，只解决了“值不值得测”这个问题。  
接下来的关键，不是马上扩更多组合，而是把验证流程压到足够小，确保最后能回答：

**这组组合到底是在真实放大主线，还是只是在扩大复杂度。**

建议交互候选层一律按下面 4 步验证，不要自由发挥：

1. **冻结 1 个锚点 P0，不换主线**  
   - 组合验证必须围绕一个已经站稳的 P0 展开；  
   - 验证期间不允许把另一个因子偷偷升成新主线。
2. **只比较 3 个版本，不开更多支线**  
   - `P0 单独`  
   - `第二因子单独`  
   - `P0 + 第二因子组合`  
   - 不要同时加第三个因子，也不要同时试很多阈值分支。
3. **固定只看 5 类结果，不让组合实验无限长大**  
   - A 层 executable 是否继续改善；  
   - B / C / D 层有没有被一起抬高；  
   - raw / executable 差值是否扩大；  
   - coverage 是否缩得过头；  
   - 组合解释是否仍然能被 1 到 2 句讲清。
4. **验证后只允许给出 4 类结论，不准写模糊话**  
   - 保留为交互候选；  
   - 退回单因子候选；  
   - 把第二因子降级为辅助 / 约束 / relief 线索；  
   - 直接排除这组组合。

可以把这件事压成一张最小验证板：

| 验证块 | 固定要回答什么 | 目的 |
| --- | --- | --- |
| **锚点板** | 这次围绕哪个 P0 验证，为什么不换主线 | 防止组合验证中途漂主线 |
| **对照板** | P0 单独、第二因子单独、P0+第二因子 三者分别怎样 | 保证组合增益可比较 |
| **质量板** | A 层、B/C/D、快达标、平滑样本是否改善 | 判断是不是高质量增益 |
| **执行板** | raw / executable 裂口、流动性、不可复制样本是否恶化 | 防止组合变成更难交易的纸面优化 |
| **结论板** | 保留交互 / 退回单因子 / 降级角色 / 排除 | 强制收口到动作 |

为了让组合验证真正“受控”，建议再加 5 条硬纪律：

1. **一次只验证 1 个 P0 + 1 个第二因子组合**  
   - 不并行铺很多组合，再靠事后挑最好看的那组。
2. **第二因子阈值只能小范围固定，不准边看结果边漂**  
   - 否则组合实验很容易变成阈值过拟合。
3. **组合改善若主要来自 coverage 大幅收缩，不算真增益**  
   - 这通常只是样本被越切越窄。
4. **组合若把 raw 结果拉高、却把 executable 裂口拉大，优先判失败**  
   - 这类组合更像纸面增强器，不像交易增强器。
5. **组合若解释开始依赖复杂条件串联，优先退回单因子层**  
   - 如果连 alpha / beta / gamma 都讲不清，后面更不可能稳定 rollout。

也可以直接记住下面这条最短验证法：

- **先看 P0 单独能不能站住，再看组合是不是在 executable A 层继续加分，而不是靠更窄 coverage、更大裂口、更复杂解释换来的。**

为避免和前面的四选一、交互晋升门槛脱节，建议把交互验证后的结论固定解释成：

- **保留交互候选**：这组组合值得进入下一轮继续受控验证；  
- **退回单因子候选**：主线仍然主要来自 P0，本次组合没有形成足够独立增益；  
- **降级第二因子角色**：它更像辅助解释、约束或边界修正，而不是组合放大器；  
- **排除组合**：这组组合主要是在制造复杂度，不是在制造高质量收益结构。

一句话说，**交互候选层之后最重要的方法纪律，不是继续试更多组合，而是把验证压成“一个锚点、三个对照、五类结果、四类结论”；只有这样，组合验证才是在放大主线，而不是重新打开一轮失控扩搜。**

#### 什么情况下，才允许进入“运行时假设层”

到了这一步，最容易犯的错是不知不觉把“可以形成运行时假设”说成“已经差不多能上线”。  
这两者差别很大：

- **进入运行时假设层**，表示研究已经能把这个候选放进一个清楚的运行时语义位里，开始讨论它未来更像准入（admission）、过滤门（gate）、排序分（score）、阈值（threshold）、放行修正（relief）还是阻断（blocker）；  
- **运行时就绪 / 可发布**，则意味着它后面还要经过更严格的 rollout、回放、结果面验证和风险约束。

因此建议明确规定：**运行时假设层是研究层到系统层的接口草图，不是发布层。**

只有下面 6 条同时成立时，才允许某个候选进入运行时假设层：

1. **候选层级已经基本稳定**  
   - 它不再在观察层、候选层、交互候选层之间来回漂移；  
   - 如果层级结论还在频繁变，就先不要谈 runtime 假设。
2. **运行时语义已经有唯一主答案**  
   - 能明确说它更像准入（admission）、过滤门（gate）、排序分（score）、阈值（threshold）、放行修正（relief）或阻断（blocker）中的哪一种；  
   - 不能同时写成两三种都像。
3. **它改变的是清楚的结果面，而不是抽象 uplift**  
   - 例如更像减少 blocked、改善 near-miss、还是在 selected 内部重新排序；  
   - 如果只能说“统计上变好一些”，还不够进入 runtime 假设层。
4. **beta 能解释它的执行意义**  
   - 加入运行时后，它更可能改变买点前筛选、买点后确认、排序优先级，还是边界放行；  
   - 如果执行意义说不清，假设层会很空。
5. **gamma 能说明它的影响面受控**  
   - 不会因为进入某个语义位，就把原本无关的大量样本一起污染；  
   - 如果影响面不可控，只能继续留在离线研究层。
6. **失败后的退路已经写清**  
   - 如果运行时回放不支持这条假设，应退回候选层、交互候选层，还是直接降级；  
   - 没有退路的运行时假设，通常会被越推越硬。

可以把它压成下面这张最小准入表：

| 判断项 | 进入运行时假设层前必须满足什么 | 不满足时该怎么处理 |
| --- | --- | --- |
| **层级是否稳定** | 观察 / 候选 / 交互结论已基本不漂 | 继续停在当前研究层 |
| **语义是否唯一** | 有唯一主语义位，不混写多种角色 | 回到语义判断，先去歧义 |
| **结果面是否明确** | 能说清主要改变 selected / near-miss / blocked 哪一面 | 继续留在离线研究层 |
| **执行意义是否明确** | beta 能说明它在实际流程里改变什么 | 先补执行解释 |
| **影响面是否受控** | gamma 能说明不会大面积误伤无关样本 | 退回影响面排查 |
| **退路是否明确** | 回放失败后知道退回哪一层 | 不升 runtime 假设层 |

为了防止“假设层”被滥用，建议再加 4 条硬纪律：

1. **禁止把运行时假设层当成发布预告**  
   - 它只是“可以开始画 runtime 位置”，不是“马上就能接到系统里”。
2. **禁止在语义不唯一时硬写激活假设（activation hypothesis）**  
   - 一会儿像排序分（score），一会儿像过滤门（gate），通常说明研究还没收敛。
3. **禁止只凭离线 uplift 就进入运行时假设层**  
   - 必须同时说清结果面、执行意义和影响边界。
4. **禁止没有回退路径的运行时假设**  
   - 任何假设都必须预先写明：如果回放不支持，退回哪一层。

也可以直接记住下面这条最短判断法：

- **能否进入运行时假设层，不看它“强不强”，而看它是否已经能被清楚地放进一个语义位、解释一个结果面、对应一个执行动作，并且失败时知道退回哪里。**

如果要把这层和前面的层级关系压成一句话，可以写成：

- **观察层 / 候选层 / 交互候选层** 解决的是“值不值得继续研究”；  
- **运行时假设层** 解决的是“如果继续推进，它未来更可能在系统里扮演什么角色”。

一句话说，**进入运行时假设层的前提，不是研究结论看起来已经足够强，而是它已经具备“唯一语义、明确结果面、可解释执行动作、受控影响边界、失败可回退”这 5 类接口信息；没有这些，研究就还没有真正接上系统层。**

#### 进入运行时假设层后，如何做“结果面验证”和“回退判定”

进入运行时假设层之后，下一步不能只看回放里“整体结果有没有更好”。  
真正关键的是：**它到底改变了它声称要改变的那一块结果面没有。**

如果一个候选声称自己更像：

- **准入（admission）/ 过滤门（gate）**，那它应主要改变的是 **被阻断（blocked）** 和部分 **临界未入选（near-miss）** 的边界；  
- **排序分（score）**，那它应主要改变的是 **已选（selected）内部排序质量**；  
- **放行修正（relief）**，那它应主要改变的是本来被错杀的 **临界未入选（near-miss）→ 已选（selected）**；  
- **阈值（threshold）/ 阻断（blocker）**，那它应主要改变的是**过宽 / 过松边界下的误放行样本**。

因此，运行时假设层之后的验证，建议一律固定成下面 3 步：

1. **先写清“预期结果面”**  
   - 这条假设主要应该改变已选（selected）、临界未入选（near-miss）还是被阻断（blocked）；  
   - 不写清就不进入回放验证。
2. **再核对“实际变化面”**  
   - 回放后到底是哪个结果面真的变了；  
   - 如果变的是别的面，就要警惕语义放错位。
3. **最后再判“变化是否值得保留”**  
   - 变化是不是主要提升了 A 层 executable；  
   - 有没有顺便抬高 B / D 层噪声；  
   - raw / executable 裂口是否扩大。

可以把这件事压成下面这张最小结果面验证表：

| 运行时假设 | 预期该改变什么 | 真正要盯住什么 | 如果没发生，优先怎么判 |
| --- | --- | --- | --- |
| **准入（admission）** | 更早挡掉差样本，减少错误入池 | 被阻断（blocked）是否更干净、A 层占比是否提升、D 层是否下降 | 退回候选层，重新检查是不是更像过滤门（gate） |
| **过滤门（gate）** | 在主线后再筛一层，减少假阳性 | 临界未入选（near-miss）/ 被阻断（blocked）边界是否更清楚、B / D 是否下降 | 退回语义判断，检查是否被误写成准入（admission）或排序分（score） |
| **排序分（score）** | 不一定改入池数量，但会改已选（selected）内部质量 | 已选（selected）内部快达标、平滑样本、A 层内部质量是否提升 | 退回候选层，检查是否只是“看起来会排序” |
| **放行修正（relief）** | 救回原本被误杀的强样本 | 临界未入选（near-miss）→ 已选（selected）是否主要发生在高质量边界样本 | 若放宽过广，直接退回并缩回放行修正（relief）触发条件 |
| **阈值（threshold）/ 阻断（blocker）** | 调整边界过宽或过松问题 | 误放行样本是否减少、边界是否更稳定 | 退回影响面排查，不要直接保留假设 |

为了不让结果面验证再次失控，建议再固定 4 条纪律：

1. **禁止只看总命中率，不拆结果面**  
   - 总体变好，但已选（selected）/ 临界未入选（near-miss）/ 被阻断（blocked）哪一面在变，必须拆开看。
2. **禁止结果面变了，但语义位不跟着重判**  
   - 如果一个“排序分（score）假设”实际主要改变的是被阻断（blocked），说明它可能更像过滤门（gate）/ 准入（admission）。
3. **禁止结果面改善靠放宽边界硬换出来**  
   - 如果 selected 变多，但 D 层和 raw / executable 裂口一起扩大，这不叫真改善。
4. **禁止没有回退分支的结果面验证**  
   - 每条运行时假设都必须预写：验证失败时退回候选层、交互候选层，还是直接降级语义。

可以直接记住下面这条最短回放判断法：

- **不是看回放后“整体更好没有”，而是看它声称要改变的那一面，是否真的按预期变好了，而且没有把别的面一起搞脏。**

建议把回退动作也固定成下面 4 类，不要临场讨论：

1. **语义回退**  
   - 结果面变了，但不是原来声称的那一面；  
   - 例如原本写成排序分（score），结果更像过滤门（gate）。
2. **层级回退**  
   - 结果面变化不稳定，或影响面开始外溢；  
   - 从运行时假设层退回候选层 / 交互候选层。
3. **边界回退**  
   - 放行修正（relief）/ 阈值（threshold）假设放宽得过头；  
   - 先缩触发条件，再决定是否继续。
4. **直接终止**  
   - 结果面改善主要靠 B / D 扩张、coverage 异常收缩或裂口扩大换来；  
   - 这类假设应直接停掉，不再继续保留。

一句话说，**运行时假设层之后最重要的不是继续讲语义，而是把语义翻译成结果面验证：它宣称改变哪一面，就必须在那一面上被验证；如果改变错面、放大脏面、或没有明确回退路径，就不该继续往前推。**

#### 一页版运行时假设验证卡模板

如果前面的 runtime 语义、结果面验证、回退规则只停在说明层，真正执行时还是容易各写各的。  
因此建议每个进入运行时假设层的候选，都固定出一张**一页版运行时假设验证卡**。

推荐至少固定包含下面 7 个模块：

| 模块 | 固定内容 | 作用 |
| --- | --- | --- |
| **候选身份** | 候选名、当前层级、对应原型、对应 P0 / P1 / P2 关系 | 先把研究对象钉死，避免换口径 |
| **主语义位** | 准入（admission）/ 过滤门（gate）/ 排序分（score）/ 阈值（threshold）/ 放行修正（relief）/ 阻断（blocker） | 强制只写一个主语义位 |
| **预期结果面** | 主要应改变已选（selected）/ 临界未入选（near-miss）/ 被阻断（blocked）哪一面 | 防止只讲抽象 uplift |
| **实际变化面** | 回放后到底是哪一面真的变了 | 检查语义是否放对位 |
| **质量与执行** | A 层、B/C/D、raw / executable 裂口、coverage、影响面 | 防止结果变好只是纸面或窄样本幻觉 |
| **回退规则** | 若失败，退回候选层 / 交互候选层 / 降级语义 / 终止 | 防止假设只进不退 |
| **一句话结论** | 保留假设 / 改写语义 / 退回上一层 / 终止 | 让会议里能直接消费 |

可以直接套下面这版最简骨架：

- 回放后实际变化最多的是：
- 是否与预期一致：

为了让这张卡真正起作用，建议再固定 3 条纪律：

1. **必须同时写“预期结果面”和“实际变化面”**  
   - 少写任何一个，这张卡都不合格。
2. **必须写“明确不像什么”**  
   - 这样能减少准入（admission）/ 过滤门（gate）/ 排序分（score）/ 放行修正（relief）的语义漂移。
3. **必须给出可执行的回退动作，不接受“再看看”**  
   - runtime 假设验证卡的核心，不是记录观点，而是固定下一步。

也就是说，这张卡最重要的价值，不是让 runtime 讨论更详细，而是让它更可比：

- alpha 看它有没有把主线统计翻译成清楚的结果面；  
- beta 看它有没有把执行意义和裂口风险写清；  
- gamma 看它有没有把语义位、影响边界和回退动作收口。

一句话说，**一页版运行时假设验证卡的作用，是把“抽象运行时讨论”压成固定接口件：先写主语义位，再写预期结果面，再写实际变化面，最后写回退动作；这样研究层和系统层之间才不会断档。**

#### 每轮标签审查清单

alpha / beta / gamma 在开始首轮因子扫描前，至少要先逐项确认下面 7 条：

1. 主标签是否固定为 `future_5d_high_ge_15pct`。
2. 主研究买点代理是否固定。
3. raw 与 executable 双账本是否同时保留。
4. executable 排除规则是否已提前写明。
5. 次标签集合与阈值是否已冻结。
6. 每个样本是否能同时追溯到事件原型、买点代理和排除原因。
7. 本轮统计汇总是否会同时展示 raw 成功率与 executable 成功率。

如果这 7 条有任意一条答不上来，就说明首轮还没准备好进入因子扫描。

---

### 3.3 研究样本单位：不是全市场日线，而是“事件切片”

推荐的研究单位是：

`样本 = 股票 × 日期 × 事件原型 × 市场状态`

而不是简单的 `股票 × 日期`。

核心原因：

1. 同一只股票在不同结构阶段，因子意义完全不同。
2. “突破日”的放量和“衰竭日”的放量，不应被当成同一个信号。
3. 同一因子只有在特定事件原型里才可能显著有效。

因此，先把候选样本拆成几类事件宇宙：

1. **突破启动型**：刚形成结构突破，`breakout_freshness` 高。
2. **趋势延续型**：已经有上升结构，但仍处于加速段，`trend_acceleration` 与强收盘共振。
3. **蓄势释放型**：前期压缩后放量释放，关注量价压缩-扩张切换。
4. **催化共振型**：题材/事件/板块共振驱动的次日接力型。
5. **供给释放型**：上方供给消化后重新获得空间，关注 `overhead_supply` 与 `extension_without_room` 的修复。

这一步的本质，是先回答：

**哪些股票-日期，值得进入“可能 5 日内跑出 15%”的研究池。**

---

### 3.4 因子不是单点搜索，而是按“因子家族”挖掘

alpha 的因子研究不应直接从一万列特征开始暴力穷举（brute force），而应先定义因子家族，再在每个家族里做细化。

推荐优先研究 8 个家族：

1. **启动新鲜度家族**
   - breakout freshness
   - first acceleration day
   - breakout after compression
2. **趋势延续家族**
   - trend acceleration
   - trend continuation
   - multi-day relative strength persistence
3. **量能质量家族**
   - volume expansion quality
   - abnormal turnover quality
   - intraday demand persistence
4. **收盘承接家族**
   - close strength
   - tail strength
   - close near session high under volume confirmation
5. **催化与共振家族**
   - catalyst freshness
   - sector resonance
   - thematic breadth support
6. **拥挤与延伸约束家族**
   - stale trend repair
   - extension without room
   - overhead supply
7. **行为切换家族**
   - reversal corrected into continuation
   - false breakout recovery
   - shakeout absorption
8. **可交易性与微观结构家族**
   - entry liquidity quality
   - gap risk
   - open auction stability

这样做的好处是：

1. 容易解释。
2. 容易排除冗余。
3. 容易把结果沉淀为运行时因子，而不是黑箱权重。

---

### 3.5 真正要挖的是“条件有效因子”和“交互因子”

很多 BTST 强因子不是单因子，而是条件交互：

1. `breakout_freshness` 高 **且** `volume_expansion_quality` 高
2. `trend_acceleration` 高 **且** `close_strength` 高
3. `catalyst_freshness` 中高 **且** `sector_resonance` 高
4. `trend_continuation` 高 **且** `extension_without_room_penalty` 低

所以方法上必须显式区分三类对象：

1. **基础因子**：单变量预测力。
2. **条件因子**：某因子只在某个市场状态（regime）/ 事件切片（event）/ 流动性分桶（liquidity bucket）中有效。
3. **交互因子**：两个或多个因子同时出现时 uplift 明显放大。

推荐的挖掘顺序：

1. 先做单因子分层和单调性检查。
2. 再做条件切片。
3. 最后才做二阶或三阶交互搜索。

不要一开始就做大规模高阶组合搜索，否则过拟合风险极高。

---

### 3.6 alpha 的统计挖掘框架

alpha 负责三件事：标签、统计稳健性、文档。

#### 第一步：做事件宇宙内的单因子预筛

预筛不看单次收益峰值，而看：

1. 命中率 uplift
2. 分位数单调性
3. rank IC / 点二列相关（point-biserial）
4. 样本覆盖度
5. 正负窗口占比

#### 第二步：做多窗口稳健性筛选

每个候选因子至少要过：

1. 滚动时间窗验证
2. 市场状态切片验证
3. 不同流动性 bucket 验证
4. 行业/题材分布漂移检查
5. 事件原型迁移检查

#### 第三步：做反过拟合治理

对每个候选因子记录：

1. win-rate CI 宽度
2. uplift 方差
3. factor drift
4. parameter drift
5. redundancy / correlation cluster
6. extreme-sample dependence

如果一个因子只靠少数极端样本撑起来，就算短期表现好，也不进入下一轮。

---

### 3.7 beta 的执行约束框架

beta 的职责不是“研究完再想怎么下单”，而是从方法阶段就提前剔除纸面因子。

beta 需要把以下约束前置进方法：

1. **可成交性**：一字板、极端跳空、开盘即无流动性的样本不能被算成可复制成功。
2. **滑点敏感性**：若因子只在极窄价差条件下成立，应降级处理。
3. **成交承接**：次日若无法形成足够容量，因子不能直接进入主交易层。
4. **拥挤成本**：若某类形态在高热题材时容易被高开透支，需要单独做执行折减（execution haircut）。

beta 需要给 alpha 的不是抽象提醒，而是研究输入约束：

1. 统一买点代理（entry proxy）
2. 统一流动性底线（liquidity floor）
3. 统一滑点假设（slippage assumption）
4. 统一不可成交排除规则（impossible-fill exclusion rule）

这样挖出来的因子，才不是“视觉上很好看”却无法执行的假机会。

---

### 3.8 gamma 的风险与 rollout 治理框架

gamma 负责两层问题：

1. 这个因子是否真的稳健。
2. 即便稳健，是否适合进入 runtime。

gamma 的治理门如下：

1. **市场状态门（regime gate）**：牛市、震荡市、风险释放期是否表现一致。
2. **时间窗门（window gate）**：不同时间窗是否连续存在正 uplift，而不是只在单个窗口亮眼。
3. **覆盖度门（coverage gate）**：样本太少的因子只能留在观察层，不能直接升主线。
4. **尾部风险门（tail-risk gate）**：虽然命中率高，但若伴随极端回撤或跳空风险（gap risk），不进入正式交易层。
5. **激活门（activation gate）**：候选因子若不能实际改变 selected / near_miss / blocked 结果面，说明只是离线好看。

gamma 最重要的一条纪律是：

**离线有效，不等于 runtime 可发布。**

### 3.8.1 短线高胜率高盈亏比因子，如何映射到准入 / 过滤门 / 排序分 / 放行修正

研究里最常见的落地错误，不是“因子没效果”，而是**因子明明有用，却被放错了 runtime 位置**。  
同一个因子，如果该做过滤门（gate）却被硬塞进排序分（score），或者该做放行修正（relief）却被拿去当准入（admission），很容易把短线高胜率高盈亏比结构用坏。

因此，方法上不只要问“这个因子有没有 uplift”，还要问：

**它更适合决定谁能进池、谁该被挡掉、谁该排得更靠前，还是谁在特殊条件下可以被放松？**

建议先把四类 runtime 语义分清：

| 运行时语义 | 更适合承载什么样的因子 | 典型研究信号 | 最不该混淆成什么 |
| --- | --- | --- | --- |
| **准入（admission）** | 高胜率、稳定、覆盖较广、能先把差样本挡在外面的主线因子 | A 层 executable 样本占比稳定提升，D 层不扩张 | 不要误当成只负责排序的排序分（score） |
| **过滤门（gate）** | 确认型、过滤型、保护型因子 | 能明显减少 B / D 层、降低假阳性、提升可复制性 | 不要误当成主收益贡献因子 |
| **排序分（score）** | 在已可入池样本里继续拉开优先级的因子 | 不一定显著改变入池率，但能改善 A 层内部排序或快达标质量 | 不要误当准入（admission）硬门 |
| **放行修正（relief）** | 在特殊结构下放松过严限制、挽回被错杀强样本的因子 | 只在明确条件下改善 near-miss / false negative，不应大面积放宽 | 不要误当普适加分因子 |

可以把它再压成一个更直观的判断法：

1. **它主要是在挡坏样本，还是在挑好样本？**  
   - 挡坏样本，更像准入（admission）/ 过滤门（gate）。  
   - 挑好样本，更像排序分（score）。
2. **它主要是在多数样本里稳定起作用，还是只在少数特定条件下救回被错杀样本？**  
   - 多数样本稳定起作用，更像准入（admission）/ 过滤门（gate）/ 排序分（score）。  
   - 只在少数条件下救回错杀样本，更像放行修正（relief）。
3. **它是扩大主线，还是修正边界？**  
   - 扩大主线，更像准入（admission）/ 排序分（score）。  
   - 修正边界，更像过滤门（gate）/ 放行修正（relief）。

围绕短线高胜率高盈亏比主目标，建议优先按下面规则映射：

#### 1. 更偏“先守胜率”的因子，优先看准入 / 过滤门

如果一个因子主要特点是：

1. executable A 层占比提升更明显；
2. B / D 层样本明显减少；
3. 覆盖度尚可，跨窗不乱翻；

那它更像 **准入（admission）** 或 **过滤门（gate）**，而不是排序分（score）。

两者区别是：

- **准入（admission）**：更像“进不进池”的前置门，负责先把大方向做对。  
- **过滤门（gate）**：更像“满足主线后，还要不要再过一层确认”，负责减少假突破、假延续、假放量。

#### 2. 更偏“在主线内争夺更高赔率”的因子，优先看排序分

如果一个因子：

1. 不一定明显减少入池总数；
2. 但能在已较优的样本里，把快达标 / 平滑达标样本排得更靠前；
3. 主要改善的是 A 层内部质量，而不是简单扩大样本覆盖；

那它更像 **排序分（score）**。

这类因子适合做 frontier 排序，而不是准入（admission）硬门。  
否则很容易把“应该用来排序”的信息，误用成“进不进池”的信息。

#### 3. 更偏“修复过严过滤副作用”的因子，优先看放行修正

如果一个因子主要作用是：

1. 识别原本会被准入（admission）/ 过滤门（gate）挡掉、但实际上仍可能跑出高质量 A 层结果的样本；
2. 这种改善只在清楚条件下成立，而不是普遍成立；
3. 放宽后不会明显扩大 D 层噪声；

那它更像 **放行修正（relief）**。

放行修正（relief）的关键不是“让更多票进来”，而是：

**只在明确、可解释、可复核的边界条件下，少量放行原本会被误杀的强样本。**

#### 4. 什么时候不该进入任何 runtime 语义

如果一个因子出现下面任意一种情况，方法上都不该急着讨论准入 / 过滤门 / 排序分 / 放行修正：

1. 主要优势只停留在 raw 账本；
2. A 层提升不明显，却主要抬高 B 层或 D 层；
3. 只能靠单窗、单批样本、单次题材行情成立；
4. 根本说不清它是在“挡坏样本”还是“排好样本”；

这时更合理的动作，是先把它留在观察层或退回研究层，而不是急着塞进 runtime。

可以直接记住下面这条方法总线：

- **先把坏样本挡出去** → 准入（admission）/ 过滤门（gate）。  
- **再把好样本排前面** → 排序分（score）。  
- **最后只在明确边界下救回被误杀样本** → 放行修正（relief）。  

一句话说，**短线高胜率高盈亏比因子的最佳方法，不只是挖出“有效因子”，而是尽早识别它到底更适合做准入（admission）、过滤门（gate）、排序分（score）还是放行修正（relief）；放对语义位，往往和挖出因子本身一样重要**。

#### 3.8.2 准入 / 过滤门 / 排序分 / 放行修正四类语义，各自最容易出现的误判清单

真正把方法落到 runtime 时，最危险的常常不是“完全没因子”，而是**有因子，但放错位置**。  
一旦放错位置，短线高胜率高盈亏比结构往往不是慢慢变差，而是会出现非常典型的“好像更严格 / 更聪明，实际上 A 层没上去”的假优化。

可以固定按下面这张表排查：

| 被放错的语义位 | 最常见误判 | 典型症状 | 正确纠偏动作 |
| --- | --- | --- | --- |
| **准入（admission）被当成排序分（score）** | 本来应该先决定“进不进池”的主线过滤条件，被降成轻量排序项 | 候选池变宽，但 A 层占比不升，D 层和 near-miss 一起增多 | 把该因子前移回准入（admission），先恢复主线样本质量，再看是否仍需细排 |
| **过滤门（gate）被当成准入（admission）** | 本来只是确认 / 防假阳性的二次检查，被误升级成第一道硬门 | coverage 急剧收缩，很多原本可复制的 A 层样本在首门就被挡掉 | 退回过滤门（gate）语义，只让它在主线入池后做确认，不要抢准入（admission）的位置 |
| **排序分（score）被当成准入 / 过滤门** | 本来应该在优质样本内部排序的信息，被硬做成一票否决阈值 | 排名前几名质量不错，但整体样本显著变少，A 层总量不升反降 | 去掉硬阈值，改回排序权重或 frontier 优先级，而不是硬门 |
| **放行修正（relief）被当成普适排序分（score）** | 本来只该在少数边界情形下放松限制，却被扩成广泛加分项 | 被“救回”的样本明显超出少量例外，B / D 层同步抬高 | 把放行修正（relief）限回明确触发条件，要求可解释、可复核、可回放 |
| **放行修正（relief）被当成准入（admission）** | “例外放行”被错误升级成“默认准入” | 候选池突然变宽，但 raw / executable 差值扩大，纸面优势变多 | 放行修正（relief）只保留为边界修正，不得替代准入（admission）主线条件 |
| **过滤门（gate）被当成排序分（score）** | 本来是保护性过滤，被当成轻量加减分 | 看似排序更细，但假突破 / 假延续样本并未真正减少 | 改回 hard gate 或 semi-hard gate，让它真正承担挡坏样本责任 |

如果想更快判断，可以直接用下面 4 条最短排错法：

1. **候选池明显变宽，但 A 层不升**  
   - 更像 admission 失守，或 relief 被滥用。
2. **coverage 明显收缩，但 A 层总量也一起掉**  
   - 更像把 gate / score 误升级成 admission。
3. **排序前列看起来更漂亮，但整体 executable 质量没改善**  
   - 更像 score 被拿去假装“硬门”或只做了表层美化。
4. **raw 账本变好，executable 账本没同步改善**  
   - 更像 relief 或 score 被用来遮盖执行口径问题，而不是解决主线问题。

围绕 5 日 / 55% / 15% 主目标，可以再加一条纪律：

- **凡是会扩大 B 层、D 层，却说不清为什么还要放过去的 runtime 变更，都先视作误判。**

也就是说，runtime 语义纠偏时，不要先看“这个因子名义上像什么”，而要先看：

1. 它到底是在**挡坏样本**、**排好样本**，还是**救回被误杀样本**；  
2. 它把改善带到了 **A 层**，还是把噪声带到了 **B / D 层**；  
3. 它改善的是 **executable 结果**，还是只改善了 **raw 表象**。

一句话说，**admission / gate / score / relief 的误判，本质上不是命名错误，而是把“主线筛选”“二次确认”“内部排序”“边界放行”四种不同职责混成一件事；一旦混了，短线高胜率高盈亏比结构通常会先体现在 A 层停滞、B / D 层扩张、raw / executable 裂口变大上**。

### 3.9 推荐的完整挖掘流水线

这是本次最核心的方法结论。

#### 第 1 步：定义可执行标签

统一 `T+1` 买点、`T+1 ~ T+5` 观察窗、主次标签和不可成交剔除规则。

#### 第 2 步：构建事件原型宇宙

先把样本限制在突破启动、趋势延续、蓄势释放、催化共振、供给释放等结构区间。

#### 第 3 步：在每个事件宇宙内生成因子家族

按启动、趋势、量能、承接、催化、拥挤、行为切换、微观结构八大类生成候选因子。

#### 第 4 步：做单因子预筛

保留具备：

1. 命中率 uplift
2. 分位数单调性
3. 足够 coverage
4. 跨窗口不过度漂移

的候选因子。

#### 第 5 步：做条件切片与交互挖掘

只对通过单因子筛选的候选做二阶组合和 regime 条件搜索。

#### 第 6 步：做 beta 执行约束回灌

把不可成交、高滑点、低容量样本从统计结果中剔除或降权。

#### 第 7 步：做 gamma rollout 审核

检查：

1. 是否稳定提升 5 日 15% 命中率
2. 是否保留合理 payoff
3. 是否在多个窗口继续有效
4. 是否真的能改变 runtime 结果面

#### 第 8 步：只把通过治理门的因子写进 runtime 候选池

通过的因子，再进一步讨论是进入：

1. score 权重
2. threshold / rank cap 调整
3. relief / blocker
4. watchlist 观察项

而不是一律直接加权进总分。

### 3.10 这套方法最容易挖出的，不是“万能因子”，而是“高质量局部因子”

这个认知必须提前统一。

目标如此激进时，更现实的结果通常不是挖出一个全市场通杀因子，而是挖出：

1. 在**突破启动型**事件里很强的因子
2. 在**趋势延续型**事件里很强的因子
3. 在某些**regime** 里显著有效的条件因子
4. 能明显降低假突破和低质量接力的负向约束因子

所以最终产物更可能是：

1. 一个事件模板
2. 若干正向因子
3. 若干负向约束因子
4. 若干交互规则
5. 一套 rollout 治理门

而不是“找到一个指标，直接解决所有问题”。

## 4. 未来实验如何判断“这个方法真的有用”

方法是否有效，不能只看是否产出更多因子，而要看它是否更高效地挖出**可升级候选**。

推荐用 5 个结果指标衡量：

1. 候选因子进入下一轮验证池的比例是否提高。
2. 通过多窗口稳健性筛选的比例是否提高。
3. 通过 beta 可交易性审查的比例是否提高。
4. 通过 gamma rollout gate 的比例是否提高。
5. 最终进入 runtime 后，是否提升：
   - `future_5d_high_ge_15pct` 命中率
   - next-high / next-close payoff
   - execution-ready 结果面质量

如果一套方法只是更容易“挖出故事”，而不是更容易“挖出可升级候选”，那它就不是好方法。

如果第 `4` 节回答的是“这套方法有没有效”，那么下面第 `5` 节开始回答的就是“这套方法由谁来执行、各自交什么”。

## 9. 把“下一阶段建议”收紧成首轮研究蓝图

上面的 5 条建议，还偏方法论。为了真正进入研究阶段，下面把首轮范围进一步收紧。

### 9.1 首轮目标

首轮目标不是“直接找到最终上线因子”，而是验证这套方法是否真的能稳定产出**候选强因子**。

首轮只回答 4 个问题：

1. 事件驱动样本切片，是否明显优于全市场直接扫标签。
2. 趋势延续 / 突破启动 / 量能质量三大家族，是否能挖出至少一批有稳健 uplift 的候选。
3. beta 的可交易性过滤，是否会显著改变候选因子的排序。
4. gamma 的 rollout gate，是否能把“离线好看”和“可能进入 runtime”有效区分开。

如果这 4 个问题里有 2 个以上回答不成立，说明方法本身还要继续收缩或修正，不进入更大范围扩展。

### 9.2 首轮只研究 3 个事件原型

为了避免研究面一开始就失控，首轮只保留下面 3 个事件原型：

| 事件原型 | 为什么优先 | 首轮重点看什么 |
| --- | --- | --- |
| 趋势延续型 | 最符合 BTST “今天强、明天还能强”的主语义 | 趋势是否仍在加速，是否具备次日承接 |
| 突破启动型 | 最容易与 5 日 15% 目标形成高赔率映射 | 突破是否新鲜、是否刚脱离拥挤区 |
| 蓄势后放量释放型 | 容易补足“不是追高，而是刚开始释放”的样本 | 量价压缩后是否出现高质量扩张 |

本轮明确不先铺开的事件原型：

1. 纯催化共振型
2. 复杂供给释放型
3. 过多依赖题材叙事的行为型模板

原因不是它们没价值，而是首轮更需要验证方法本身，而不是一开始就扩大故事面。

#### 趋势延续型：进入 / 退出 / 排除规则

**进入条件**

1. 样本已经具有明确上升结构，不是纯粹刚起步的一日脉冲。
2. `trend_acceleration` 明显不弱，且 `close_strength` 没有显示出明显承接缺失。
3. `extension_without_room_penalty` 与 `stale_trend_repair_penalty` 不能主导解释。

**退出条件**

1. 趋势加速明显衰减，开始更像高位钝化而不是继续强化。
2. 收盘承接不再支持“次日仍有弹性”。
3. 空间约束开始主导，样本从延续段滑向过度延伸段。

**排除条件**

1. 实际更像旧趋势修复、补涨尾声或均值回归反抽。
2. 上方供给与拥挤约束明显高于趋势延续证据。
3. `T+1` 执行条件差到即便判断正确也难复制。

**首轮要回答的核心问题**

它到底是“继续走的趋势延续”，还是“看上去还强、其实已经接近尾声”的伪延续。

#### 突破启动型：进入 / 退出 / 排除规则

**进入条件**

1. `breakout_freshness` 足够高，说明它更像刚形成突破，而不是突破后多日钝化。
2. `volume_expansion_quality` 或 `close_strength` 至少有一项能支持“不是空心突破”。
3. `sector_resonance`、`catalyst_freshness` 不能同时明显偏弱。

**退出条件**

1. 突破的新鲜度快速衰减，样本不再属于“刚启动”。
2. 后续更多体现为高开后的消化或横盘，而不是继续扩张。
3. 短时间内已经转成“过度延伸”语义，不再适合用启动型框架解释。

**排除条件**

1. 主要上涨只来自跳空，而不是来自真实放量与承接。
2. `volume_expansion_quality` 过弱，属于典型假突破风险。
3. `stale_trend_repair_penalty` 反而更强，说明这不是真启动，而是旧趋势再包装。

**首轮要回答的核心问题**

它到底是“刚启动的有效突破”，还是“高开看起来像突破、其实没有真实跟随”的假启动。

#### 蓄势后放量释放型：进入 / 退出 / 排除规则

**进入条件**

1. 样本此前存在可识别的压缩、收敛或蓄势阶段，而不是连续多天无序波动。
2. 释放日出现 `volume_expansion_quality` 改善，且最好伴随 `close_strength` 或尾盘承接增强。
3. 当前仍处于“刚释放、尚未充分定价”的阶段，而不是释放完成后的后排追涨阶段。

**退出条件**

1. 释放阶段已经完成，后续不再体现新的量价扩张。
2. 延伸约束与上方供给开始主导解释。
3. 次日更像高位追价，而不是刚从压缩转入扩张。

**排除条件**

1. 没有真实量能改善，只是价格脉冲。
2. 流动性质量差，释放信号大概率不可复制。
3. 释放来自单日情绪扰动，而不是有结构的压缩-扩张切换。

**首轮要回答的核心问题**

它到底是“结构能量释放的开始”，还是“一天热度冲高后的噪声放大”。

#### 原型归类的优先级规则

首轮同一个 `trade_date:ticker` 只允许有一个**主原型**，避免同一样本在多个原型里重复记功。

推荐的归类顺序：

1. 如果“新鲜突破”证据最强，优先归入**突破启动型**。
2. 如果突破并不新鲜，但趋势强化更强，优先归入**趋势延续型**。
3. 如果最核心的变化是“压缩后放量释放”，优先归入**蓄势后放量释放型**。

如果一个样本必须同时占据两个原型才能讲得通，首轮先把它降为复杂样本，不进入主研究池。

### 9.3 首轮只研究 3 个因子家族

首轮与 3 个事件原型配套的因子家族，也只保留下列 3 组：

| 因子家族 | 主要作用 | 对应首轮原型 |
| --- | --- | --- |
| 趋势延续家族 | 识别“不是尾声，而是延续段”的强度 | 趋势延续型 |
| 启动新鲜度家族 | 识别“刚启动，不是旧趋势修复” | 突破启动型 |
| 量能质量家族 | 识别“不是假突破，而是有承接的释放” | 蓄势后放量释放型 |

首轮不把收盘承接、催化共振、微观结构家族独立拉成主线，而是作为辅助解释项嵌进这 3 个主家族中。

这样做的原因：

1. 主线更聚焦。
2. 更容易发现真正的一阶 separating factor。
3. 可以把复杂度控制在“先找强一阶，再加少量二阶”的范围内。

### 9.4 首轮允许的二阶交互只有 3 组

首轮不是不允许交互，而是只允许最有交易语义的少量交互：

1. `trend_continuation × volume_expansion_quality`
2. `breakout_freshness × close_strength`
3. `breakout_freshness × trend_acceleration`

为什么只保留这 3 组：

1. 它们都能直接解释成“次日还能不能走”的结构逻辑。
2. 它们与当前仓内已有 BTST 指标语义兼容，未来更容易映射到 runtime。
3. 它们还没有把搜索空间放大到失控。

首轮明确不做：

1. 三阶以上交互搜索
2. 无约束自动组合爆搜
3. 先用黑箱模型跑特征重要度、再回头解释

### 9.5 首轮候选因子优先级矩阵

首轮不应把所有 BTST 因子一股脑并列扫描，而应分成**主因子、确认因子、约束因子**三层。

#### 第一层：主因子，先找一阶 separating power

| 优先级 | 因子 | 对应原型 | 首轮角色 | 为什么先看 |
| --- | --- | --- | --- | --- |
| P0 | `breakout_freshness` | 突破启动型 | 主驱动因子 | 最直接回答“是不是刚启动，而不是旧趋势修复” |
| P0 | `trend_acceleration` | 趋势延续型 | 主驱动因子 | 最直接回答“趋势是否仍在强化，而不是只剩尾声” |
| P0 | `volume_expansion_quality` | 蓄势后放量释放型 | 主驱动因子 | 最直接回答“释放是否有承接，而不是假突破” |

首轮的默认纪律是：

1. 每个事件原型先只争夺一个最强 P0 因子。
2. 如果连 P0 因子都没有稳定 executable uplift，不要急着往高阶交互推进。

#### 第二层：确认因子，只负责提高解释力与筛假信号

| 优先级 | 因子 | 更适合搭配谁 | 首轮角色 | 为什么放第二层 |
| --- | --- | --- | --- | --- |
| P1 | `close_strength` | `breakout_freshness` / `trend_acceleration` | 确认因子 | 它单独不够，但能过滤“尾盘看起来强、实则承接不足”的样本 |
| P1 | `sector_resonance` | `breakout_freshness` / `trend_acceleration` | 确认因子 | 用来区分孤立强势与板块共振，不适合单独当主线 |
| P1 | `catalyst_freshness` | `breakout_freshness` | 确认因子 | 更适合解释“为什么今天的强还有信息增量”，不宜首轮单独抬成主因子 |

P1 的纪律是：

1. 先看它是否能提升 P0 的 executable 命中率。
2. 不因为 P1 本身好解释，就把它提前抬成首轮主驱动因子。

#### 第三层：约束因子，负责避免把假强票当成真机会

| 优先级 | 因子 | 首轮角色 | 主要拦截什么 |
| --- | --- | --- | --- |
| P2 | `stale_trend_repair_penalty` | 负向约束 | 旧趋势修复、补涨尾声、陈旧反抽 |
| P2 | `extension_without_room_penalty` | 负向约束 | 已经过度延伸、剩余空间不足 |
| P2 | `overhead_supply_penalty` | 负向约束 | 上方供给和套牢压制仍然明显 |

P2 的纪律是：

1. 它们首轮主要负责“别放错票”，不是负责主动挖主线。
2. 如果某个候选只能靠大幅放松 P2 才成立，首轮先记为高风险候选，不升主线。

#### 首轮暂不优先的因子

下面这些更适合放到首轮后半程或第二轮，而不是一开始就拉进主矩阵：

1. `layer_c_alignment`：更像解释增强项，不是首轮主驱动。
2. `t0_tail_strength`：更适合在 P0/P1 稳住后，再做日内强化研究。
3. `volume_price_divergence_score`：更适合做结构补充，不适合作为首轮主轴。
4. `momentum_confirmation_score` 与 `rs_sector_rank`：更适合作为扩展确认层，而不是第一批分离因子（separating factor）。
5. 任何高度依赖单日盘口细节但当前样本覆盖不稳的候选。

### 9.6 候选因子的晋升逻辑

首轮不只要排优先级，还要定义因子如何晋升。

推荐分成 4 层：

1. **观察层因子**：raw 账本有结构提示，但 executable 证据不足。
2. **候选层因子**：在至少一个事件原型里，raw 与 executable 都有同方向 uplift。
3. **交互候选层**：与一个 P0 因子组合后，executable 质量继续改善，且解释不变形。
4. **运行时假设层**：已经能明确说明它更适合进入排序分、过滤门、阈值还是放行修正。

首轮默认只允许把因子从第 1 层推进到第 2 或第 3 层，不追求直接进入正式运行时发布讨论。

#### 哪些情况应直接降级

如果出现下面任意一种情况，候选因子应直接降级或暂停：

1. raw 提升明显，但 executable 提升消失。
2. 只在单一窗口或少数极端样本中成立。
3. 必须配合多个额外条件才勉强成立，解释成本过高。
4. 只能靠放松 P2 约束才能“看起来更好”。
5. 无法说清它未来要映射成哪种运行时语义。

到这里，第 `9` 节回答的是“首轮蓝图该怎么收紧”；下面第 `10` 节开始回答的就是“三个人具体怎么按这张蓝图执行”。

## 10. 首轮研究的 alpha / beta / gamma 执行蓝图

### 10.1 alpha：先产出“研究底稿”，不是先产出结论

alpha 首轮要先交付 4 份研究底稿：

1. **标签规范表**：主标签、次标签、排除规则、观测窗、买点代理价。
2. **事件原型定义表**：3 个事件原型分别进入、退出、排除什么样本。
3. **单因子候选板**：每个原型下，三大家族的基础候选因子清单。
4. **统计验收板**：每个候选因子的 uplift、单调性、coverage、跨窗表现、冗余簇。

alpha 在首轮最重要的纪律：

1. 先做一阶 separating factor，再做交互。
2. 先看多窗口稳定性，再看最亮眼单窗。
3. 不因为某个因子“讲得通”就让它进入下一轮。

### 10.2 beta：先产出“可执行排除层”，不是最后补风控说明

beta 首轮要先交付 4 个执行约束块：

1. **买点代理说明（entry proxy）**：`T+1` 使用什么可执行买点代理。
2. **不可成交排除表（impossible-fill）**：什么样本虽然命中高点，但默认不记为可复制成功。
3. **流动性 / 滑点降权规则（liquidity / slippage）**：哪些样本不删除，但要在统计上降权。
4. **可执行解释模板（execution-ready）**：未来如果某因子进入运行时，beta 如何解释它的执行意义。

beta 首轮最重要的纪律：

1. 不让不可成交样本抬高因子排名。
2. 不让极端跳空样本主导 uplift 结论。
3. 不把“高点出现过”误当成“可复制吃到过”。

### 10.3 gamma：先产出“治理门”，不是最后做通过/不通过表态

gamma 首轮要先交付 4 个治理块：

1. **window 切分方案**：哪些窗口视为独立证据，不允许相互污染。
2. **regime 切分方案**：至少区分趋势、震荡、风险释放环境。
3. **候选晋升门（candidate promotion gate）**：什么样的候选只能进观察层，什么样的候选才值得进运行时候选池。
4. **激活假设（activation hypothesis）**：候选因子未来更可能映射成排序分、过滤门、阈值还是放行修正。

gamma 首轮最重要的纪律：

1. 不让单一窗口决定生死。
2. 不让样本过稀的因子直接升级。
3. 不让“离线优胜”自动等于“runtime 可发布”。

### 10.4 alpha / beta / gamma 的首轮交付模板

上面的分工如果只停在“负责什么”，执行时仍然容易发散。首轮建议直接使用下面 3 套交付模板。

#### alpha 交付模板：因子研究卡

alpha 每次交付，建议固定写成下面 7 段：

1. **研究对象**：本次只研究哪个事件原型、哪一层因子（P0 / P1 / P2）。
2. **主假设**：这个因子为什么可能提高 `future_5d_high_ge_15pct` 命中率。
3. **标签口径**：使用的主标签、次标签、raw / executable 版本。
4. **统计结果**：至少写 raw / executable uplift、coverage、单调性、跨窗稳定性。
5. **冗余判断**：它和现有因子是否高度重复，还是提供了新的结构信息。
6. **候选结论**：观察层、候选层、交互候选层，三选一。
7. **下一步建议**：继续单因子深挖、转交 beta 审查、还是直接暂停。

alpha 模板最重要的纪律：

1. 不写空泛故事，必须把“因子为什么成立”绑定到具体事件原型。
2. 不只报 raw 结果，必须同时报 executable 结果。
3. 不因为解释顺耳就跳过冗余判断。

#### beta 交付模板：执行可复制性卡

beta 每次交付，建议固定写成下面 7 段：

1. **买点代理**：这轮默认采用哪一种 `T+1` entry proxy。
2. **排除规则**：哪些样本被判为 impossible-fill 或明显不可复制。
3. **降权规则**：哪些样本虽然保留，但会因流动性、滑点、拥挤被降权。
4. **执行偏差**：raw 与 executable 结果差异最大的样本簇是什么。
5. **容量风险**：这个因子若放大，会不会明显冲击成交与滑点。
6. **执行结论**：可直接进入候选、仅保留观察、或应直接降级。
7. **下一步建议**：是补执行约束、改 entry proxy，还是把问题退回 alpha / gamma。

beta 模板最重要的纪律：

1. 不只说“可能不好成交”，要说明是哪类样本不可复制。
2. 不把执行问题包装成统计问题。
3. 不让少数极端样本主导执行结论。

#### gamma 交付模板：治理晋升卡

gamma 每次交付，建议固定写成下面 7 段：

1. **证据范围**：用了哪些窗口、哪些 regime、哪些原型样本。
2. **稳定性判断**：结果是多窗口同向，还是高度依赖单窗。
3. **风险提示**：样本稀疏、尾部风险、回撤或结构漂移是否明显。
4. **晋升层级**：观察层、候选层、交互候选层、runtime 假设层。
5. **runtime 映射假设**：更适合进 score、gate、threshold 还是 relief。
6. **治理结论**：继续放大、受控保留、或暂停推进。
7. **下一步建议**：继续补证据、切换原型，还是结束这条线。

gamma 模板最重要的纪律：

1. 不直接给“能不能上线”的结论，先给“现在属于哪一层”。
2. 不混淆样本不足和因子失效。
3. 不把当前窗口的漂亮结果包装成普遍规律。

#### 三个人共同遵守的统一收口格式

为了方便后续复盘和交接，alpha / beta / gamma 三份卡片最后都统一收口成 4 句：

1. 这轮只研究了什么。
2. 这轮最强证据是什么。
3. 这轮最大风险是什么。
4. 下一轮只做什么，不做什么。

如果三个人的交付最后都不能被压缩成这 4 句，说明交付仍然不够聚焦。

### 10.5 gamma 的首轮 rollout gate

首轮不需要把因子一路推到正式发布，但必须先把“是否继续推进”这件事做成统一的门。

推荐按下面 6 道 gate 依次判断：

| Gate | 先问什么 | 通过后去哪 | 不通过怎么处理 |
| --- | --- | --- | --- |
| G0 范围锁定 | 这轮是否只研究一个原型、一个主问题、少量候选 | 进入 G1 | 直接退回重定义范围 |
| G1 输入契约 | 标签、entry proxy、排除规则、样本追溯是否已固定 | 进入 G2 | 不进入任何晋升讨论 |
| G2 证据稳定性 | 结果是否不是单窗偶然样本 | 进入 G3 | 只保留观察层 |
| G3 可执行性 | executable 账本下是否仍保留同方向证据 | 进入 G4 | 降级为结构提示因子 |
| G4 影响面 | 是否主要改变预期样本，而不是污染无关样本 | 进入 G5 | 暂停推进并回查机制 |
| G5 runtime 假设 | 能否说清更适合进 score、gate、threshold 还是 relief | 给出层级结论 | 只能留在离线研究层 |

#### G0：范围锁定门

gamma 首先不是看结果，而是看这轮研究有没有资格被评审。

至少要同时满足：

1. 只研究一个首轮事件原型，或一个清晰的主矛盾。
2. 不把 admission、score frontier、execution 三条线混在一起。
3. 候选因子数量仍处于首轮可解释范围内。

如果这一层不过，后面即使有漂亮结果，也先不进入 rollout 讨论。

#### G1：输入契约门

gamma 必须先确认：

1. 主标签、次标签、raw / executable 双账本已经冻结。
2. `T+1` 买点代理与 impossible-fill 规则已经固定。
3. 每个样本能追溯到事件原型、原始标签、执行标签和排除原因。

这一步的本质是防止“口径漂移伪造 uplift”。

#### G2：证据稳定性门

gamma 不直接看单窗最亮结果，而先看：

1. 是否跨多个窗口保持同方向。
2. 是否在不同 regime 下没有彻底翻向。
3. 是否样本覆盖足以支撑“继续研究”而不是“只是巧合”。

G2 通过，不代表可升级；只代表它值得继续被保留。

#### G3：可执行性门

这是首轮最重要的分水岭。

如果一个因子：

1. raw 账本很好看；
2. 但 executable 账本明显衰减；

那 gamma 应把它明确标为**结构提示因子**，而不是交易候选因子。

只有 raw / executable 同向，才允许继续往上走。

#### G4：影响面门

gamma 还必须确认它改变的是不是正确那批样本。

至少回答：

1. 是不是主要影响目标原型样本。
2. 有没有把大量原本稳定的无关样本也一起带偏。
3. 它的改善是低成本释放，还是靠大范围放宽才换来的。

如果影响面失控，这个因子就算统计上成立，也不值得推进。

#### G5：runtime 假设门

首轮的终点不是“发布”，而是形成清晰的 runtime 假设。

gamma 必须能明确写出：

1. 它更像正向 score 因子；
2. 还是确认型 gate；
3. 还是 threshold / rank cap 调整线索；
4. 还是 relief / blocker 候选。

如果连这一层都说不清，这个因子还不具备进入候选池的资格。

#### Gate 通过后的 4 类结论

首轮经过 G0-G5 后，只允许给出下面 4 类结论：

1. **暂停**：范围、契约或影响面不过关。
2. **观察层保留**：结构有意思，但 executable 或稳定性还不够。
3. **候选层保留**：可以进入下一轮继续补证据。
4. **交互候选层保留**：适合和某个 P0 因子做受控组合验证。

首轮默认不直接给“runtime ready”结论，除非后续窗口证据额外充分。

如果第 `10` 节回答的是“怎么执行和怎么治理”，那么下面第 `11` 节开始回答的就是“做到什么程度，才算这套方法在首轮真正过关”。

## 11. 首轮验收标准：什么叫“方法有效”

首轮不是考核最终收益，而是考核这套方法是否能稳定筛出**值得继续研究的候选**。

下面 5 条里，至少要满足大部分，才算首轮方法过关：

1. 在 3 个事件原型里，至少有 1 到 2 个原型能稳定产出候选，而不是全部沉默。
2. 在三大家族里，至少有若干候选因子在多窗口里保持同方向 uplift，而不是只靠单窗爆点。
3. beta 执行过滤后，候选排序仍大体稳定，说明它们不是被不可成交样本硬抬上去的。
4. gamma 能为候选因子给出明确的 runtime 映射假设，而不是只能停留在“离线统计有点好看”。
5. 首轮文档沉淀后，下一轮能明确知道：该继续放大哪个事件原型、砍掉哪个家族、保留哪组交互。

如果首轮结束后，仍然不能明确这 3 件事：

1. 哪个事件原型最值得继续放大；
2. 哪个因子家族最像真正主线；
3. 哪些样本属于明显不可复制成功；

那就说明文档仍不够好，方法也仍未真正收敛。

### 11.1 如果首轮不过关，优先怎么排查

首轮不过关时，最怕的不是结果差，而是**不知道该回查标签、原型、执行，还是治理**。

建议先按下面这张表排障：

| 看到的失败信号 | 第一优先回查 | 更像哪类问题 | 正确动作 |
| --- | --- | --- | --- |
| 3 个原型几乎都沉默 | 事件原型定义文档 | 原型范围错了，样本切片过窄或过偏 | 先改原型范围，不要急着扩因子库 |
| raw 有提升，但 executable 集体明显衰减 | 执行约束说明 | 买点代理、排除规则或流动性假设失真 | 先修 beta 口径，不要急着否定 alpha 因子 |
| 只有单窗好看，跨窗就翻向 | rollout gate + window 切分 | 稳健性不足，样本被局部行情误导 | 先降级为观察层，不急着升候选层 |
| P1 / P2 比 P0 更像主线 | 候选因子优先级表 | 主因子定义漂了，解释层和主驱动层混淆 | 先重排层级，不要继续做交互扩搜 |
| 三个人各说各话，结论压不成 4 句 | 首轮复盘文档 + 三方卡片 | 交付格式不统一，证据没有被压缩 | 先统一收口格式，不要直接开第二轮 |
| 看起来什么都做了，但下一步动作仍超过 2 个 | 首轮复盘文档 | 不是研究不够，而是收口失败 | 先删动作、定主线，不要继续加分析 |

排障顺序的默认纪律是：

1. **先查定义层**：标签、原型、层级是不是先漂了。
2. **再查执行层**：是不是 beta 口径把纸面优势洗掉了。
3. **最后查治理层**：是不是 gamma 把本该观察的东西误当成可推进候选。

### 11.2 排障之后，下一步先改什么

排障完成后，不要同时重写 5 个东西。默认回退顺序建议固定成下面 4 步：

1. **先改定义，不先扩搜索面**  
   - 如果怀疑原型或标签漂了，先改定义文档。  
   - 不要一边改定义，一边继续往因子库里加新候选。
2. **再改执行口径，不先否定整条主线**  
   - 如果 raw 还行、executable 明显衰减，先修 entry proxy、排除规则、降权规则。  
   - 不要直接把 alpha 全部因子一票否决。
3. **再改因子层级，不先做更高阶交互**  
   - 如果 P0 / P1 / P2 排位看起来不对，先重排主线层级。  
   - 不要拿更多交互去掩盖一阶主线没站稳的问题。
4. **最后才决定要不要缩样本或换主题**  
   - 只有当前三步都查完，仍然没有主原型和主因子，才考虑缩首轮范围或换下一轮主题。  
   - 这一步是最后手段，不是默认动作。

一句话说，**先修口径，再修执行，再修层级，最后才考虑改主题**。

### 11.3 首轮失败后，最不该做什么

很多首轮研究不是死在“因子不行”，而是死在**失败后改得太多、太快、太乱**。  
为避免把一次可诊断的失败，改成一次不可追溯的失败，建议明确禁止下面 6 类动作：

1. **禁止同时改标签、原型、执行口径**  
   - 一次只改一层，最多带动一处紧邻依赖。  
   - 如果三层一起改，下一轮即使变好，也不知道是哪一步起作用。
2. **禁止因为一轮失败就大扩因子库**  
   - 首轮不过关，优先怀疑切片和口径，不优先怀疑“候选还不够多”。  
   - 扩库太早，通常只会把解释复杂度抬高。
3. **禁止用更复杂交互掩盖一阶主线不稳**  
   - P0 没站稳时，不要急着上多因子交叉、条件门、二阶组合。  
   - 复杂度上升不等于分离度上升。
4. **禁止把 raw 失败和 executable 失败混在一起讲**  
   - raw 差，通常是 alpha 定义问题。  
   - raw 可以、executable 明显衰减，通常是 beta 口径问题。  
   - 这两类失败必须拆开写，不能合并成“整体没效果”。
5. **禁止 gamma 在证据没收口时提前推进主题切换**  
   - gamma 可以暂停推进，但不应在定义层、执行层还没查清时直接宣布“这一类方法不值得做”。  
   - 主题切换只能是最后动作，不是第一反应。
6. **禁止为了让结论好看而重写成功标准**  
   - `5 日内超过 15%`、`概率大于 55%` 这类主目标不能因为首轮压力而临时改口。  
   - 成功标准一旦漂移，整个方法文档就失去比较基线。

如果要把这一节压成一句纪律，就是：**失败后先保可追溯性，再求更高胜率；先减少乱改，再增加新想法**。

### 11.4 什么情况下，才允许重开首轮

并不是首轮一失败，就应该“再跑一轮看看”。  
所谓重开首轮，不是换一轮运气，而是**带着一个更清楚的修订假设，重新验证同一条研究主线**。

建议只有同时满足下面 5 条时，才允许正式重开首轮：

1. **已经定位到主问题层**  
   - 至少已经知道，这次主要是在定义层、执行层，还是层级/治理层出问题。  
   - 如果连主问题层都说不清，说明还在排障，不叫重开。
2. **本次只准备验证 1 个主修订动作**  
   - 例如只改原型边界，或只改 entry proxy，或只重排 P0/P1/P2。  
   - 如果一次准备同时改 3 件事，这不是重开，是重洗牌。
3. **其余关键口径已经冻结**  
   - 成功标准、窗口定义、样本周期、其余未被点名的问题层都先不动。  
   - 否则新一轮结果无法与上一轮比较。
4. **首轮产物链没有断**  
   - 标签规范、原型定义、因子优先级、执行约束、rollout gate、复盘文档至少要保持可追溯。  
   - 可以修订，但不能边跑边丢。
5. **重开后的结束条件已经写清**  
   - 必须事先写明：这轮结束后要回答哪一个问题、产出哪一份修订版文档、以及什么情况下仍判定不过关。  
   - 没有结束条件，重开很容易变成无限拖延。

可以把它压成一张更直观的判定表：

| 判断项 | 允许重开时应满足什么 | 不满足时该做什么 |
| --- | --- | --- |
| 主问题是否明确 | 已锁定主要问题层 | 继续留在 `11.1` 排障 |
| 修改是否单一 | 只改 1 个主动作 | 回到 `11.2` 继续删动作 |
| 基线是否冻结 | 主目标和大口径不漂 | 回到 `11.3` 检查是否乱改 |
| 产物是否还在 | `12` 的产物链还能接上 | 先补文档链，不重开 |
| 结束条件是否清楚 | 能写成 1 到 2 句验收语句 | 先写清复盘目标 |

一句话说，**重开首轮的前提，不是还有热情，而是已经有单一修订假设、冻结比较基线、保住文档链条**。

#### 11.4.1 重开之后，什么信号说明这次修订仍然失败

允许重开首轮，不等于允许无限拖延。  
如果带着单一修订假设重开后，下面这些信号仍然出现，就应明确判定：**这次修订没有把主问题解决掉**。

建议重点看下面 5 个退出信号：

1. **被点名的主问题层没有改善**  
   - 例如本来就是为了解释 executable 塌陷而重开，但重开后 executable 仍然没有任何更清楚的剥离逻辑。  
   - 这说明修订动作没有打到主问题。
2. **唯一修订动作之外，又额外冒出了多个新问题**  
   - 本轮本来只该验证 1 个动作，结果越做越多，新的动作重新超过 2 个。  
   - 这通常说明修订没有收敛，反而把问题面再次放大了。
3. **重开后只能解释局部好看，不能解释整体稳定**  
   - 某一小段样本或单一窗口变好，但跨窗、跨阶段、跨账本仍然没有更清楚的主线。  
   - 这种情况不能算修订成功，只能算局部噪声更会讲故事。
4. **冻结基线被偷偷改动后才显得变好**  
   - 如果不改窗口、不改成功标准、不改 entry proxy 时看不到改善，而一改口径就“似乎好了”，要直接判为本轮不可比较。  
   - 这不叫修订成功，叫比较基线漂移。
5. **重开后的复盘，仍然写不出唯一主结论**  
   - 如果最后还是压不出“这次修订到底验证了什么、排除了什么、下一步只剩什么”，那就说明这轮依旧失败。  
   - 不能因为做了更多分析，就把失败包装成进展。

可以把它压成一张退出板：

| 退出信号 | 说明什么 | 应该怎么判 |
| --- | --- | --- |
| 主问题层没改善 | 修订没打中核心 | 判本次修订失败 |
| 新动作再次膨胀 | 修订没收敛 | 回到 `11.5` / `11.7` 缩主题 |
| 只有局部窗口好看 | 稳定性仍不足 | 不进入第二轮 |
| 靠改基线才变好 | 比较失效 | 判为不可横向比较 |
| 写不出唯一主结论 | 复盘没收口 | 判为重开失败 |

重开失败后，默认不要立刻再次重开。  
更合理的动作顺序是：

1. 回到 `11.5` 判断是否必须缩主题；
2. 用 `11.6` 的一页版复盘把这次失败收口；
3. 再用 `11.7` 判断下一次若继续收缩，先缩哪一层。

一句话说，**重开的成功标准不是“看起来比上次忙了很多”，而是被点名的问题层真的更清楚了、动作真的更少了、复盘真的能收成一句主结论**。

### 11.5 什么情况下，必须缩主题，而不是继续加因子

首轮不过关后，团队最容易产生一种错觉：  
**“也许不是方法错了，只是因子还不够多。”**

但在大多数情况下，如果主原型、主标签、执行口径都还没收敛，继续往里加因子，通常只会把问题越藏越深。  
因此，下面这些信号一旦同时或反复出现，正确动作就不是继续扩候选，而是**先缩主题**：

1. **三个原型都没有稳定胜出者**  
   - 说明不是“缺更多因子”，而是研究主题本身还过宽。  
   - 正确动作：先把研究焦点压到 1 个最像样的原型。
2. **同一家族在不同窗口里结论反复翻向**  
   - 说明当前主题下，因子解释权不稳定。  
   - 正确动作：先缩家族范围，只保留最能解释的那一支。
3. **raw / executable 分歧大到无法解释**  
   - 说明问题还停留在执行可达性层，不适合继续加新因子。  
   - 正确动作：先缩样本与执行条件，把不可成交噪声压下去。
4. **P0 始终选不出来，P1 / P2 不断轮流像主线**  
   - 说明主题太宽，导致“谁都像主线，谁也不像主线”。  
   - 正确动作：先缩主题，不先扩交互。
5. **下一步动作已经多于 2 个**  
   - 说明收口失败，不是候选不足。  
   - 正确动作：先删动作，保主线，再决定是否重开。

可以直接用下面这张判定表：

| 看到的现象 | 更应该做什么 | 不该做什么 |
| --- | --- | --- |
| 原型没有胜出者 | 缩到 1 个主原型 | 再铺更多原型因子 |
| 家族解释反复翻向 | 缩到 1 个主家族 | 同时补更多家族 |
| raw / executable 无法解释 | 缩执行口径与样本 | 跳过 beta 直接扩 alpha |
| P0 长期选不出来 | 缩主题与层级 | 用更多交互强行选主线 |
| 下一步动作过多 | 缩任务面与结论面 | 继续并行加分析 |

缩主题时，建议遵守下面的顺序：

1. **先缩原型，再缩家族**  
   - 先决定只保留哪一个事件原型。  
   - 不要三个原型一起保留，再去做家族删减。
2. **再缩家族，再缩交互**  
   - 先保留最能解释的主家族。  
   - 交互项只留给已经站稳的主线，不拿来救火。
3. **最后才考虑彻底换题**  
   - 只有当原型、家族、执行口径都缩过以后，仍然没有主线，才考虑换主题。  
   - 换题是最后动作，不是首轮一受挫就切题。

一句话说，**当问题像“主题过宽、解释过散、动作过多”时，优先缩主题；只有当主题已经很窄，才值得讨论是否继续加因子**。

### 11.6 缩主题后，如何用一页版复盘收口

缩主题不是一句“先做小一点”就结束。  
如果没有把缩完之后的结论压成一页，团队下一轮通常又会把刚删掉的东西重新捡回来。

因此，每次执行完 `11.5` 的缩主题动作后，建议立刻补一份**一页版缩主题复盘**。  
它的目标不是讲完整故事，而是把“这次到底删掉了什么、保住了什么、下一轮只剩什么”固定下来。

推荐固定写成下面 6 段：

1. **这次缩掉了什么**  
   - 缩掉了哪个原型、哪个家族、哪类交互、哪类样本。  
   - 直接写删减结果，不要写成长篇讨论过程。
2. **为什么必须缩**  
   - 对应 `11.5` 里的哪一个失败信号。  
   - 最好只写 1 到 2 条主因，不要把所有次要噪声都塞进来。
3. **缩完后保留什么主线**  
   - 只保留哪个主原型、哪个主家族、哪个主口径。  
   - 如果主线还不清楚，就明确写“仍未形成主线”，不要强行美化。
4. **哪些比较基线保持不动**  
   - 主目标、窗口、entry proxy、大的执行定义哪些没有变。  
   - 这一步是为了保证下一轮还能和上一轮比较。
5. **下一轮只允许做什么**  
   - 最多写 1 到 2 个动作。  
   - 例如“只验证收缩后原型的 P0 分离度”或“只解释 executable 塌陷原因”。
6. **下一轮明确不做什么**  
   - 例如“不恢复已删原型”“不新增家族”“不扩交互”。  
   - 这段越短越好，但必须存在。

如果希望更快落地，可以直接套下面这个一页版骨架：

| 模块 | 只需要回答什么 |
| --- | --- |
| 本次删减 | 我们删掉了哪些原型 / 家族 / 交互 / 样本 |
| 删减原因 | 是哪一个失败信号逼着我们缩主题 |
| 当前主线 | 缩完以后还剩下什么值得继续保留 |
| 冻结基线 | 哪些标准和口径本轮没有动 |
| 下一轮动作 | 只剩哪 1 到 2 个动作 |
| 下一轮禁区 | 明确哪些内容禁止恢复或扩展 |

什么时候算这份一页版复盘写合格？建议只看 3 个标准：

1. **别人接手时，能一眼看懂删了什么**  
   - 如果接手人还得再读长篇讨论，说明没收口。
2. **下一轮动作不超过 2 个**  
   - 超过 2 个，说明还没真正缩完。
3. **不会把刚删掉的内容偷偷加回来**  
   - 如果“禁区”写不出来，说明缩主题只是口头动作。

一句话说，**缩主题的完成标志，不是口头上说范围变小了，而是已经用一页文档把删减、保留、冻结、禁区都钉住了**。

### 11.7 缩主题时，先缩原型、先缩家族，还是先砍交互

很多团队知道“该缩了”，但真正动手时又会卡住：  
**到底是先缩事件原型，还是先缩因子家族，还是先把交互项砍掉？**

建议不要凭感觉决定，而是按失败信号分流：

| 主要失败信号 | 第一优先缩什么 | 为什么 | 暂时不要先动什么 |
| --- | --- | --- | --- |
| 3 个原型都没有明显胜出者 | **先缩原型** | 连研究场景都没站稳，先谈家族和交互没有意义 | 不要先扩家族或补交互 |
| 同一原型下，多个家族轮流像主线 | **先缩家族** | 场景大体对了，但解释层太散，需要先收主家族 | 不要急着换原型 |
| 一阶因子还能看，但交互一上就结论漂 | **先砍交互** | 主线还没坏，坏的是复杂度过高 | 不要把一阶主线一起删掉 |
| raw 还能看，executable 一加约束就塌 | **先缩样本 / 执行口径** | 问题优先出在 beta 可成交层，不在 alpha 扩搜层 | 不要先加新因子 |
| P0 总是立不住，P1 / P2 不断轮换主线 | **先缩家族，再检查原型** | 先看是否家族面过宽；若缩后仍不稳，再回头缩原型 | 不要先上更复杂交互 |
| 下一步动作已经多到压不成 2 个 | **先砍交互和次级动作** | 这通常是收口失败，而不是研究面不足 | 不要继续并行铺分析 |

可以把这张表压成一个更快的判断顺序：

1. **先问：是不是连场景都没站稳**  
   - 如果是，先缩原型。
2. **再问：场景稳了，但解释是不是太散**  
   - 如果是，先缩家族。
3. **再问：主线还在，但复杂度是不是把结论带偏了**  
   - 如果是，先砍交互。
4. **最后问：是不是其实卡在 executable**  
   - 如果是，先缩样本和执行口径，不要误判成因子不够。

为了避免缩错层，建议再加一条反向纪律：

- **原型没站稳时，不讨论交互救火。**
- **家族没站稳时，不讨论更多家族补洞。**
- **一阶主线没站稳时，不拿交互项强行制造主线。**

一句话说，**场景不稳先缩原型，解释太散先缩家族，复杂度过高先砍交互，可成交性失真先缩执行口径**。

### 11.8 缩主题时，alpha / beta / gamma 各自该交什么短结论

当团队已经决定缩主题时，不需要再交一整套长报告。  
这时候更重要的是：**三个人都用最短的话，把“为什么缩、缩掉什么、还剩什么”说清楚。**

这一节不是替代 `10.4` 的完整交付模板，而是提供一套**缩主题专用短结论卡**。  
每个人最好都压缩到 3 句，避免一边说要收缩，一边又把结论写散。

#### alpha：研究主线收缩卡

alpha 的短结论建议固定成 3 句：

1. **本轮决定缩掉什么研究面**  
   - 例如“暂停 breakout ignition，只保留 trend continuation”。
2. **为什么必须缩**  
   - 例如“当前没有稳定 P0，多个家族轮流像主线，说明解释面过宽”。
3. **缩完后只保留什么统计主线**  
   - 例如“下一轮只验证 trend continuation 下的 volume-expansion P0 分离度”。

alpha 最重要的纪律：

- 不把“还想再看看”的候选偷偷留在主线里。
- 不把未收敛的解释写成已经成立的统计结论。

#### beta：执行收缩卡

beta 的短结论建议固定成 3 句：

1. **本轮决定缩掉什么执行噪声**  
   - 例如“删除高跳空 impossible-fill 样本簇，暂不讨论低流动性尾部样本”。
2. **为什么必须缩**  
   - 例如“raw / executable 分歧主要来自不可复制成交，而不是 alpha 因子本身”。
3. **缩完后只保留什么执行口径**  
   - 例如“下一轮只保留一种 entry proxy 和一套降权规则，不再同时比较多套执行假设”。

beta 最重要的纪律：

- 不把执行问题重新包装成“再加几个因子看看”。
- 不在缩主题轮次里同时试很多套 entry proxy。

#### gamma：治理收缩卡

gamma 的短结论建议固定成 3 句：

1. **本轮决定把哪些内容降级或暂停**  
   - 例如“将 compression-to-expansion 暂停，仅保留观察层，不进入下一轮放大”。
2. **为什么必须降级**  
   - 例如“当前证据只能支撑局部窗口，不足以支撑候选层晋升”。
3. **缩完后下一轮只允许推进什么**  
   - 例如“下一轮只允许验证 trend continuation 在观察层到候选层之间能否形成稳定晋升证据”。

gamma 最重要的纪律：

- 不因为团队已经投入很多时间，就放宽晋升门。
- 不让“先留着吧”变成隐性继续扩范围。

#### 三方统一的缩主题收口格式

如果要在会议里更快收口，alpha / beta / gamma 三个人都可以统一压成下面 3 句：

1. **我们这次缩掉了什么。**
2. **我们为什么非缩不可。**
3. **我们缩完后下一轮只保留什么。**

只要三个人有任意一个人的第三句还超过 2 个动作，就说明这次缩主题还没真正完成。

一句话说，**缩主题轮次里，alpha 负责压研究面，beta 负责压执行噪声，gamma 负责压晋升范围；三个人都应该把结论压到“删什么、为什么删、删完只留什么”这 3 句里**。

### 11.9 缩主题后的结果，什么时候允许重新升级

缩主题不是永久降级。  
但也绝不能因为范围缩小后“看起来顺眼了一些”，就立刻把观察层结果重新升回候选层。

更稳妥的做法是：只有当**缩主题后的结果，在更窄范围内依然保持可解释、可比较、可执行，并且没有靠偷改基线换来改善**，才允许讨论重新升级。

建议至少同时满足下面 5 条，才允许从“缩主题后的保留状态”重新升级：

1. **主线已经重新变清楚**  
   - 缩完后，alpha 能明确说出唯一主原型或唯一主家族。  
   - 如果缩完仍然是谁都像主线，就还不配升级。
2. **raw / executable 同向恢复**  
   - beta 需要证明：缩完以后，不只是 raw 更好看，executable 也更一致。  
   - 如果只是纸面收益变漂亮，执行口径仍然塌，最多继续留观察层。
3. **gamma 的稳定性门重新通过**  
   - 至少不能再是“只靠单窗、只靠局部样本、只靠局部 regime”才能成立。  
   - 如果稳定性问题没解决，不能因为研究面变小就默认升级。
4. **缩掉的内容没有被偷偷加回来**  
   - 被删掉的原型、家族、交互、样本，不应在升级判断时重新混进来。  
   - 否则这不是缩后升级，而是悄悄恢复旧范围。
5. **下一轮动作仍然少于等于 2 个**  
   - 升级后的下一轮，动作仍必须很少。  
   - 如果一升级就重新膨胀出很多动作，说明这次回升不稳。

可以直接用下面这张“回升判断板”：

| 判断项 | 允许重新升级时应满足什么 | 不满足时该怎么做 |
| --- | --- | --- |
| 主线清晰度 | 已只剩 1 条主线 | 继续留在缩主题状态 |
| 执行一致性 | raw / executable 同向恢复 | 退回 beta 继续压执行噪声 |
| 稳定性 | 不再只靠局部窗口 / 局部样本 | 继续停留观察层 |
| 范围纪律 | 已删内容未被恢复 | 回到 `11.6` 检查禁区是否失守 |
| 动作数量 | 下一轮动作仍不超过 2 个 | 回到 `11.5` 再次缩主题 |

为了避免“假升级”，还建议加 3 条反向纪律：

1. **不要因为缩后 uplift 变大，就默认可以升级。**  
   - 先问 uplift 变大是不是来自样本变干净，还是来自基线漂移。
2. **不要因为团队更有信心，就默认可以升级。**  
   - 信心不是证据，回升必须靠更清楚的主线和更稳定的执行口径。
3. **不要因为已经停太久，就默认该往上推。**  
   - 时间压力不能替代晋升证据。

一句话说，**缩主题后的回升，必须同时证明“主线更清楚、执行更一致、稳定性更好、范围没偷扩、下一轮动作仍然很少”，否则宁可继续留在观察层，也不要假升级**。

### 11.10 缩主题后仍停在观察层时，什么时候继续观察，什么时候彻底暂停

不是所有没能重新升级的结果，都应该立刻暂停。  
但也不是所有“还有点意思”的结构，都值得一直挂在观察层里慢慢看。

更合理的做法是：把“继续观察”和“彻底暂停”分成两类完全不同的结论。  
**继续观察**，意味着它还有清楚的未验证问题；**彻底暂停**，意味着它已经暂时不值得再占用研究注意力。

建议优先按下面 5 个信号判断：

1. **主问题是否已经被压缩到 1 个清楚问题**  
   - 如果只剩一个明确问题还没验证，比如“executable 为什么仍然略弱”，可以继续观察。  
   - 如果连主问题都说不清，只剩一堆模糊疑点，更像应该暂停。
2. **观察层结果是否还有稳定结构提示**  
   - 如果 raw / executable 虽然还不够升级，但方向至少不再乱翻，可以继续观察。  
   - 如果连结构提示都不稳定，继续挂观察层只会拖延收口。
3. **下一步是否只剩 1 个低成本验证动作**  
   - 例如只补一个窗口切分、只验证一个 entry proxy 假设。  
   - 如果继续观察还需要很多新动作，就不该叫观察，而应暂停。
4. **观察不会重新打开旧范围**  
   - 继续观察的前提是：不会把已经删掉的原型、家族、交互重新捡回来。  
   - 如果一观察就要恢复旧范围，直接暂停更干净。
5. **这条线是否仍能说清对主目标的潜在价值**  
   - 如果还能说明它未来可能帮助 `5 日内超过 15% / 概率 > 55%` 这条主目标，可以继续观察。  
   - 如果已经说不清它和主目标的关系，就该暂停。

可以直接用下面这张“留观还是暂停”判断板：

| 判断项 | 更像继续观察 | 更像彻底暂停 |
| --- | --- | --- |
| 主问题清晰度 | 只剩 1 个明确问题 | 问题仍然发散 |
| 结构提示 | 还有稳定方向感 | 连方向感都不稳 |
| 下一步动作 | 只剩 1 个低成本动作 | 仍需多步并行验证 |
| 范围纪律 | 不会恢复旧范围 | 一继续就会偷扩范围 |
| 主目标关联 | 仍能说明潜在价值 | 已说不清为何值得继续 |

如果还想更快判断，可以直接记住下面这条经验线：

1. **还有清楚问题、清楚价值、清楚单步动作** → 继续观察。  
2. **没有清楚问题、没有清楚价值、没有清楚单步动作** → 直接暂停。

为了防止“观察层变成垃圾回收站”，再加 3 条纪律：

1. **观察层不能长期容纳没有主问题的问题。**
2. **观察层不能容纳一继续就会扩回旧范围的主题。**
3. **观察层不能代替暂停决定。**

一句话说，**观察层只保留“还有清楚问题、清楚价值、清楚单步验证动作”的结果；如果三者缺任意两个，就更应该直接暂停，而不是继续挂着不收口**。

#### 11.10.1 观察层条目每次复查时，必须回答的固定问题清单

观察层最怕的不是证据暂时不够，而是**每次复查都像第一次看它**。  
为了避免观察层条目无限期滞留，建议每次复查都固定回答下面 7 个问题：

1. **这条线现在还剩下哪 1 个未验证主问题？**  
   - 如果已经说不出唯一主问题，默认不该继续留观。
2. **如果这个问题被验证，优先改善的是胜率，还是盈亏比？**  
   - 必须二选一先说主贡献方向。  
   - 如果两者都想兼顾但说不清主次，通常说明价值叙事还太散。
3. **raw / executable 现在是否仍保持同方向结构提示？**  
   - 如果方向又重新乱翻，观察价值已经显著下降。
4. **下一步是否仍只剩 1 个低成本验证动作？**  
   - 例如只补 1 个窗口切分、只验证 1 种 entry proxy、只检查 1 个样本簇。  
   - 如果复查后又冒出多个动作，说明它不该继续挂在观察层。
5. **这次复查有没有偷偷恢复已删范围？**  
   - 包括已删原型、家族、交互、样本。  
   - 一旦恢复旧范围，这次复查结果默认失真。
6. **这条线现在离三种结论哪一种更近：回升、继续观察、暂停？**  
   - 复查后必须明确更接近哪一边，不能只写“再看看”。
7. **如果再给它一次复查机会，最晚下次必须看到什么变化？**  
   - 必须写成一句可判断的话。  
   - 例如“executable 不再显著弱于 raw”或“主线只剩 1 个 P0 家族”。

也可以直接用下面这张复查表：

| 固定问题 | 复查时必须给出的最短答案 |
| --- | --- |
| 未验证主问题是什么 | 只写 1 个问题 |
| 主贡献方向是什么 | 胜率 / 盈亏比 二选一主方向 |
| 结构是否仍同向 | 是 / 否 |
| 下一步动作是否单一 | 是 / 否 |
| 是否偷扩旧范围 | 是 / 否 |
| 当前更接近哪种结论 | 回升 / 留观 / 暂停 |
| 下次复查看什么 | 1 句验收条件 |

如果 7 个问题里，下面任意 2 条同时出现，就更应该从留观转向暂停：

1. 说不出唯一主问题。
2. 说不清它主要改善胜率还是盈亏比。
3. 下一步动作重新超过 1 个。
4. 复查时偷偷恢复了旧范围。
5. 写不出下次复查的唯一验收条件。

一句话说，**观察层复查不是重复看旧材料，而是强制回答：它还在解决哪个问题、优先改善胜率还是盈亏比、下一次如果还没变好就该往哪里去**。

### 11.11 把 11.1-11.10 压成一张总决策树

如果前面十段已经分别解释了“怎么排障、怎么缩、怎么回升、怎么暂停”，  
那这一段的目标就是把它们压成一条可以顺着走的总路径。

建议把第 `11` 章整体理解成下面这棵树：

| 当前问题 | 先看哪一节 | 要回答的核心问题 | 默认去向 |
| --- | --- | --- | --- |
| 首轮为什么不过关 | `11.1` | 主问题先落在定义、执行，还是治理 | 进入排障 |
| 排障后先改什么 | `11.2` | 先改定义、执行、层级，还是最后才改主题 | 进入单点修订 |
| 什么不能乱改 | `11.3` | 哪些动作会破坏可追溯性 | 锁住比较基线 |
| 能不能重开首轮 | `11.4` | 是否已具备单一修订假设和冻结基线 | 能则重开，不能则继续收口 |
| 重开后是否仍失败 | `11.4.1` | 这次修订有没有真正打中主问题 | 失败则转向缩主题 |
| 是否必须缩主题 | `11.5` | 当前问题是主题太宽，还是证据还不够 | 需要时进入缩主题 |
| 先缩哪一层 | `11.7` | 先缩原型、家族、交互，还是执行口径 | 进入定向收缩 |
| 缩完如何收口 | `11.6` + `11.8` | 如何用一页版复盘和三方短结论固定结果 | 形成缩主题闭环 |
| 缩后能否回升 | `11.9` | 是否已经重新具备升级条件 | 能则回升，不能则留观/暂停 |
| 不能回升时怎么办 | `11.10` | 继续观察是否仍有价值，还是该彻底暂停 | 留观或暂停 |

如果想更快走，可以直接按下面 7 步：

1. **先定位主问题层**  
   - 用 `11.1` 判断先查定义、执行还是治理。
2. **再决定是否只做单点修订**  
   - 用 `11.2` 和 `11.3` 决定能否安全地只改一个主动作。
3. **如果能单点修订，就检查是否满足重开条件**  
   - 满足 `11.4` 才允许重开；不满足则不重开。
4. **重开后立刻判断修订有没有打中主问题**  
   - 用 `11.4.1` 看是否仍然失败。
5. **如果单点修订仍失败，就判断是否必须缩主题**  
   - 用 `11.5` 决定是否从“修补”切到“收缩”。
6. **进入缩主题后，先定向收缩，再完成收口**  
   - 用 `11.7` 决定先缩哪层；用 `11.6` 和 `11.8` 把结果写短、写死。
7. **缩完后只在两条路里二选一**  
   - 满足 `11.9` 就重新升级；不满足则按 `11.10` 判断继续观察还是暂停。

还可以把这条路再压成一句更短的总线：

**先排障 → 再单点修订 → 修订失败就缩主题 → 缩完后只允许回升、留观、暂停三选一。**

为了避免读完还是不会用，再补 4 条总纪律：

1. **不要在还没排障时就直接缩主题。**
2. **不要在还没冻结基线时就直接重开。**
3. **不要在缩主题还没收口时就急着回升。**
4. **不要在留观已经失去单步价值时还拒绝暂停。**

一句话说，**第 `11` 章不是十个平行技巧，而是一条连续决策链：先判断问题在哪，再判断能否修、该不该缩、缩后能否升、升不了是留还是停**。

#### 11.11.1 当高胜率与高盈亏比冲突时，方法上怎么优先取舍

短线研究里最危险的错觉之一，是把“胜率更高”自动等同于“方法更好”，或者把“单笔盈亏比更大”自动等同于“更值得放大”。  
对这份文档的主目标来说，**两者都不是单独的终点**。真正该优先守住的，是：

1. **主目标不漂移**  
   - 仍然围绕“`5` 个交易日内、`55%+` 概率、涨幅超过 `15%`”这条主线。  
   - 不能因为某因子更容易提高胜率，就把目标偷偷放宽成更低涨幅；也不能因为某因子单笔收益更高，就接受胜率明显衰减。
2. **先守可执行命中，再谈更大弹性**  
   - 如果一个因子让 raw 胜率看起来更高，但 executable 明显更差，这不是更优方法。  
   - 如果一个因子让极少数样本爆发更强，但大多数样本根本买不到，也不是更优方法。
3. **先守稳定主线，再争夺额外赔率**  
   - 首轮与收缩轮次里，优先保住“能稳定解释、能稳定执行、能稳定留在候选层”的主线。  
   - 只有主线已经站稳，才值得再去争取更高单笔盈亏比。

换成更实用的话，就是下面这条优先顺序：

**先排除假高胜率 → 再排除假高盈亏比 → 再比较谁更值得继续放大。**

建议遇到冲突时，固定按下面 4 问判断：

| 问题 | 如果答案是“否” | 默认动作 |
| --- | --- | --- |
| 它是否仍然服务于 `5 日 / 55% / 15%` 主目标 | 主目标已漂移 | 直接降级或重定义 |
| 它的优势是否在 executable 账本下仍成立 | 只是纸面改善 | 退回 beta / 观察层 |
| 它的改善是否跨窗口、跨样本仍保方向 | 只是局部高光 | 退回 gamma 稳定性审查 |
| 它是否让下一轮动作仍然少于等于 2 个 | 放大后又重新发散 | 不放大，先收口 |

如果上面 4 问都能过，再比较“更偏胜率”还是“更偏盈亏比”：

1. **优先保留“胜率先稳住、盈亏比不明显恶化”的因子**  
   - 这类因子更适合作为 admission / gate / baseline 筛选线。
2. **谨慎保留“盈亏比更高、但胜率略降且仍在主目标线内”的因子**  
   - 这类因子更像 frontier / score / relief 候选，但前提是胜率没有跌穿主目标要求。
3. **直接降级“靠极少数暴利样本抬高盈亏比、但整体命中显著转弱”的因子**  
   - 这通常不是高赔率因子，而是尾部样本幻觉。
4. **直接降级“靠放宽标签或放宽 entry proxy 提高胜率”的因子**  
   - 这通常不是高胜率因子，而是口径漂移。

也可以直接记成一条简单规则：

- **胜率提升，但靠口径漂移换来** → 不要。  
- **盈亏比提升，但靠极少数不可复制样本换来** → 不要。  
- **胜率先稳住，盈亏比不恶化** → 优先保留。  
- **盈亏比更高，但胜率仍守住主目标底线** → 受控保留。  

一句话说，**短线高胜率高盈亏比因子的最佳方法，不是二选一押胜率或押盈亏比，而是先守住可执行主目标和稳定主线，再在不破坏主目标的前提下争取更高赔率**。

## 12. 首轮产物清单：文档必须沉淀成什么

为了让“继续研究”不变成继续空谈，首轮之后至少要沉淀出下面这些产物：

1. 一份**标签规范文档**：统一主/次标签、entry proxy、排除规则。
2. 一份**事件原型定义文档**：只保留首轮 3 个事件原型。
3. 一份**候选因子优先级表**：按原型和家族整理一阶候选。
4. 一份**执行约束说明**：不可成交、滑点、容量、拥挤处理方式。
5. 一份**rollout gate 说明**：候选如何进入观察层、候选池、runtime 候选池。
6. 一份**首轮复盘文档**：明确哪些原型继续、哪些家族暂停、哪些交互保留。

这些产物如果没有沉淀，下一轮研究很容易再次回到“每次都从头讨论”的低效状态。

### 12.1 首轮复盘文档模板

首轮最重要的不是“做过很多分析”，而是最后能不能沉淀出一份别人接手也能继续做的复盘文档。

建议首轮复盘文档固定写成下面 8 段：

1. **本轮范围**  
   - 这轮只研究了哪个事件原型、哪几个因子家族、哪几组交互。  
   - 明确写出“这轮没有研究什么”。

2. **标签与执行口径**  
   - 主标签、次标签、raw / executable 双账本。  
   - 使用的 entry proxy 与 impossible-fill 规则。  
   - 如果这部分口径有变动，直接判为本轮不可横向比较。

3. **事件原型结论**  
   - 3 个首轮原型里，哪个原型最值得继续放大。  
   - 哪个原型虽然有故事，但当前证据还不够。  
   - 哪个原型应暂时停止投入。

4. **因子层级结论**  
   - P0 主因子里谁最强。  
   - 哪些 P1 因子只是确认项，不能误升为主线。  
   - 哪些 P2 约束因子证明了“别放错票”比“多放票”更重要。

5. **raw vs executable 分歧总结**  
   - 哪些候选 raw 很强但 executable 明显衰减。
   - 哪些候选在 executable 账本下仍然成立。  
   - 这部分必须直接决定哪些因子只能留在观察层。

6. **alpha / beta / gamma 三方共识**  
   - alpha 最强证据是什么。  
   - beta 最大执行问题是什么。  
   - gamma 最终把它放在哪一层。  
   - 如果三方结论冲突，要明确写冲突点，而不是只写平均结论。

7. **本轮最终判定**  
   - 只允许写成：暂停、观察层保留、候选层保留、交互候选层保留。  
   - 不允许在首轮复盘里直接写“可上线”。

8. **下一轮唯一动作**  
   - 只允许保留 1 到 2 个下一步动作。  
   - 每个动作都必须同时写“为什么现在做”和“为什么其他动作现在不做”。

### 12.2 首轮复盘文档必须回答的 6 个问题

如果没空逐段检查 `12.1` 的完整模板，至少先用下面 6 个问题做一轮快筛：

1. 这轮最值得继续放大的事件原型是哪一个。
2. 这轮最像主线的 P0 因子是哪一个。
3. 哪个候选只在 raw 账本成立、不能进入交易候选讨论。
4. 最大执行风险来自哪一类样本。
5. gamma 最终把最强候选放在哪一层。
6. 下一轮为什么只做那 1 到 2 个动作。

### 12.3 首轮复盘文档的禁区

为了避免复盘重新变成空话，首轮复盘文档明确不允许：

1. 同时推荐 4 个以上下一步动作。
2. 不区分 raw 与 executable 就直接给因子结论。
3. 不区分原型就把所有候选混成一张表。
4. 用单窗亮点替代多窗口结论。
5. 把“结构上有意思”直接写成“可进入 runtime”。

### 12.4 首轮 6 份产物的依赖关系

首轮产物不是 6 份平行文档，而是一条有先后顺序的链路。

推荐按下面顺序产出：

1. **标签规范文档**
2. **事件原型定义文档**
3. **候选因子优先级表**
4. **执行约束说明**
5. **rollout gate 说明**
6. **首轮复盘文档**

它们之间的依赖关系如下：

| 产物 | 依赖什么 | 为什么必须排在后面 |
| --- | --- | --- |
| 标签规范文档 | 无 | 它定义所有后续研究的口径底座 |
| 事件原型定义文档 | 标签规范文档 | 没有统一标签，原型的进入/退出标准会漂 |
| 候选因子优先级表 | 标签规范文档 + 事件原型定义文档 | 不先定原型，就无法判断哪个因子是主线、哪个只是确认项 |
| 执行约束说明 | 标签规范文档 + 候选因子优先级表 | 不先知道研究对象和标签口径，执行过滤无法有针对性 |
| rollout gate 说明 | 前四项 | gamma 只有在样本、因子、执行语义都固定后，才有资格做晋升治理 |
| 首轮复盘文档 | 前五项 | 复盘是收口，不是替代前置定义 |

#### 哪些顺序是错误的

首轮最容易犯的 4 个顺序错误是：

1. 还没定标签，就先做因子优先级排序。
2. 还没定原型，就先讨论哪个因子值得升级。
3. 还没补执行约束，就先给 gamma 做晋升判断。
4. 还没把前 5 份产物写清，就直接写复盘文档。

#### 最小可执行路径

如果时间不够，首轮至少也要保住这条最小链路：

1. 标签规范文档
2. 事件原型定义文档
3. 候选因子优先级表
4. 执行约束说明
5. 一页版 gamma rollout gate
6. 一页版首轮复盘文档

这 6 个里缺任意一个，下一轮都很容易重新回到“从头讨论”的低效状态。

## 13. 第二轮什么时候开始

第二轮启动不重新发明标准，它直接建立在 `11` 的首轮验收和 `12` 的产物沉淀之上。

更直白地说，只有当**主原型、主因子/家族、执行剥离逻辑、gamma 层级结论**已经基本成形时，才进入第二轮；否则正确动作不是继续铺更多因子，而是回到首轮文档继续压缩范围、修正定义、补齐排除规则。

### 13.1 第二轮启动判定板

为了避免“感觉差不多了就开第二轮”，建议正式进入第二轮前，逐项勾下面这张判定板：

| 维度 | 必须回答的问题 | 通过标准 | 不通过时该做什么 |
| --- | --- | --- | --- |
| 原型主线 | 是否已有最值得继续放大的首轮原型 | 至少 1 个原型明确胜出 | 回到事件原型定义继续收缩 |
| 因子主线 | 是否已有最像主线的 P0 因子 | 至少 1 个 P0 因子进入候选层或交互候选层 | 回到因子矩阵继续删减或重排 |
| 执行口径 | raw / executable 分歧是否已被解释清楚 | beta 能说明哪些样本该剥离、哪些可保留 | 回到执行约束说明补规则 |
| 治理层级 | gamma 是否已给出清晰层级结论 | 最强候选至少能进入观察层以上 | 回到 rollout gate 继续补证据 |
| 下一轮主题 | 第二轮是否只剩 1 到 2 个动作 | 动作足够少且彼此不打架 | 回到首轮复盘文档重新收口 |

只有当这张板大体全绿时，第二轮才真正开始。

### 13.2 第二轮启动的绿灯 / 黄灯 / 红灯

这一节其实就是 `13.1` 启动判定板的颜色版表达：

#### 绿灯：可以进入第二轮

至少同时满足：

1. 已明确最强事件原型；
2. 已明确至少一个主线候选因子；
3. raw / executable 分歧已被 beta 解释；
4. gamma 已经给出候选层或交互候选层结论；
5. 下一轮动作被压缩到 1 到 2 个。

#### 黄灯：可以继续研究，但还不该叫第二轮

常见情况：

1. 原型大致有方向，但还没有明显胜出者。
2. 候选因子只停在观察层。
3. executable 证据还不够稳定。
4. 下一轮动作仍然过多。

黄灯状态下，继续补首轮证据，不要把“还在收敛”误叫成“已经进入第二轮”。

#### 红灯：禁止进入第二轮

只要出现下面任意一种，就不应进入第二轮：

1. 标签口径仍在漂。
2. 原型归类仍然混乱。
3. 候选因子必须靠大范围放松约束才好看。
4. beta 仍无法解释最强候选的可复制性。
5. gamma 还不能说清它更适合进哪类 runtime 语义。

### 13.3 第二轮开始后，首轮不再做什么

第二轮一旦开始，首轮阶段默认停止下面 4 类回滚动作：

1. 不再回头扩大首轮原型范围。
2. 不再重置首轮标签口径。
3. 不再把低优先级因子重新塞回主矩阵。
4. 不再让复盘文档承担前置定义的职责。

这样做是为了避免第二轮换个名字重新回到发散状态。

## 14. 从首轮到第二轮的一张总决策板

如果只想用一张板判断当前研究卡在哪一层，直接按下面顺序过：

| 步骤 | 先问什么 | 通过后去哪 | 不通过时回到哪里 |
| --- | --- | --- | --- |
| 1 | 标签口径是否固定 | 看事件原型 | 回到标签规范文档 |
| 2 | 事件原型是否收敛 | 看因子优先级 | 回到事件原型定义文档 |
| 3 | P0 / P1 / P2 层级是否清楚 | 看执行约束 | 回到候选因子优先级表 |
| 4 | raw / executable 分歧是否被解释 | 看 rollout gate | 回到执行约束说明 |
| 5 | gamma 是否给出清晰层级结论 | 写首轮复盘 | 回到 rollout gate 说明 |
| 6 | 首轮复盘是否把下一步压缩到 1-2 个动作 | 判定是否进入第二轮 | 回到首轮复盘文档 |
| 7 | 第二轮启动板是否大体全绿 | 进入第二轮 | 回到首轮文档继续补证据 |

### 14.1 这张总决策板最适合什么时候用

它最适合下面这类时候：你已经写了不少文档却不知道还缺哪块，或你正准备开启第二轮、接手别人材料，想快速判断现在该补定义、补证据，还是可以继续放大。

### 14.2 一句话用法

如果当前问题答不上来，就不要跳去下一步；先回到该问题对应的上游文档，把定义或证据补齐。
