# Web 后端问题分析

> 范围：`app/backend/` 下的 FastAPI、auth、routes、repositories、database、schemas

---

## 总结

Web 后端的主要风险集中在**安全边界、敏感信息暴露、错误处理一致性、数据库完整性**四个方面。当前实现更接近“本地开发可用”，离“可安全部署”还有明显差距。

---

## 一、严重问题

### 1. 认证可被环境变量整体绕过

- **证据**：`app/backend/auth/dependencies.py:16,33`
- **问题**：`AUTH_DISABLED=true` 时直接返回模拟管理员身份
- **风险**：
  - 若配置泄漏或部署失误，整个系统等于裸奔
  - 不需要登录即可访问受保护接口

这是典型的开发开关进入生产的问题，必须在生产环境禁止。

### 2. JWT 密钥存在硬编码默认值

- **证据**：`app/backend/auth/utils.py:12-14`
- **问题**：未配置 `AUTH_SECRET_KEY` 时使用默认密钥
- **风险**：
  - 攻击者可伪造合法 token
  - token 安全性失效

正确做法应是：启动即失败，而不是自动退回到默认值。

### 3. 文件保存接口存在路径遍历漏洞

- **证据**：`app/backend/routes/storage.py:31`
- **问题**：`request.filename` 未做路径校验，直接拼到目标目录
- **潜在攻击**：
  - `../../.env`
  - `../../../app/backend/main.py`
- **影响**：
  - 覆盖任意可写文件
  - 污染配置、日志或输出目录外文件

### 4. API Key 通过接口明文暴露

- **证据**：`app/backend/routes/api_keys.py:170`
- **问题**：接口响应中返回 `key_value`
- **风险**：
  - 任意具备访问权限的调用方都能拿到真实密钥
  - 无法满足最基本的密钥最小暴露原则

## 二、高风险问题

### 1. CORS 过宽

问题表现：

- `allow_methods=["*"]`
- `allow_headers=["*"]`

风险：

- 接口暴露面不必要扩大
- 后续加入认证 Cookie / Header 后更容易出现边界错误

### 2. 默认管理员密码存在弱默认值

问题：

- 管理员默认密码有硬编码 fallback
- 即便只用于初始化，也容易被忽视并流入真实环境

### 3. 路由层大量 `except Exception`

涉及：

- `flows.py`
- `flow_runs.py`
- `api_keys.py`
- 其他 route 模块

问题：

- 真实错误被包装成统一 500
- 错误详情可能直接暴露给客户端
- 数据库错误、参数错误、业务错误没有分层

### 4. 敏感操作缺少审计日志

特别是：

- API key 新增/更新/读取
- 登录失败
- 邀请码使用
- 流程删除

没有审计日志意味着安全事件发生后很难追溯。

### 5. 缺少速率限制

影响接口：

- 登录
- 注册/邀请码
- 保存/运行流程
- API key 管理

风险：

- 暴力破解
- 枚举攻击
- 资源耗尽

---

## 三、中风险问题

### 1. DELETE 语义和响应不一致

部分接口出现：

- `204 No Content` 但返回 body
- 删除集合接口语义不清

这类问题不会马上造成事故，但会持续增加前后端联调成本。

### 2. 输入校验不完整

例如：

- 复制 flow 时 `new_name` 没有长度/格式约束
- 部分分页参数没有统一的最大值约束

### 3. 错误响应模型过弱

当前 `ErrorResponse` 无法稳定表达：

- 错误类别
- 可重试 / 不可重试
- 机器可读错误码

导致前端只能显示“失败了”，无法做更细的用户提示或重试策略。

### 4. 异步路由中执行同步文件 IO

- **证据**：`app/backend/routes/storage.py`
- **问题**：async route 内直接 `open(..., 'w')`
- **影响**：
  - 阻塞事件循环
  - 高并发时拖慢整个 API

### 5. 缺少统一服务层

当前很多 route 直接 new repository 并执行业务逻辑。

后果：

- 路由越来越重
- 测试难 mock
- 业务规则分散在 controller 层

---

## 四、数据库层问题

### 1. SQLite 方案更像开发态，不适合更高并发场景

当前连接配置没有体现更成熟的生产级连接池治理。

风险：

- 锁争用
- 并发写冲突
- 恢复能力弱

### 2. 删除 flow 时可能遗留孤儿数据

问题：

- flow 删除逻辑没有清晰体现级联删除
- flow_run / cycle / artifact 等关联对象可能残留

这会带来：

- 数据脏污
- 统计错误
- 存储膨胀

### 3. 运行编号等逻辑依赖应用层递增

如果多个请求同时创建 run，可能出现竞争条件。

### 4. 事务边界不清晰

尤其在：

- 注册
- 邀请码消费
- 多表写入

如果中间失败，容易出现部分提交。

---

## 五、架构判断

### 1. 当前后端仍偏“原型态”

表现为：

- 开发开关很多
- 默认值偏宽松
- 安全约束靠约定，不靠机制

### 2. 路由、仓储、服务职责分层不够清晰

建议演进为：

- routes：只做协议转换
- services：只做业务规则
- repositories：只做数据读写
- policies / guards：负责权限与安全规则

### 3. 缺少统一的系统级错误治理

应当收敛为：

- 统一异常类型
- 统一 HTTP 映射
- 统一日志字段
- 统一错误码

---

## 六、建议修复顺序

1. 先修复认证绕过、JWT 默认密钥、路径遍历、API key 明文返回
2. 然后补齐 CORS、速率限制、审计日志
3. 再处理事务边界、孤儿数据、删除语义和错误模型
4. 最后推进服务层抽象和更长期的数据库演进

---

## 七、核心判断

Web 后端目前**最大短板不是功能缺失，而是安全边界过松**。如果暴露到真实网络环境，在没有额外反向代理和访问控制保护的前提下，风险较高。
