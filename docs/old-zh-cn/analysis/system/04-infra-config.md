# 基础设施与配置问题分析

> 范围：`pyproject.toml`、`.flake8`、`docker/`、`tests/`、`.env.example`、`.github/`、根目录诊断脚本

---

## 总结

基础设施层最大的问题是**工程治理不统一**：依赖管理双轨并存、Docker 偏开发态、测试门禁缺失、CI/CD 缺失、环境变量治理弱。这些问题不会像安全漏洞那样立刻爆炸，但会持续拖累交付质量。

---

## 一、严重问题

### 1. 同时存在 Poetry 与 uv 两套依赖治理

- **证据**：
  - `pyproject.toml` 同时包含 `[project]`、`[tool.poetry]`、`[dependency-groups]`
  - 仓库同时存在 `poetry.lock` 与 `uv.lock`

问题：

- 谁是唯一真实依赖源不清楚
- 团队成员安装结果可能不同
- Docker、本地、CI 若各用不同工具会出现漂移

这是工程层的核心不确定性来源之一。

### 2. 没有 CI/CD 工作流

- **证据**：`.github/workflows/` 不存在
- **影响**：
  - PR 无自动测试
  - 无 lint 门禁
  - 无构建验证
  - 无安全扫描

这意味着当前质量控制主要依赖人工自觉。

### 3. 已出现专门的“调试/修复脚本”散落在根目录

包括：

- `debug_agent_facts.py`
- `debug_data_analysis.py`
- `diagnose_akshare.py`
- `fix_agent_prompts.py`

这说明项目已经积累了一批系统性问题，但处理方式仍偏人工排查，尚未沉淀为正式工程能力。

---

## 二、高风险问题

### 1. 依赖版本声明不一致

- **证据**：`pyproject.toml:8-37` 与 `50-77`
- **问题**：
  - `[project.dependencies]` 与 `[tool.poetry.dependencies]` 版本写法不同
  - 有的使用 `>=`
  - 有的固定版本
  - 有的带 extras，有的不带

影响：

- 环境不可重复
- 排查依赖问题成本高

### 2. Docker 构建更像开发容器，不像生产镜像

- **证据**：`docker/Dockerfile`
- 主要问题：
  - 直接安装 Poetry 到镜像
  - 没有多阶段构建
  - 依赖和运行时未精简
  - 未体现更严格的镜像瘦身和安全策略

### 3. `docker-compose.yml` 大量重复服务定义

- **证据**：`docker/docker-compose.yml`
- 问题：
  - 多个服务重复 build 同一镜像
  - 大量重复环境变量和 volume 配置
  - `.env` 通过相对路径挂载，移植性弱

### 4. 缺少 `.dockerignore`

影响：

- 构建上下文过大
- 把 `.git`、缓存、日志、虚拟环境等无关文件一起送进构建
- 构建更慢，镜像层更脏

### 5. 环境变量缺少启动时校验

`.env.example` 列了很多变量，但系统没有清晰的：

- 必填项校验
- 可选项说明
- provider 优先级
- 缺失后的失败策略

---

## 三、中风险问题

### 1. Lint 配置过宽松

- **证据**：`.flake8:1-3`
- 问题：
  - 忽略未使用导入 `F401`
  - 忽略 bare except `E722`
  - 忽略超长行 `E501`
  - 行长设为 420

后果：

- 死代码更难被发现
- 危险异常模式不被拦截
- 代码审阅成本上升

### 2. 格式化规则不利于长期维护

- **证据**：`pyproject.toml:94-107`
- 问题：Black / Flake8 都使用 420 行宽

这虽然“减少换行”，但会：

- 降低可读性
- 让 code review 更痛苦
- 放大复杂表达式

### 3. 测试治理仍不完整

虽然存在大量测试文件，但仍有几个明显问题：

- `pytest-cov` 在依赖中，但没有看到覆盖率门槛治理
- 缺少清晰的 coverage 策略
- manual test 混在自动化测试目录
- 集成测试与普通测试边界不够清晰

### 4. 作者与项目元数据仍带模板值

- **证据**：`pyproject.toml:5,43`
- 问题：作者信息仍是占位值

这类问题不影响运行，但说明工程收尾不完整。

### 5. 缺少安全扫描与 pre-commit

未见：

- dependabot
- bandit / safety
- pre-commit hooks

这意味着很多问题只能依赖人工发现。

---

## 四、架构判断

### 1. 当前基础设施偏“单人研发工作流”

特征包括：

- 依赖工具混用
- CI 缺失
- 调试脚本直接放根目录
- Docker 偏便捷运行，而非可复制交付

### 2. 工程流程还没有形成稳定质量门

一个成熟项目通常至少会自动校验：

1. 依赖可解析
2. 单元测试通过
3. 样式/静态检查通过
4. Docker 能构建
5. 安全扫描不过线就阻断

当前这些门基本都不完整。

---

## 五、建议修复顺序

1. 明确只保留一种包管理工具
2. 增加最小 CI：安装、lint、pytest、构建
3. 增加 `.dockerignore`，收敛 compose 重复配置
4. 建立环境变量校验与文档
5. 把根目录诊断脚本迁移到 `scripts/` 或 `tools/diagnostics/`
6. 再逐步收紧 flake8 / black / coverage / pre-commit

---

## 六、核心判断

这个项目已经具备较强功能规模，但工程基础设施仍有明显“原型期遗留”。如果不尽快统一依赖、建立 CI 和收敛配置，后续每次加功能都会把维护成本继续推高。
