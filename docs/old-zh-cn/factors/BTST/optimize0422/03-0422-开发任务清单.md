## 1. 使用边界

在开始开发前，先统一 6 条边界：

1. 0422 专项的主目标是把 BTST 的 T+1 期望收益从负值拉回正值或接近正值，不是扩大覆盖率本身。
2. P2 之前禁止多主题联动改造；每轮只能动一类机制。
3. 所有新逻辑必须先支持 `shadow` 或 feature flag 关闭态，不能直接写死成默认行为。
4. 0422 专项不把 T+2 / T+3 持仓延长作为主线；它们只保留为观察指标。
5. 不允许以“为了保住强势日收益”为理由，继续让小样本、零命中先验或弱延续样本混入正式 selected。
6. 每个阶段都必须输出窗口级对照结果；没有对照结果的改动，不得进入默认升级讨论。

## 3. 开发优先级总览

| 优先级 | 任务 | 目标 | 是否允许改默认行为 | 必须输出 |
| --- | --- | --- | --- | --- |
| P0 | 基线冻结与字段盘点 | 固定当前 BTST 口径与数据合同 | 否 | baseline 字段盘点、20 日基线页、开关注入点说明 |
| P1 | 择日门控 shadow | 只打标签，不拦截交易 | 否 | gate shadow 报表、单测、artifact 字段透传 |
| P2 | 择日门控 hard enforce | 正式压缩弱势日 selected 与仓位 | 是 | 窗口对照报告、降级逻辑测试、回滚开关 |
| P3 | 先验质量硬门槛 | 清理小样本、零命中、弱延续正式精选 | 是 | prior 质量分类、审计脚本、字段透传 |
| P4 | 先验 shrinkage | 用可靠性折算重排中低样本段 | 是 | raw vs shrunk 对照、窗口统计、阈值说明 |
| P5 | 精选/观察层 contract | 拉开 selected 与 near_miss 的执行语义 | 是 | execution eligibility 字段、review 文案、集成测试 |
| P6 | 风险预算 overlay | 把错误日和低质量票的正式暴露压到可控范围 | 是 | 风险矩阵、仓位约束测试、session summary 对照 |

建议真实顺序固定为：`P0 -> P1 -> P2 -> P3 -> P4 -> P5 -> P6`。

禁止跳级规则：

1. P1 未完成，不做 P2。
2. P2 未确认弱势日暴露改善，不做 P3-P6 默认改造。
3. P3 未把显性坏样本清出去，不做 P4 精细化折算。
4. P5 未明确 selected / near_miss contract，不做 P6 正式风险放大。

## 13. 每轮开发的固定交付件

每完成一轮任务，必须同时交付以下 7 件东西：

1. 代码改动说明：本轮只改了哪一层。

3. 测试结果：单测、运行级测试、必要的脚本测试。
4. 对照报表：至少一份窗口级 JSON + Markdown。
5. 风险说明：当前结论是否仍受样本量或窗口偏差限制。
6. 文档更新：README 或本专题文档必须同步。
7. 决策结论：`go`、`shadow only`、`rollback` 三选一。

如果交付件不完整，本轮默认只能记为 `shadow only`。

## 14. 周会汇报时只回答 6 个问题

1. 这一轮到底修的是哪一层。
2. 它改变了哪些正式决策，哪些只是在打标签。
3. 弱势日暴露有没有下降。
4. selected 相对 near_miss 的差距有没有拉大。
5. 有没有误伤强势日主收益样本。
6. 这轮应该继续推进、只保留 shadow，还是立即回滚。

如果这 6 个问题里有任意 2 个答不清，这轮变更就不应进入默认升级讨论。

## 15. 最终执行建议

0422 专项的正确开发姿势，不是“先写代码再看结果”，而是：

1. 先冻结口径。
2. 再做 shadow 标签。
3. 再做弱势日硬门控。
4. 再清理低质量先验。
5. 再重写 selected / near_miss contract。
6. 最后才加风险预算。

只要按这个顺序推进，每一层都能单独归因、单独验收、单独回滚，0422 的优化主线才不会再次退化成“看到单日成功就继续堆规则”。
