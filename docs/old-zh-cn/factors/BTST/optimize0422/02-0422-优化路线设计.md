## 1. 这份设计文要回答什么

这份文档重点回答 5 个问题：

1. 为什么 4/22 单日 4/4 全胜，仍不能推出“策略已经可持续盈利”。
2. 当前 BTST 的主矛盾，到底是择股不准，还是择日错误，还是风险治理缺失。
3. 下一轮应先改哪一层，后改哪一层，避免再次掉进单日样本或小窗口过拟合。
4. 这些优化动作分别应该改到哪些模块，如何用现有脚本做验证。
5. 如果某一层优化无效，应该怎样停损、回滚，而不是把多个变量一起改乱。

阅读完本文档后，你应该能回答：

1. 为什么“先择日，再选股，再控风险”是 0422 之后的正确顺序。
2. 为什么像 `002028`、`300395` 这种小样本或零命中先验，不应该再进入正式精选。
3. 为什么像 `300394`、`688313` 这类“日内空间还行、收盘延续偏弱”的票，更适合保留为观察层或确认型候选，而不是默认精选。
4. 为什么 0422 之后不应继续优先做“大范围阈值放松”，而应先把弱势日暴露和先验质量门槛补齐。
5. 如果后续要落地，这套方案应该怎么拆成阶段化实验。

## 2. 先讲结论

如果只保留最重要的结论，请先记住下面 8 条：

1. 当前 BTST 的主亏损来源不是强势日不赚钱，而是弱势日还在持续入场。
2. 4/22 的 4 只精选全部收涨，说明系统在“强势市场 + 高弹性板块共振”时可以命中好票，但这不是长期正期望的充分条件。
3. 20 日回测里，强势日贡献与弱势日亏损几乎完全对冲，说明优先级第一的改造必须是择日过滤，而不是继续微调单票分数。
4. 当前精选相对 near-miss 的超额只有 2 个百分点，说明精选层没有形成足够强的“执行资格溢价”。
5. 小样本、高幸存者偏差、甚至历史从未命中过 `high>=2%` 的票仍能进入精选，这是第二个必须修的结构漏洞。
6. 风险管理当前更多停留在“没选进去就不交易”，还缺少“即使先验统计看似达标，也要防极端隔夜风险”的显式治理层。
7. T+2 / T+3 胜率继续衰减，说明 0422 之后不应把精力放在延长持有周期，而应先把 T+1 本身修成正期望。
8. 正确的优化路线应拆成 4 层：`择日门控 -> 先验质量门槛 -> 精选/观察层契约 -> 风险预算与验证闭环`。

一句话总结：

> 0422 之后 BTST 的正确主线，不是“让系统多放出几只票”，而是“先停止在错误的日子出手，再禁止低质量先验混进精选，最后把正式交易资格与观察资格彻底分开”。

## 3. 证据基线与问题定义

### 3.1 已被事实确认的 5 个矛盾

根据 [01-0422-实盘复盘与数据证据.md](./01-0422-实盘复盘与数据证据.md)，当前系统已经暴露出 5 个明确矛盾：

| 矛盾 | 证据 | 结论 |
|---|---|---|
| 单日很强，但长期为负 | 4/22 单日精选 4/4 全胜；20 日汇总期望收益 -0.0448%，扣费后更差 | 单日结果不能代表长期期望 |
| 强势日赚钱，弱势日亏得更多 | 强势日贡献 +6749，弱势日贡献 -7502 | 首要问题是弱势日暴露 |
| 精选和候补几乎没差开 | selected 47.2%，near_miss 45.2% | 精选层缺少足够超额 |
| 小样本/零命中先验仍能入精选 | `002028` 历史 `high>=2%` 为 0 仍进精选；`300395` 仅 2 样本也进精选 | 先验质量硬门槛缺失 |
| 机会池有黑天鹅 | `300857` 先验达标却次日 -14.38% | 风险治理不能只依赖历史胜率 |

### 3.2 本轮优化目标

本轮不追求“一次性把 BTST 做成最优系统”，而是先把最能改变期望值的三类问题修掉。

本轮目标按优先级排序如下：

1. 把弱势日的正式入场次数显著压缩。
2. 把小样本、零命中、高回落风险票从正式精选中清出去。
3. 让 selected 相对 near_miss 的统计超额重新拉开。
4. 保留强势日抓高弹性票的能力，不用过度保守把全部收益空间一起砍掉。
5. 在报告层与执行层显式记录“为什么这只票只配观察，不配交易”。

### 3.3 本轮非目标

以下事项在 0422 专项里都不应成为第一优先级：

1. 不优先优化 T+2、T+3 持有策略。
2. 不优先做全局阈值大网格扫描。
3. 不把机会池或 `upgrade_only` 直接升级成正式交易来源。
4. 不把单日 4/22 的成功模式直接外推成默认规则。
5. 不在没有独立验证前，把更多慢变量因子重新写回 admission 主线。

## 4. 根因拆解：当前 BTST 为什么会负期望

### 4.1 根因一：系统没有真正做到“错日不做”

当前 20 日数据已经说明：

1. 强势日平均收盘胜率 76.0%，盈亏比 1.66。
2. 弱势日平均收盘胜率 22.6%，盈亏比 0.93。
3. 两者几乎完全对冲，最终把总期望拖到接近 0 甚至扣费后转负。

这说明：

1. BTST 不是“任何市场环境都可用”的日内外延续策略。
2. 市场环境本身已经比单票细节更能决定收益方向。
3. 如果继续在弱势日硬做精选，再好的单票过滤都会被宏观错误抵消。

### 4.2 根因二：历史先验没有经过可靠性折算

当前精选里同时出现了三类先验质量问题：

1. `n=2`、`n=3` 这种统计不稳定样本，仍被直接当作精选依据。
2. 历史 `high>=2%` 为 0 的票，仍可能被选中。
3. 历史 `close+` 明显低于 50% 的票，没有被降级为“日内确认型观察票”。

这意味着当前系统更像在使用“原始命中率”，而不是“带可靠性惩罚的命中率”。

### 4.3 根因三：精选资格和观察资格没有拉开

根据 [01-0422-实盘复盘与数据证据.md](./01-0422-实盘复盘与数据证据.md) 中的 selected vs near-miss 对比：

1. 胜率只多 2 个百分点。
2. 盈亏比几乎相同。
3. 强势日里 near-miss 还会多次高于 selected。

这说明当前系统并没有把“正式可交易名单”做成足够稀缺、足够有信息含量的集合。

### 4.4 根因四：风险治理只防住了一部分错误

`300857` 的意义很关键：

1. 它说明先验统计即使表面达标，也不等于隔夜风险可控。
2. 这类风险不是“是否入精选”一个动作就能完全解决的。
3. 需要把“交易资格”“仓位规模”“观察层保留”三件事拆开治理。

## 5. 总体设计：四层优化架构

本轮设计采用四层架构，而不是继续把所有问题都扔给单个打分器。

```text
Layer A  择日门控
  先决定今天是否允许 BTST 正式出手，以及允许多大风险预算。

Layer B  先验质量门槛
  再判断单票历史统计是否具有足够可靠性，是否配进入 selected / near_miss。

Layer C  精选/观察层契约
  把 selected、near_miss、upgrade_only 的语义彻底分开，不再混写成“都差不多”。

Layer D  风险预算与验证闭环
  即使前面都通过，也要按风险等级决定仓位、集中度和回滚条件。
```

这四层的顺序不能乱：

1. 先修 Layer A，否则弱势日仍会吞掉全部 alpha。
2. 再修 Layer B，否则强势日里仍会混入低质量样本。
3. 再修 Layer C，否则 selected 与 near_miss 统计上依旧分不开。
4. 最后补 Layer D，用于把黑天鹅损失控制在系统可以承受的范围内。

## 6. 详细方案一：择日门控层

### 6.1 设计目标

择日门控层要解决的问题不是“预测明天涨跌”，而是更务实的三件事：

1. 识别哪些日子继续做 BTST，大概率只会贡献负期望。
2. 把交易状态从“要么做、要么不做”扩展成多档门控。
3. 让短线 profile 不再只是建议，而是能真正约束执行资格。

### 6.2 建议引入的 4 档日级状态

建议把 BTST 的日级状态从隐式 profile，升级成显式 gate：

| Gate 状态 | 含义 | 对交易的影响 |
|---|---|---|
| `halt` | 明显弱势或高翻转风险日 | 不允许产生正式 selected，只保留 research/观察层 |
| `shadow_only` | 环境偏差但未到全面停手 | near_miss 可保留，selected 强制降级为观察层 |
| `normal_trade` | 中性可交易日 | 使用默认 BTST 规则与默认仓位 |
| `aggressive_trade` | 强势确认日 | 可使用更积极的 selected 配置，但仍不放宽低质量先验 |

### 6.3 输入信号与现有代码落点

现有系统已经具备做日级门控所需的大部分输入，不需要从零重建。主要入口如下：

1. 市场状态指标与 profile 推荐：../../../../../src/screening/market_state_helpers.py
2. 执行计划挂接与 target shell 生成：../../../../../src/execution/daily_pipeline.py
3. 短线 profile 配置：../../../../../src/targets/profiles.py

其中 [../../../../../src/screening/market_state_helpers.py](../../../../../src/screening/market_state_helpers.py) 已经提供：

1. `breadth_ratio`
2. `daily_return`
3. `limit_ratio`
4. `style_dispersion`
5. `regime_flip_risk`
6. `regime_gate_level`
7. `recommend_short_trade_profile()`

这意味着 0422 方案不应再单独造一套“择日系统”，而应在现有 market state 基础上增加显式 BTST gate 输出。

### 6.4 MVP 规则设计

第一版不建议上复杂模型，先上可解释、可回放的规则门控。

建议规则：

1. 如果 `regime_gate_level` 属于 `risk_off` 或 `crisis`，直接标记为 `halt`。
2. 如果 `recommend_short_trade_profile()` 返回 `conservative`，但未到 `halt`，优先标记为 `shadow_only`。
3. 如果 `breadth_ratio` 偏低、`regime_flip_risk` 偏高、且近端市场扩散明显恶化，则把默认状态从 `normal_trade` 下调一档。
4. 只有在 breadth、风险扩散和翻转风险同时偏优时，才进入 `aggressive_trade`。

注意：

1. 这层不负责挑票，只负责限制“今天允许不允许正式做票”。
2. 这层宁可先略保守，也不要继续让弱势日大量出手。
3. 所有阈值必须用滚动窗口 shadow 验证，不允许直接拿 4/22 单日设默认值。

### 6.5 对下游的约束动作

不同 gate 状态建议对应以下动作：

| Gate | 对 selected | 对 near_miss | 对仓位 |
|---|---|---|---|
| `halt` | 全部降级为观察层 | 保留 | 0 |
| `shadow_only` | 仅允许进入 shadow report，不下正式单 | 保留 | 0 或极低 |
| `normal_trade` | 正常 | 正常 | 默认 |
| `aggressive_trade` | 正常，但只对高质量先验放宽 rank 容量，不放宽低质量门槛 | 正常 | 可上调到策略默认上限 |

### 6.6 验证口径

择日门控层的验收，不看单日命中，而看下面 5 个指标：

1. 弱势日正式 selected 数量下降幅度。
2. 弱势日的平均期望亏损收敛程度。
3. 强势日 retained selected 占比，防止把正收益窗口一起砍没。
4. `halt/shadow_only` 下被挡掉的样本里，是否出现大规模真实优质漏票。
5. 20 日滚动 post-fee expectation 是否由负转正或至少显著改善。

## 7. 详细方案二：先验质量门槛层

### 7.1 设计目标

这层要解决的不是“历史命中率多高”，而是“当前这份历史统计值值不值得信”。

具体要修三类问题：

1. 小样本极端值不再直接当真。
2. 历史从未打出过目标空间的票，不再进入正式精选。
3. 只擅长日内冲高、收盘延续偏弱的票，不再和真正 close continuation 型候选混在一起。

### 7.2 建议新增的 3 级先验质量分类

建议把每只 BTST 候选的历史先验分成 3 级：

| 质量级别 | 定义 | 允许动作 |
|---|---|---|
| `execution_ready` | 样本量足够，历史高点命中和收盘延续都达标 | 可进入 selected |
| `watch_only` | 有一定日内空间，但收盘延续或样本量不足 | 只允许 near_miss / 观察层 |
| `reject` | 小样本过低、历史从未命中、或统计无意义 | 不进入 short trade 正式名单 |

### 7.3 硬门槛规则

第一版建议先上硬门槛，先把最明显的脏样本清掉：

1. `n < 5`：不允许进入 selected。
2. `n < 3`：默认不进入 near_miss，除非仅作为 research 观察样本。
3. `n >= 3` 且历史 `high>=2% = 0`：直接 `reject`。
4. `close+ < 50%`：默认不进入 selected，只能进入 `watch_only`，除非后续独立验证证明该票在强势日具备稳定确认型收益。
5. `n` 很小但命中率很高时，不允许因为 100% 或 66.7% 这种表面数值直接升级到正式精选。

按这个规则回看 [01-0422-实盘复盘与数据证据.md](./01-0422-实盘复盘与数据证据.md) 中的样本：

1. `002028` 会被直接挡下，因为 `high>=2%` 历史为 0。
2. `300395` 不会进入正式精选，因为 `n=2`。
3. `002916`、`300476` 这类极低样本量名字，也不会再以“正式精选”身份出现。
4. `300394`、`688313` 会保留为 `watch_only` 候选，而不是直接与高质量 selected 混排。

### 7.4 可靠性折算：不要只看原始命中率

仅有硬门槛还不够，还需要对中低样本统计做 shrinkage（收缩）。

建议引入如下折算思路：

```text
sample_reliability = n / (n + k)

shrunk_high_hit = sample_reliability * raw_high_hit
                  + (1 - sample_reliability) * regime_baseline_high_hit

shrunk_close_positive = sample_reliability * raw_close_positive
                        + (1 - sample_reliability) * regime_baseline_close_positive
```

其中：

1. `k` 初始建议取 `8`，先让 `n<10` 的样本明显受到收缩。
2. `regime_baseline_*` 不用全市场固定值，而是按日级 gate 状态分桶，例如 `halt/shadow_only/normal_trade/aggressive_trade` 四档分别维护基准值。
3. selected 与 near_miss 的判断，用收缩后的命中率，而不是原始命中率。

这一步的好处：

1. `n=2` 或 `n=3` 的表面高命中率不会再虚高。
2. 大样本票仍然可以保留其真实统计优势。
3. 先验统计会自动与当天市场环境挂钩，而不是完全脱离 regime 使用。

### 7.5 建议新增的 profile 字段

这层最适合落在短线 profile，而不是硬编码在脚本里。建议在 [../../../../../src/targets/profiles.py](../../../../../src/targets/profiles.py) 对应的 `ShortTradeTargetProfile` 中新增以下字段：

1. `selected_min_historical_evaluable_count`
2. `near_miss_min_historical_evaluable_count`
3. `selected_min_high_hit_rate`
4. `selected_min_close_positive_rate`
5. `selected_zero_high_hit_reject_min_count`
6. `selected_use_shrunk_prior_rates`
7. `prior_shrinkage_k`
8. `low_sample_force_watch_only`

这样可以做到：

1. 规则可配置。
2. 不同 gate 状态可以使用不同 prior 严格度。
3. 后续验证时可以只改 profile，不必反复改业务代码。

### 7.6 对 selected / near_miss 的实际影响

建议把这层的影响写成强制 downgrade，而不是仅做提示：

1. `execution_ready`：允许 selected。
2. `watch_only`：即使分数足够，也强制 downgrade 到 near_miss。
3. `reject`：即使上游分数通过，也不进入 short trade 正式列表。

也就是说，这一层的职责是判断“有没有资格参加正式交易排序”，而不是给研究员看个备注就算完。

### 7.7 验证口径

先验质量层的验收看下面 6 个指标：

1. `n < 5` 的 selected 数量是否降到 0。
2. `high>=2% = 0` 的 selected 数量是否降到 0。
3. selected 样本的 median `n` 是否明显抬升。
4. selected 的 `close+` 先验分布是否明显改善。
5. selected 相对 near_miss 的胜率差是否从 +2% 拉开到更有意义的水平。
6. 强势日里是否仍能保留大样本高弹性票的观察层信息，不至于因过严门槛把所有候选都砍空。

## 8. 详细方案三：精选/观察层契约重构

### 8.1 设计目标

0422 之后，BTST 必须明确承认：

1. “值得研究”不等于“值得交易”。
2. “值得盯盘确认”不等于“可以直接列入正式买入名单”。
3. “历史会冲高”不等于“适合拿到收盘”。

因此需要把当前名单层级重新定义成明确 contract。

### 8.2 建议的名单语义

| 层级 | 新语义 | 执行动作 |
|---|---|---|
| `selected` | 正式交易名单 | 允许进入 buy order 流程 |
| `near_miss` | 观察名单 | 只跟踪，不自动执行 |
| `upgrade_only` / `research_only` | 研究名单 | 不参与正式交易统计 |

### 8.3 关键改动

建议在 [../../../../../src/targets/router.py](../../../../../src/targets/router.py) 对应的 selection target 汇总逻辑里增加两类强制语义：

1. 如果日级 gate 不是 `normal_trade` 或 `aggressive_trade`，则 selected 强制降级到 near_miss 或 research。
2. 如果先验质量不是 `execution_ready`，则 selected 强制降级到 near_miss。

这意味着 selected 不再只是“分数排前面”，而是“通过了日级门控和先验门槛之后仍然排前面”。

### 8.4 针对 `close+ < 50%` 票的单独语义

像 `300394`、`688313` 这种票的特点是：

1. 历史 `high>=2%` 还可以。
2. 历史 `close+` 偏弱。
3. 强势市场里可能收得住，但弱一点就容易高开回落。

这类票不应直接删除，但应改写语义为：

1. 默认归入 `watch_only`。
2. 只在 `aggressive_trade` 日允许升级到确认型观察票。
3. 在没有真实执行桥接验证前，不进入默认 selected。

这样做的收益是：

1. 不会因为过度保守把高弹性方向全部清空。
2. 也不会继续把“会冲高但不一定收住”的票混进正式精选，稀释 selected 的信息含量。

### 8.5 报告层必须新增的解释字段

为了让后续复盘不再靠猜，建议在 `selection_review.md`、`selection_snapshot.json`、`session_summary.json` 中补齐以下字段：

1. `btst_regime_gate`
2. `historical_prior_quality_level`
3. `historical_prior_reliability`
4. `raw_high_hit_rate`
5. `raw_close_positive_rate`
6. `shrunk_high_hit_rate`
7. `shrunk_close_positive_rate`
8. `downgrade_from_selected_reason`
9. `execution_eligible`

这样后续看到一只票在 report 里是 near_miss，就能明确回答：

1. 是因为市场状态不允许交易。
2. 还是因为历史先验不足。
3. 还是因为它只适合观察、不适合正式执行。

## 9. 详细方案四：风险预算与持仓约束

### 9.1 设计目标

风险层不解决“选不选”，而解决“即使选了，也该不该给它正式风险预算”。

0422 之后，这层至少要覆盖 4 件事：

1. 弱势日总风险预算为 0 或极低。
2. 低可靠性先验样本不给正式仓位。
3. 观察层样本不再通过语义漂移混进正式交易。
4. T+1 是主目标，禁止默认拖到 T+2/T+3 去“靠持仓修收益”。

### 9.2 建议的风险预算规则

| 条件 | 风险预算建议 |
|---|---|
| `halt` | 0 |
| `shadow_only` | 0 或仅 paper/shadow |
| `normal_trade + execution_ready` | 默认仓位 |
| `aggressive_trade + execution_ready` | 可放大到上限 |
| `watch_only` | 0 |
| `reject` | 0 |

### 9.3 单票风险控制

建议新增如下单票级限制：

1. `n < 5` 的票禁止分配正式仓位。
2. `close+ < 50%` 的票即使保留在观察层，也不能自动升级为正式买入。
3. `upgrade_only` 和机会池样本永远不进入正式 BTST 绩效统计。
4. 若日级 gate 为 `shadow_only`，则所有 short trade 结果都只能进入观察层报告，不进入执行层。

### 9.4 持仓周期约束

根据 [01-0422-实盘复盘与数据证据.md](./01-0422-实盘复盘与数据证据.md) 中的 T+2 / T+3 衰减数据，建议在本轮优化期间固定以下边界：

1. 默认绩效目标仍以 T+1 收盘了结为准。
2. T+2 / T+3 只作为观察指标，不纳入“本轮优化是否成功”的主要验收口径。
3. 只有当 T+1 已经稳定转正后，才考虑继续研究持仓延长。

## 11. 验证框架与实验顺序

### 11.1 为什么必须分阶段

0422 专项最忌讳的事情，就是同时改：

1. 日级 gate
2. 先验门槛
3. 评分阈值
4. 风险仓位

因为一旦结果变好或变坏，根本无法知道是哪一层起作用。

因此建议严格使用单主题实验。

### 11.2 推荐阶段顺序

| 阶段 | 主题 | 是否影响正式决策 | 目标 |
|---|---|---|---|
| Phase 0 | 基线冻结 | 否 | 固定 20 日统计口径与 4/22 个案样本 |
| Phase 1 | 择日门控 shadow | 否 | 只打标签，不拦截，评估弱势日识别效果 |
| Phase 2 | 择日门控 hard enforce | 是 | 真正压缩弱势日正式入场 |
| Phase 3 | 先验质量硬门槛 | 是 | 清理小样本、零命中、低 close+ 正式精选 |
| Phase 4 | 先验 shrinkage | 是 | 优化中样本段的排序与 downgrade 质量 |
| Phase 5 | execution eligibility contract | 是 | 拉开 selected 与 near_miss 的语义与统计差距 |
| Phase 6 | 风险预算 overlay | 是 | 控制黑天鹅与错误日暴露 |

### 11.3 每阶段的最小验收标准

#### Phase 1 / 2：择日门控

1. 弱势日 selected 数量显著下降。
2. 强势日保留下来的 selected 不出现明显塌缩。
3. 总体期望值改善方向清晰。

#### Phase 3：先验质量硬门槛

1. `n < 5` 的 selected 归零。
2. `high>=2% = 0` 的 selected 归零。
3. selected 的样本质量分布明显上移。

#### Phase 4：先验 shrinkage

1. 中样本票排序更稳定。
2. selected 与 near_miss 的统计分离进一步拉大。
3. 没有明显伤害强势日主收益样本。

#### Phase 5：contract 重构

1. selected 的统计超额明显改善。
2. near_miss 成为真正的观察层，而不是弱 selected。
3. 报告中能解释每一次 downgrade。

#### Phase 6：风险预算

1. 单日极端回撤风险下降。
2. 观察层不再混入正式交易统计。
3. T+1 目标仍保持清晰，不被持仓拖期掩盖。

## 13. 预期收益与成功标准

### 13.1 本轮最现实的成功定义

0422 专项不要求一步到位做到极高收益，先看三个是否达成：

1. 20 日滚动期望从负值提升到接近 0 以上，最好转正。
2. selected 相对 near_miss 的统计差距明显拉开。
3. 明显错误样本不再进入正式精选。

### 13.2 建议追踪的核心指标

| 指标 | 当前基线 | 目标方向 |
|---|---|---|
| 20 日 T+1 期望收益 | -0.0448%（扣费后更差） | 转正 |
| 弱势日平均胜率 | 22.6% | 提升或大幅减少参与 |
| selected vs near_miss 胜率差 | +2.0% | 明显扩大 |
| `n<5` selected 占比 | 非 0 | 归零 |
| `high>=2%=0` selected 数量 | 非 0 | 归零 |

### 13.3 失败信号

出现以下任一情况，都说明方案要暂停或回滚：

1. 弱势日暴露没有显著下降。
2. 强势日覆盖被大幅误砍，导致整体收益更差。
3. selected 与 near_miss 的差距仍然拉不开。
4. 低质量样本虽然少了，但总体期望没有改善。
5. 为了救 selected，不得不同时放松过多阈值，导致系统重新变回“多放票、大回撤”。

## 14. 为什么这个顺序比“继续调阈值”更合理

如果继续优先调阈值，会有三个问题：

1. 弱势日问题仍然存在，再好的阈值也会被大环境吞掉。
2. 小样本幸存者偏差仍会继续污染正式精选。
3. selected 与 near_miss 的语义继续模糊，系统会越来越难解释。

而本方案的优点是：

1. 每一层都能单独验证。
2. 每一层都能单独回滚。
3. 每一层都直接对应 [01-0422-实盘复盘与数据证据.md](./01-0422-实盘复盘与数据证据.md) 中已经确认的具体问题。

---

## 15. 最终建议

0422 之后的 BTST，不应再把主要精力放在“继续扩 coverage”或“继续松阈值”上，而应严格按下面顺序推进：

1. 先做择日门控，优先解决弱势日负期望问题。
2. 再做先验质量门槛，优先清掉小样本、零命中、低延续票。
3. 再把 selected 与 near_miss 的 contract 写实，恢复正式名单的统计含量。
4. 最后用风险预算把剩余黑天鹅损失压住。

如果这 4 步能按顺序跑通，BTST 才有可能从“偶尔在强日选对票”，升级到“长期具备正期望的可执行策略”。
