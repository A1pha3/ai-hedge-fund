修改 3 个 skill 文件，纯 markdown 编辑，不碰 production code。每个修改都解决一个我过去两轮实际踩过的坑。

### 修改 1：`SKILL.md` — 运行流程增加两级停止 + false exhaustion 时效检查 + 停止条件统一清单

**改动 A — 运行流程第 2 步（读 state）增加 false exhaustion 时效检查：**

当前：
```
2. 读取或初始化 decision state；若 JSON strict parse 失败或 caps 无法 reduce，停止而不是猜测。
```
改为：
```
2. 读取或初始化 decision state；若 JSON strict parse 失败或 caps 无法 reduce，停止而不是猜测。若 state 含 `candidate_space_exhausted` 声明，验证其时效：声明写入超过 7 天，或期间有新 commit 落地，视为 stale，必须重新构建 frontier 而非直接信任旧声明（false exhaustion 防护）。
```

**改动 B — 运行流程第 3 步（构建 frontier）增加候选级跳过逻辑：**

当前：
```
3. 构建 candidate frontier：产品 gap、测试盲区、风险、release-readiness gap、calibration/regret 都可以进入候选。
```
改为：
```
3. 构建 candidate frontier：产品 gap、测试盲区、风险、release-readiness gap、calibration/regret 都可以进入候选。遇到需 owner 授权/裁定的 candidate 时标记为 blocked，**继续扫描其他不依赖 owner 的候选**，不要因单个 blocked candidate 就停止整个 session。
```

**改动 C — 运行流程之后新增 `## 停止条件` section（统一清单）：**

在"运行流程"和"Method Improvement"之间插入：
```markdown
## 停止条件

停止分两级，agent 必须正确区分：

### Session 停止（终止整个 session）

只有以下情况才停止整个 session：
- decision state JSON parse 失败或 caps 无法 reduce（无法安全继续）。
- budget 耗尽（campaigns_executed >= max_campaigns）。
- 所有 candidate 都 blocked（需 owner 授权/裁定/外部条件），且 frontier 已无可用候选。
- Method Patch Contract 缺项（method improvement 阻断）。

### Candidate 停止（跳过当前 candidate，继续下一个）

以下情况只跳过当前 candidate，不停止 session：
- candidate 需要 owner 授权的动作（push/deploy/覆盖文件等）→ 标记 blocked，继续扫描。
- candidate 需要 owner 裁定的产品决策 → 标记 blocked，继续扫描。
- candidate 需等待外部条件（数据累积/时间）→ 标记 blocked，继续扫描。

### 聚合判断

当所有剩余 candidate 都 blocked 时，才升级为 session 停止。此时输出 `next_trigger` 记录每个 blocked candidate 的解除条件，让 owner 知道做什么能恢复。
```
