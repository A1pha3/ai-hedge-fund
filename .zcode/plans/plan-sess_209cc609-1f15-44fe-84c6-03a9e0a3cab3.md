## 修复方案：DiskCache 线程安全 + 数据完整性

### 第一性原理分析

日志里的 4 种错误变体，根因是 3 个独立 bug，全部集中在 `src/data/enhanced_cache.py` 的 `DiskCache` 类：

| 错误信息 | 根因 | 严重度 |
|---|---|---|
| `bad parameter or other API misuse` | **竞态条件**：读路径 `get()` 通过 `_ensure_conn()` 拿到 `self._conn` 引用后，在 `conn.execute()` 执行期间，另一线程的 `close()` 或 `_ensure_conn()` 重建可能 `.close()` 掉底层连接 → sqlite3 Connection 对象失效 → SQLITE_MISUSE | **根因**（占日志 80%+） |
| `a bytes-like object is required, not 'NoneType'` | **缺 None 守卫**：DB 的 `value` 列读到 NULL（历史写入失败/外部篡改），`pickle.loads(None)` 抛 TypeError | 中 |
| `invalid load key` / `Ran out of input` | **BLOB 被截断/污染**：pickle 反序列化失败，且损坏行不被清理导致每次访问都报错 | 低 |

**共同症状**：全部被 L470 的 `except Exception: return _sentinel` 吞掉 → 缓存静默失效 → 回源拉数据 → 不影响正确性但刷屏日志 + 浪费网络请求。

### 修复策略（3 个 bug，一把钥匙）

**核心思路**：`check_same_thread=False` 的 sqlite3 连接，SQLite 官方文档明确要求"使用方必须自行序列化所有访问"。当前代码只序列化了写（`_write_lock`），读路径裸奔——这就是竞态的根源。一把 `RLock` 保护所有连接访问即可根治。

#### 改动 1：`_write_lock` → `_conn_lock`（RLock），保护所有连接访问

- `__init__`：`self._write_lock = threading.Lock()` → `self._conn_lock = threading.RLock()`
  - 用 RLock 而非 Lock：`get()` → `_ensure_conn()` → `_is_alive()` 三层调用栈，RLock 允许同线程重入，无需手动传递"已持锁"状态
- 所有公共方法（`get/set/delete/clear/count_entries/close`）在入口 `with self._conn_lock:` 包裹**包括 `_ensure_conn()` 在内的完整操作**
- 内部方法（`_is_alive`、`_ensure_conn`）也获取 `_conn_lock`（RLock 允许重入），保证从任何上下文调用都安全（测试直接调用 `_get_conn()` 时也不会裸奔）
- 性能影响：每次 `get()` 的锁开销 ≈ 3 × RLock acquire/release ≈ 300ns；SQLite point query ≈ 50µs。300 股 × 5 通道 = 1500 次查询，多数命中 LRU 不触达磁盘，磁盘层序列化开销 < 100ms，可接受

#### 改动 2：NULL value 守卫（Bug 2）

`get()` 中 `pickle.loads(value)` 前加 `if value is None: return _sentinel`，并删除该脏行。

#### 改动 3：拆分 try 块，区分错误等级 + 自动清理脏行（Bug 3）

将 `get()` 的大 try 拆成两段：
- **SQL 查询段** → 连接级错误（SQLITE_MISUSE 等）记 WARNING（基础设施问题，operator 应感知）
- **反序列化段** → 数据级错误（pickle 失败）记 DEBUG（已自动清理，无需人工干预），同时 DELETE 该脏行避免重复报错

#### 改动 4：现有方法统一改名 `_write_lock` → `_conn_lock`

涉及行：L283(定义)、L395/464/492/510/528(使用)。

### 不改什么

- **不改公共 API**：`get/set/delete/clear/count_entries/close` 签名不变，`_sentinel` 参数语义不变
- **不改 `_conn` 属性名**：测试直接访问 `cache._conn`（6 处），改名破坏面大
- **不改 `_get_conn()`**：保持向后兼容（测试 L154-155 仍调用）
- **不改 WAL/PRAGMA/惰性删除逻辑**：这些是正确的，不动
- **不把 unpickle 移到锁外**：YAGNI，典型缓存值 ~50KB，unpickle < 1ms，锁内完成更简洁

### 验证计划

1. **现有测试全过**（`tests/test_enhanced_cache_wal.py`，7 个测试）：
   - 并发读写测试（4 writer + 4 reader）：序列化后仍无 SQLITE_BUSY、无丢 key
   - 惰性删除竞态测试（R20.32）：条件 DELETE 逻辑不变
   - 连接复用测试：锁不影响连接复用（不触发重连）
   - close 后行为测试：close 获取 `_conn_lock`，后续 get/set 静默降级

2. **新增 3 个测试**：
   - `test_disk_cache_get_handles_null_value`：直接插 NULL value → get 返回 sentinel + 脏行被删
   - `test_disk_cache_get_handles_corrupted_blob`：插无效 BLOB → get 返回 sentinel + 脏行被删 + 不产生 WARNING
   - `test_disk_cache_corrupted_blob_deleted_after_failure`：同一脏 key 第二次 get 不再报错（行已被清理）

3. **运行验证**：
   ```bash
   uv run pytest tests/test_enhanced_cache_wal.py -v
   uv run pytest tests/offensive/ -v  # 回归
   ```

### 影响面

- 改动文件：`src/data/enhanced_cache.py`（1 个文件）、`tests/test_enhanced_cache_wal.py`（加 3 个测试）
- 风险：低。改动是"收紧并发约束"，不改变任何方法签名或返回值语义。最坏情况是并发读性能下降（从并行退化到串行），但磁盘层仅是 LRU→Redis→Disk 三级缓存的最后一级，实测影响可忽略。
