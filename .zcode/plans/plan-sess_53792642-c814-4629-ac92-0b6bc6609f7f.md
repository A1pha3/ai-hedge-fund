## 目标

把 tushare 持久缓存从 `~/.cache/ai-hedge-fund/cache.sqlite`（占 HOME 卷 3GB，HOME 仅剩 32Gi）迁到项目内 `data/cache/cache.sqlite`（项目卷剩 760Gi）。

## 改动范围（共 4 处，其中代码改动仅 1 处）

### 1. 修 bug：`src/screening/data_freshness_guard.py:354-360`（唯一的代码改动）

现状：当 `cache_path is None` 时硬编码回退到 `Path.home() / ".cache" / "ai-hedge-fund" / "cache.sqlite"`，**不读 `DISK_CACHE_PATH` 环境变量**。缓存搬走后这里会找不到、静默失效。

改成先读 `DISK_CACHE_PATH`，再回退到 HOME 默认路径：

```python
if cache_path is None:
    env_cache = os.environ.get("DISK_CACHE_PATH")
    if env_cache:
        default_cache = Path(env_cache)
    else:
        default_cache = Path.home() / ".cache" / "ai-hedge-fund" / "cache.sqlite"
    if default_cache.exists():
        cache_path = default_cache
    else:
        return result
```

文件顶部已有 `from pathlib import Path`，需补 `import os`（确认下是否已导入）。

### 2. 配置：`.env` 取消注释并改路径

`.env.example:64-65` 已有该变量的文档化模板。在 **`.env`**（实际生效文件，非 example）中设置：

```
DISK_CACHE_PATH=data/cache/cache.sqlite
```

说明：
- 项目内其他 data 路径都用相对路径（`paper_tracker.py:17` 的 `Path("data/paper_trading/")`、`cache_refresh.py:21-22` 的 `Path("data/price_cache")` 等），保持一致用相对路径。
- 但 `--daily-action` / `--auto` 通过 `uv run python src/main.py` 从项目根执行，cwd 一致，相对路径安全。如果你有时从其他 cwd 跑，届时再换绝对路径。

### 3. `.gitignore`：补一行 `data/cache/`

现状：`.gitignore:97` 已有 `*.sqlite`，能拦住 `cache.sqlite` 本体。但 SQLite 的 WAL 模式会产生 `cache.sqlite-wal` / `cache.sqlite-shm` 两个 sidecar 文件，**这两个不被 `*.sqlite`（要求完整后缀）也不被 `*.db-wal`/`*.db-shm` 匹配**——会漏进 git。

补一行目录级忽略，一劳永逸覆盖 db + 所有 sidecar：

```
data/cache/
```

加在 `.gitignore` 现有 `*.sqlite3`（第 98 行）附近。

### 4. 迁移现有 3GB 缓存（手动执行，非代码改动）

确保没有进程在跑 `--auto` / `--daily-action`，然后：

```bash
mkdir -p data/cache
cp ~/.cache/ai-hedge-fund/cache.sqlite data/cache/cache.sqlite
# 如果有 -wal / -shm sidecar 也一并拷
cp ~/.cache/ai-hedge-fund/cache.sqlite-wal data/cache/ 2>/dev/null
cp ~/.cache/ai-hedge-fund/cache.sqlite-shm data/cache/ 2>/dev/null
```

迁移后验证：
```bash
uv run python scripts/manage_data_cache.py stats
```
确认 `file_size_bytes` 指向新路径、条目数没丢。验证 OK 后可删旧文件释放 HOME 空间：
```bash
rm ~/.cache/ai-hedge-fund/cache.sqlite*
```

> 备注：缓存是无状态可重建的（tushare 数据重新拉一次就有），所以即使不迁移、直接删旧的让它重建也完全没问题——只是会损失已有的缓存命中，首次 `--auto` 会变成冷跑。

## 不需要改的地方（已核实）

- **`enhanced_cache.py:273`**：已读 `DISK_CACHE_PATH`，是 singleton 入口，`manage_data_cache.py` / `tushare_api.py` / `akshare_api.py` 全走这里——一处配置全局生效。
- **`scripts/manage_data_cache.py`**：通过 `get_cache_runtime_info()` / `clear_cache()` 走 singleton，自动跟随。
- **`akshare_api.py:21`**：只是 docstring 提到默认路径，逻辑上走同一个 `get_cache()`，无需改。
- **其他 HOME 路径**：`flywheel_health.py` 的 `~/Library/Logs/ai-hedge-fund/` 是 launchd 需要的（boot volume 才能被 launchd 正常 spawn），且体积小，不动。

## 验证步骤（实施后执行）

1. `uv run python scripts/manage_data_cache.py stats` — 确认路径指向 `data/cache/cache.sqlite`
2. `uv run python src/main.py --auto` — 跑一次，确认无报错、缓存正常读写
3. `ls -la data/cache/` — 确认 cache.sqlite 在新位置
4. `du -sh ~/.cache/ai-hedge-fund/ 2>/dev/null` — 确认旧位置可清空释放空间

## 关于 3GB 无限增长的根因（可选，不在本次范围）

缓存没有自动 eviction / VACUUM（TTL 是惰性删除、不扫全表、不回收文件空间），所以搬到项目卷后仍会持续增长。如果以后想根治，可以给 `scripts/manage_data_cache.py` 加 `prune` 子命令（删过期行 + `VACUUM`）。这是独立改进，本次先解决空间位置问题。
